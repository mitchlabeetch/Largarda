# PRD-visual — 视觉与交互

## 产品定位提醒（已更新：视觉风格方向修正）

视觉是第一印象，也是"惊艳"的来源。用户打开虚拟办公室，前3秒如果没有被震到，后面功能再强也没用。

**视觉风格方向修正（响应老乔质疑8 + 用户参考图）：**

用户提供的参考图方向是 **Lofi/Cozy 像素风**，不是赛博朋克冷色调。原PRD-visual的"赛博朋克 + 青色荧光"方向与用户意图不符，**全部更新为暖色调像素风**。

| 原PRD方向（已废弃）           | 实际方向                           |
| ----------------------------- | ---------------------------------- |
| 赛博朋克冷色调，青色/紫色荧光 | Lofi/Cozy暖色调，暖黄/橙/木色/绿植 |
| 高密度信息，全息感            | 温馨生活感，有窗帘/台灯/植物       |
| 科幻办公室场景                | 有家的感觉的个人房间               |
| 压迫感、酷感                  | 舒适感、亲切感                     |

这不是小的色调调整，是整个情感基调的改变。用户提供了参考图，这是明确信号，产品层面跟随。

## 1. 功能描述

### 1.1 整体视觉风格

**核心关键词：** Lofi / Cozy / 像素 / 暖色 / 温馨 / 有家的感觉

**色调（已更新）：**

- 背景：暖米白/浅木色/暖灰，不是深黑
- 主色：暖黄、橙、木色、柔和绿——lofi标准色
- 辅色：柔和红/橙用于警告状态（不刺眼）
- 材质感：木质纹理、布料感、温暖光源

**状态栏（顶部，已裁决 2026-04-01）：**

- TIME / STATUS 显示在房间顶部，像素风格
- **HEALTH / STRESS / PRODUCTIVITY 不显示进度条/数值**，改用卡片颜色信号传达：
  - 绿色边框 = 正常工作中
  - 橙色边框 = STRESS 超阈值（高压）
  - 红色边框 = 出错
  - 灰色 = 空闲/休眠
- 理由：不展示假数字，颜色信号够用，不破坏沉浸感

**字体：**

- 界面字体：像素点阵字体，暖色调，不是等宽冷色系

**参考意向：**

- 像素游戏：Stardew Valley 室内场景的暖色调处理
- Lofi Girl 封面画风的温馨感
- 角色规格：8-16px 像素风角色，Stardew Valley人物规格

**窗外日夜循环（新增需求）：**

- 房间窗外跟随真实系统时间变化（凌晨深蓝→清晨橙粉→白天亮蓝→黄昏橙红→夜晚深蓝）
- 室内光源随时间变化：白天自然光 → 傍晚台灯亮起 → 深夜只有屏幕光
- 技术实现：读取本地时间，PixiJS滤镜驱动颜色渐变，开发成本适中
- 优先级：P1（不是MVP必须，但沉浸感贡献很大）

### 1.2 场景渲染

**虚拟办公室场景是产品的"大地图"**，不是员工列表页。用户进入后看到的是一个像素风办公室，员工们在各自的工位上活动。

> **已确认：横版侧视（side view）**，类Stardew Valley室内视角。参考图已明确此方向，不是等距俯视（isometric）。所有家具坐标、hitbox、骨骼动画方向均基于side view制定。

**场景元素：**

| 元素       | 说明                                               |
| ---------- | -------------------------------------------------- |
| 办公桌     | 每个员工有独立工位，家具布局暂时固定（P2可自定义） |
| 角色像素人 | 骨骼动画（Spine格式），PixiJS + pixi-spine渲染     |
| 电脑屏幕   | 员工桌上的电脑，Canvas hitbox可点击，触发DOM抽屉   |
| 背景装饰   | 书架、窗户、植物、地毯等Lofi/Cozy元素              |
| 环境动效   | 窗外光线变化、台灯光晕、轻微粒子                   |

**技术选型（已由用户裁决锁定）：**

- Canvas引擎：**PixiJS**（用户裁决，不可更改）
- 骨骼动画：**Spine + pixi-spine**（用户裁决P0必须，不可降级为帧动画）
- 场景文件格式：Tiled（.tmx），方便场景编辑

### 1.3 像素角色与骨骼动画

**骨骼动画状态列表（已更新，2026-04-01用户最终确认RPG移动模型）：**

> **`resting` 和 `sleeping` 是两个独立状态**，触发条件不同，不可合并。
>
> **新增状态：`walking`（行走移动）** — 用户确认角色在房间内像RPG角色一样移动，walking是P0必须动画，不是可选。

| 动画                  | 触发条件               | 描述                                                 |
| --------------------- | ---------------------- | ---------------------------------------------------- |
| idle（待机）          | 无任务，无对话         | 轻微晃动，环顾四周，偶尔伸懒腰；≥3组随机轮换         |
| working（工作）       | 工作实例运行中         | 快速敲击键盘，眼神专注                               |
| thinking（思考）      | LLM 生成中，等待响应   | 摸下巴，来回踱步，头顶出现省略号气泡                 |
| typing（聊天）        | 社交实例响应中         | 打字动作，面向用户方向                               |
| reading（阅读）       | 记忆实例整理中         | 翻书/查看屏幕                                        |
| celebrate（庆祝）     | 任务完成               | 举拳、跳起、短暂欢呼                                 |
| resting（休息/发呆）  | 完全空闲，无任务无对话 | 伸懒腰、发呆、随机小动作；和 idle 可共用动作组       |
| sleeping（休眠）      | 用户手动暂停员工       | 躺在床上，胸部呼吸起伏，ZZZ气泡，整体变暗色调        |
| waking_up（唤醒）     | 用户点击床确认唤醒     | 揉眼睛、撑起身体、站起来；过渡到walking状态          |
| walking（行走移动）   | 状态切换时位置变化     | 自然行走循环，RPG风格侧视；移动到目标位置后进入下一状态 |
| error（出错）         | 任务失败/报错          | 抓头，问号气泡，角色轻微抖动                         |

**美术资源方向（已由用户裁决）：自绘，不购买素材包，不用开源资产。**

**已更新：骨骼动画是P0必须（用户最终确认），序列帧仅作降级备选。**

### 1.4 员工卡片设计

工作空间列表视图（非场景视图）中，员工以"战斗卡"风格展示，不是普通 SaaS 卡片。

**卡片信息层次：**

```
┌──────────────────────────────┐
│  [像素头像]  [名字]  [职位]   │ ← 头部：身份
│  [等级/经验条]                │ ← 成长状态
│  ┌─────────────────────────┐ │
│  │  当前状态：工作中         │ │ ← 实时状态（动态）
│  │  任务：Review PR #123    │ │
│  └─────────────────────────┘ │
│  [已安装模块图标]              │ ← 装备栏
│  [进入工作空间]  [发消息]      │ ← 操作按钮
└──────────────────────────────┘
```

**卡片视觉效果：**

- 边框：荧光描边，根据状态变色（工作中=青色，空闲=暗色，出错=红色）
- 卡片背景：深色半透明玻璃拟态
- 状态文字：打字机效果实时更新

### 1.5 状态动效（已更新：骨骼动画P0必须，2026-04-01用户最终确认）

**用户确认（最终，2026-04-01更新）：**

1. **角色在房间内像RPG角色一样移动（walking动画P0必须）** — 覆盖之前"不需要走路"的旧方案；路径固定预设，不做A\*寻路
2. **骨骼动画是MVP必须的，不是P1/P2**——理由是流畅性
3. 一套骨架支持多种动作（敲键盘/看书/睡觉/起身/走路），换装区分职位
4. **床可以点击唤醒**，唤醒后员工走路移动到电脑前（walking动画体现）

**技术方案：骨骼动画（主方案）+ 序列帧（降级备选）**

| 方案     | 说明                                                                    |
| -------- | ----------------------------------------------------------------------- |
| 主方案   | 骨骼动画：PixiJS + pixi-spine（Spine格式）或 PixiJS + DragonBones适配层 |
| 降级备选 | 序列帧 spritesheet，仅在骨骼动画资产加载失败时自动切换                  |

**为什么骨骼动画成本不一定更高：**
一套骨架定义骨骼结构，所有动作（敲键盘/看书/睡觉/起身）复用同一骨架，通过换装（替换皮肤纹理）区分不同职位的员工。比"每个动作单独画帧"的纯序列帧方案资产复用率更高。

---

#### A档（MVP/P0，骨骼动画，必须交付）

**角色必须是"活的"，不能是静态图片。** 每个状态有生动的骨骼循环动画：

| 状态                        | 位置             | 骨骼动画动作内容                                                        |
| --------------------------- | ---------------- | ----------------------------------------------------------------------- |
| working（工作中）           | 电脑前           | 手指敲键盘节奏循环 + 偶尔抬头看屏幕 + 偶尔端起咖啡杯                    |
| reading（看书/记忆整理）    | 桌边或书架旁     | 手部翻笔记本/翻书动作循环 + 托腮思考 + 手指划过文字                     |
| resting（空闲休息）         | 房间中央偏右     | 伸懒腰 + 环顾四周 + 发呆随机轮换（≥3组）                                |
| sleeping（用户手动休眠）    | 床上             | 躺着，胸部呼吸起伏循环 + 偶尔轻微翻身，ZZZ气泡，整体变暗色调            |
| waking_up（唤醒，P0必须）   | 床→移动路径      | 揉眼睛 + 撑身站起（约0.5s）→ 过渡到walking                              |
| walking（行走，P0必须）     | 移动路径中       | 自然侧视行走循环，到达目标后过渡到目标状态的就位动作                    |
| idle（待机等待）            | 默认位           | 环顾四周 + 拉伸/打哈欠 + 发呆随机轮换（≥3组），可与 resting 共用动作组  |
| error（出错）               | 当前位置         | 停顿 + 抓头困惑 + 盯屏幕循环                                            |

**骨骼动画质量要求：**

- 每个循环动画自然流畅，无卡帧感
- idle 至少 3 组不同小动作随机轮换（生命感主要来源）
- 每个循环末帧能自然接回首帧

**状态切换（A档）：** 当前状态淡出（200ms）→ 新位置新状态淡入（200ms），无跳变。

#### B档（P1，退场/入场骨骼过渡）

在A档骨骼动画基础上，状态切换补充专用退场/入场动作：

| 切换方向    | 退场骨骼动作                    | 入场骨骼动作                             |
| ----------- | ------------------------------- | ---------------------------------------- |
| 空闲 → 工作 | 起身动作（0.3-0.5s）→ 淡出      | 淡入 → 坐下动作（0.3-0.5s）→ working循环 |
| 工作 → 休息 | 起身动作 → 淡出                 | 淡入 → 躺下动作 → sleeping循环           |
| 任意 → 出错 | 停止当前动作 → 困惑表情（立即） | 无入场，直接进error循环                  |

**总切换时长：** ~1.0-1.4s，不积压动画队列。

#### C档（P2，真实位移行走）

在B档基础上，退场后角色实际平移到目标位置（线性平移，预设固定坐标，无需A\*寻路）。

---

**MVP决策：A档（骨骼动画+淡入淡出）P0交付，B档P1，C档P2。**

**状态指示灯：**
卡片和场景中的员工头顶都有状态指示灯（类比游戏中的血条/状态图标），颜色含义：

- 绿色：工作中
- 蓝色：聊天中
- 黄色：思考/等待
- 灰色：空闲
- 黑色：休眠

### 1.6 工作空间内部视觉

进入员工工作空间后，视觉风格一致但更"个人化"：

- **聊天视图：** 终端风格（Terminal UI），黑底绿字或青字，带光标闪烁
- **工作视图：** 代码/日志风格，等宽字体，操作步骤像 CLI 输出一样流动
- **记忆视图：** 文件系统树形结构，带像素文件夹图标

### 1.7 空状态与首次进入体验（补充：响应老乔评估）

**这是转化率的核心漏斗节点，不能缺失。**

**场景A：用户第一次打开，0个员工**

不能显示空列表或空白场景。应该显示：

- 赛博朋克办公室场景，但所有工位都是空的，有"虚影"暗示这些位置可以放员工
- 屏幕中央一个明显的入口："招募你的第一个AI员工"按钮（像素风格，发光边框）
- 背景有环境动效（数据流、光效），场景有生命感，不是白板

**首次进入的叙事节拍（3-5秒）：**

1. 场景渐入（灯光从暗到亮，像"开门进入空办公室"）
2. 环境动效启动（粒子、数据流开始流动）
3. 屏幕中央出现提示："这里还没有员工，要招募第一个吗？"
4. 按钮出现，引导进入员工创建流程（走模板路径）

**场景B：用户已有1个以上员工**

进入后直接看到场景视图，员工在工位上活动，无引导遮罩。引导逻辑通过家具上的"首次发现"微交互提示实现（员工身上/家具上偶尔闪烁提示，点击后消失）。

**员工创建完成后的回落动效：**
新员工创建完成 → 场景中对应工位亮起 → 像素角色从场景外走进来，走到工位坐下 → 工位完全激活（3-4秒）。这是用户第一次看到"自己的员工"的关键时刻，要有仪式感。

## 3. 验收标准

### P0（A档：骨骼动画 + 状态枚举 + 淡入淡出）

- [ ] 虚拟办公室有像素风场景，Lofi/Cozy暖色调，包含家具装饰元素
- [ ] 使用骨骼动画（Spine/DragonBones格式，PixiJS集成），不是纯序列帧
- [ ] 员工角色有 working / reading / sleeping / idle / error / walking / waking_up 七种骨骼动画
- [ ] 每种状态动画自然流畅，idle有≥3组不同小动作随机轮换
- [ ] 状态切换时角色通过walking动画在房间内实际移动到目标位置，不是淡出瞬移（RPG移动模型）
- [ ] 不同状态下角色位置固定（工作=电脑前，记忆整理=桌边/书架旁，空闲=房间中央，休眠=床上）
- [ ] 点击床上睡觉的员工 → 弹出唤醒确认框 → 确认后员工起身走路移动到电脑前
- [ ] 整体色调符合暖色像素风（暖黄/木色/橙，不是冷色或纯白背景）
- [ ] 序列帧降级方案可用（骨骼动画资产加载失败时自动切换）

### P1（B档动画 + 体验升级）

- [ ] 状态切换补充退场/入场专用帧：起身→淡出 / 淡入→坐下，有"结束-开始"仪式感
- [ ] 员工角色完整支持 idle / working / sleeping / error / chatting 五种状态
- [ ] 点击家具（电脑/书架/通讯设备）可打开对应功能抽屉
- [ ] 员工卡片状态文字实时更新
- [ ] PixiJS 渲染帧率在桌面端稳定 ≥ 60fps（5个员工以下）

### P2（个性化扩展）

- [ ] 工位自定义拖拽
- [ ] 日夜循环（窗外光线跟随系统时间）

| 功能                                     | 优先级 | 原因                                                         |

## 5. 风险与歧义

**风险1：前端性能**
PixiJS + 多角色骨骼动画 + Electron 在低配机器上可能有性能问题。建议设置画质等级，低配自动降级为帧动画。

**风险2：像素资产制作成本**
自制像素角色成本高，需要美术。调研免费/低成本方案：

1. 使用免费开源像素资产（需确认商用授权）
2. AI 生成像素图（Midjourney/DALL-E pixel art 风格）
3. 购买授权包

**风险3：骨骼动画方案选型**
DragonBones 已停止维护，Spine 商用需付费。MVP 阶段强烈建议先用 spritesheet 帧动画，P1 再升级。

**员工数量：无上限（用户裁决 2026-04-01）**
员工没有数量上限，UI必须支持任意数量。场景通过鼠标滚轮缩放 + 拖拽平移处理多员工布局，类似 Figma 画布。不允许强制设置上限（比如MVP最多5个）。

**歧义2（已决策）：场景视图 vs 卡片列表 → 只做场景视图**

决策（响应老乔质疑4）：**MVP只做场景视图，不做独立的卡片列表页面。**

- 场景视图是这个产品的差异化核心，两个视图都做会导致两边都是半成品
- 卡片列表的功能通过"场景中点击员工卡片展开"实现（员工工位卡片即是列表项），不需要独立页面
- 员工多了（>5个）的管理效率问题：通过场景缩放（鼠标滚轮）+ 员工状态快速筛选（顶部过滤器）解决，而不是切换到列表视图

**这个决策在开发开始前已锁定，不允许两边都做半成品。**
