## 核心原则（用户裁决后重新定义）

养成感的唯一真实来源是**记忆持久化**。

"我的架构师已经用了3个月了，他理解我们团队的代码风格，换一个从零开始我舍不得。"

这句话里没有等级，没有经验条。用户舍不得的是**记忆积累**，是员工"认识你"的程度。所有养成感都应该围绕记忆系统建立。

## 1. 记忆系统（MVP命脉 P0）

记忆系统是这个产品存在的最核心理由，不是可选功能。

### 1.1 记忆的本质

员工的记忆 = 他的工作空间文件。实现原理与 Claude 的 CLAUDE.md 机制完全相同：

- **员工的核心记忆文件 = 他自己的 CLAUDE.md**
- 每次任务开始，记忆文件注入系统提示词
- 记忆越丰富，员工工作越精准——这是真实发生的，不是数字游戏

### 1.2 记忆层级

| 层级       | 内容                   | 更新频率         | 生命周期         |
| ---------- | ---------------------- | ---------------- | ---------------- |
| 工作上下文 | 当前任务的临时信息     | 实时             | 任务结束清空     |
| 任务摘要   | 已完成任务的摘要       | 每次任务后       | 保留，可压缩     |
| 核心记忆   | 通用经验、方法论、偏好 | 记忆实例定期整理 | 永久保留，可编辑 |
| 关系记忆   | 认识的员工、协作模式   | 有新协作时更新   | 永久保留         |

### 1.3 记忆自动写入流程

1. **任务结束** → 工作实例（A实例）自动生成任务摘要，写入工作空间任务日志
2. **触发记忆整理** → 记忆实例（B实例）读取近期任务日志，提炼关键经验
3. **沉淀到核心记忆** → 合并到核心记忆.md，归档旧日志
4. **下次任务开始** → 核心记忆注入系统提示词，员工"记得"之前的经验

**这个循环是养成感的真实技术基础，不是视觉包装。**

### 1.4 记忆质量保证

- 记忆文件用 Markdown 格式存储，用户可读、可编辑
- **MVP就允许用户手动编辑记忆文件**：点击书架 → 选择文件 → 进入编辑模式
- 编辑时有警告提示："手动修改记忆文件会影响员工的工作方式，确定继续？"
- 用户有控制感 = 用户信任系统，特别是记忆整理质量不稳定的MVP阶段

### 1.5 记忆整理策略（防止文件膨胀）

- 触发条件：任务完成后、记忆文件超过阈值大小时（建议单文件上限 50KB）
- 整理流程：读取近期任务日志 → 提炼关键经验 → 合并到核心记忆 → 归档旧日志
- 整理时视觉表现：员工在书架旁翻书（对用户可见，但不强调技术概念）

## 2. 参考系统：Claude的团队记忆

本产品的记忆系统参考 `~/.claude/memory/` 的设计：

```
~/.claude/memory/
├── MEMORY.md          # 记忆索引（类比：工作空间/记忆库索引）
├── USER_STYLE.md      # 用户偏好（类比：核心记忆.md）
├── TEAM_CONSENSUS.md  # 团队共识（类比：工作风格.md）
└── roles/             # 角色记忆（类比：各员工的专项记忆）
```

**对应关系：**

| Claude 记忆系统          | 虚拟办公室员工记忆          |
| ------------------------ | --------------------------- |
| CLAUDE.md（规则注入）    | 核心记忆.md（每次任务注入） |
| memory/USER_STYLE.md     | 工作风格.md                 |
| memory/TEAM_CONSENSUS.md | 关系图谱.md                 |
| 对话上下文               | 工作上下文（任务结束清空）  |

这不是类比，是直接的技术实现参考。

## 3. 用户故事

**US-401（P0）** 作为用户，我想让员工记住我的偏好和团队约定，这样我不用每次任务都重复说明。

**US-402（P0）** 作为用户，我想看到员工的记忆文件，了解他"记住了什么"，这样我有透明感和掌控感。

**US-403（P0）** 作为用户，我想编辑员工的记忆文件，纠正错误的记忆，这样员工行为符合我的预期。

**US-404（P1）** 作为用户，我想在书架上看到记忆文件内容越来越丰富，这样员工的成长是看得见的。

## 4. 验收标准

### P0（记忆系统核心，MVP命脉）

- [ ] 任务结束后自动生成任务摘要，写入工作空间任务日志
- [ ] 核心记忆文件在每次任务开始时注入系统提示词（真实生效，不是装饰）
- [ ] **可验证：** 第一次任务后员工记住了用户的偏好，第二次任务中能体现出来
- [ ] 用户可点击书架查看并编辑记忆文件，带风险提示
- [ ] 记忆数据持久化到本地，App重启后不丢失

### P1（记忆整理优化）

- [ ] 记忆实例（B实例）自动触发：任务完成后异步整理，不阻塞用户操作
- [ ] 记忆整理时书架有"整理中"视觉效果（员工翻书 + 轻微光效）
- [ ] 记忆文件大小有上限约束，超过阈值自动压缩归档旧日志

## 5. 优先级

| 功能                   | 优先级 | 原因                                     |
| ---------------------- | ------ | ---------------------------------------- |
| 任务摘要自动写入       | P0     | 记忆系统的数据来源，没有这个就没有记忆   |
| 核心记忆注入系统提示词 | P0     | 记忆系统的效果载体，这是"越用越好"的机制 |
| 记忆文件可查看+可编辑  | P0     | 用户信任的基础                           |
| 记忆持久化（本地存储） | P0     | 没有持久化就没有养成感                   |
| 记忆实例自动整理       | P1     | 防止文件膨胀，保证记忆质量               |

## 6. 风险

**风险1：记忆文件大小失控**
员工工作多了记忆文件越来越大，影响上下文注入性能。记忆实例的整理压缩策略是关键，需要产品和架构联合设计。建议单文件上限 50KB，超过触发强制整理。

**风险2：记忆质量不稳定**
LLM 整理记忆时可能编造或遗漏重要信息。因此 MVP 必须允许用户手动编辑，这是兜底机制。

**风险3：B实例在MVP阶段的最小形态**
如果 B 实例完全不做，任务摘要由 A 实例在收尾时写入——但 A 实例的上下文任务完成后就清空了，能否还保留足够信息写有质量的摘要？这是架构组需要回答的问题。
