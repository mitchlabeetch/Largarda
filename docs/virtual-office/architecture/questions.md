## Q1：记忆写入时机 — A实例收尾还是B实例触发？

**来源：** 郭聪明 §11 Q1

**背景：** 记忆系统是MVP命脉（P0），三实例架构中，实例B负责记忆整理，但MVP范围内B实例复杂度是否合理需要明确。

**需要架构回答的问题：**

1. 任务结束后自动写摘要 — 是A实例的收尾动作，还是独立B实例的工作？两种方案的token消耗差异是什么？
2. 如果A实例收尾写：任务完成后context还在吗？写摘要需要多少额外token？A实例上下文被压缩/截断后是否影响摘要质量？
3. 如果B实例写：触发时机（同步/异步）？是否阻塞用户下一个任务？B实例是常驻进程还是按需启动？
4. MVP最小可行方案建议：A实例写粗摘要 + B实例定期整理精摘要，是否可行？

**产品要求（已锁定）：** 记忆写入P0，B实例是否P0可商量，但写入不得延到P1。

**老尺关注点：**

- B实例MVP实现如果是"按需启动的LLM调用"，需要明确触发和失败重试逻辑
- A实例收尾写如果在压缩后执行，摘要质量有保障吗？

## Q2：记忆注入策略 — 全量注入还是按需召回？

**来源：** 郭聪明 §11 Q2 + 老乔 §16.3 问题A

**背景：** 记忆要生效，必须注入LLM上下文。但注入策略直接影响token消耗和性能。

**需要架构回答的问题：**

1. 单个员工的核心记忆文件多大时开始影响性能？建议单文件上限KB？总文件集合上限KB？
2. MVP选择哪种召回策略？
   - 方案A（全量注入）：所有记忆文件内容塞进系统提示词。简单，但员工记忆积累后上下文爆炸。
   - 方案B（关键词匹配）：用任务描述和记忆description做关键词重叠匹配，选top-N注入。
   - 方案C（LLM侧查询）：Claude的做法，用Sonnet小查询做语义相关性，选top-5。准确但每次多一次API调用。
3. 记忆注入是否有缓存机制？同一员工连续任务时，记忆文件未变化是否跳过重新加载？

**老乔建议（产品侧输入）：** MVP用全量注入 + 文件大小安全阀（总量超阈值触发压缩）。这是最简实现，规模大了再迭代。

**老尺关注点：**

- 全量注入方案需要明确"安全阀阈值"的触发逻辑：谁来压缩？压缩失败怎么处理？
- 方案C每次多一次API调用，成本和延迟需要评估是否在可接受范围内

## Q3：Spine授权和骨骼动画资产流水线

**来源：** 郭聪明 §11 Q3

**背景：** 用户已锁定PixiJS + Spine（P0），Spine商用需付费。

**需要架构回答的问题：**

1. Spine商用授权费用多少？是否在预算范围内？（Spine Essential $69，Professional更贵）
2. pixi-spine（Spine Runtime）在现有依赖栈（Electron + React + PixiJS）的集成复杂度？有无已知兼容问题？
3. 备选方案评估：
   - DragonBones：已停止维护，不推荐
   - Rive：授权模式不同，runtime免费但有文件格式限制；能否达到用户要求的"动作自然流畅"？
   - 自研骨骼：工期评估？
4. 骨骼资产获取路径：自绘（用户已裁决不买素材包），需要哪些工具链？预计资产制作工期？
5. 如果Spine授权搞不定，最接近P0标准的可行替代是什么？

**老尺关注点：**

- Spine授权是外部依赖，是MVP的硬阻塞项，必须在架构方案前确认
- 自绘资产的工期是否和MVP时间线兼容？如果资产没准备好，架构层怎么用占位动画开发？

## Q4：无限员工场景下PixiJS骨骼动画性能边界

**来源：** 郭聪明 §11 Q4 + 老乔 §15 裁决3

**背景：** 用户裁决员工无上限，但骨骼动画每个角色都有渲染成本，主场景可能同时展示大量角色。

**需要架构回答的问题：**

1. Electron桌面端，PixiJS渲染多少个骨骼动画角色时帧率开始下降到60fps以下？基准测试结果？
2. 建议的性能分级策略：
   - <N个：完整骨骼动画
   - N到M个：降低动画帧率/细节
   - > M个：静态+偶发动作 或 纯色块
3. 视口外员工动画是否暂停（Visibility Culling）？PixiJS中的实现方式？
4. 场景鼠标滚轮缩放+拖拽平移（类Figma画布）的技术方案？是否用PixiJS的Container缩放还是Camera概念？
5. 主场景员工卡片是PixiJS渲染还是DOM？骨骼动画在Canvas内，卡片信息（名字/状态文字）是否叠加DOM层？

**老尺关注点：**

- "员工无上限"是产品要求，但实现上必须有性能兜底。架构层需要给出明确的降级策略，不能在产品层虚假承诺
- Canvas + DOM混合层事件穿透问题（见Q6）

## Q5：员工状态推送 — 轮询还是IPC推送？

**来源：** 郭聪明 §11 Q5

**背景：** 员工状态实时性（<2秒）是产品要求，推送机制影响整体架构复杂度。

**需要架构回答的问题：**

1. P0阶段推荐轮询还是IPC事件推送？各自的延迟和实现复杂度比较？
2. Electron IPC通信（main process → renderer）的P99延迟大概多少ms？受员工数量影响大吗？
3. 多员工并发状态变化时（比如10个员工同时在执行任务），IPC消息是否会排队、丢失、或造成渲染抖动？
4. 推荐的状态数据模型：全量状态推送还是diff推送？
5. HEALTH/STRESS/PRODUCTIVITY的刷新频率（郭聪明定义：STRESS实时推送，HEALTH/PRODUCTIVITY每任务结束更新）是否能在现有IPC架构下满足？

**老尺关注点：**

- 轮询简单但延迟有下限，且多员工时可能造成UI抖动
- IPC推送实时，但需要设计消息队列防止堆积
- 产品要求员工状态更新<2秒，这个数字需要架构验证

## Q6：Canvas/DOM混层事件处理 — 聊天面板实现方式

**来源：** 郭聪明 §11 Q6

**背景：** 用户裁决聊天入口在电脑屏幕上（点击电脑屏幕打开包含沟通记录+任务状态的悬浮面板）。这个面板需要处理富文本输入，而员工角色是PixiJS骨骼动画。

**需要架构回答的问题：**

1. 悬浮面板（电脑抽屉：任务+工作记录+沟通记录）是PixiJS层渲染还是DOM层渲染？理由是什么？
2. PixiJS Canvas层和DOM层的点击事件处理优先级：用户点Canvas里的"电脑"家具，如何精确触发DOM层面板弹出？
3. Canvas层内的家具点击区域如何定义？是像素精确还是矩形碰撞盒？
4. "点空白区域关闭抽屉"（用户裁决8）在Canvas + DOM混层架构下如何实现？Canvas的点击事件能否冒泡到DOM层的全局close handler？
5. 聊天输入框在DOM层，骨骼动画在Canvas层，两者叠加时z-index和鼠标事件穿透怎么处理？
6. Electron webview下Canvas和DOM的合成方式是否有特殊限制？

**老尺关注点：**

- 混层架构是最容易产生事件冒泡/穿透bug的地方，需要在方案阶段明确分层边界
- PixiJS自带的交互系统（InteractionManager/EventSystem）和React DOM事件的共存方案是否有经过验证？

## Q7：三实例并发 — IPC桥接和进程隔离

**来源：** 老乔 §4.2 + 郭聪明 §8

**背景：** A（工作）+ B（记忆）+ C（社交）三实例并发是P0。当前AionUi架构是main process + renderer + worker，三实例和现有进程模型的映射关系需要明确。

**需要架构回答的问题：**

1. 三个LLM实例在Electron架构中如何分配？各跑在哪个进程？是否用Worker线程隔离？
2. 实例A工作时实例C同时接收用户消息，两个LLM stream并发，对现有的Claude API调用模块影响如何？
3. 实例B触发时（任务结束后异步写记忆），如何不阻塞实例A的下一个任务或实例C的对话？
4. 三实例共享同一个员工的工作空间文件（记忆文件），并发读写需要加锁吗？哪个实例有写权限？
5. 实例B失败（写记忆失败）的降级策略：是静默丢失、重试、还是提示用户？

**老尺关注点：**

- 三实例是产品核心机制，但实现上稍有不慎会变成"三个顺序执行的LLM调用"，失去并发价值
- 文件并发读写是常见的数据一致性问题，MVP需要有明确的互斥方案（哪怕是简单的文件锁）

## Q8：本地持久化存储方案

**来源：** 郭聪明 §4.2 + 老乔 §16.2

**背景：** 员工记忆文件本地持久化，App重启不丢失。当前AionUi用better-sqlite3做主数据库，但记忆文件是Markdown格式文件。

**需要架构回答的问题：**

1. 记忆文件（Markdown + frontmatter）存储在哪里？
   - 方案A：存在文件系统（类 `~/.claude/memory/`），路径由Electron主进程管理
   - 方案B：内容存SQLite，但保留Markdown格式供用户编辑
   - 方案C：混合 — 索引在SQLite，文件内容在文件系统
2. 用户可直接编辑记忆文件，编辑后如何同步回App的内存状态（热加载 or 手动刷新）？
3. 员工无上限时，磁盘占用的上限估算？是否需要配额管理？
4. 备份机制：`.bak`文件方案（老乔建议）的具体实现？还是有更好的方案？

**老尺关注点：**

- 文件系统方案对用户最透明（可用文本编辑器直接改），但需要App监听文件变更（fs.watch）
- SQLite方案更安全但用户直接编辑体验差
- 这个决策和"书架展示记忆文件"的UI实现紧密耦合，需要一起决策

## Q9：骨骼动画预览器（第二步优先任务）

**来源：** 用户裁决12 + 老尺独立识别

**背景：** 用户裁决第二步优先任务是搭建动画预览器，让用户能预览骨骼动画效果。这是阿构当前方案的主体。

**需要架构回答的问题：**

1. 预览器的进程归属：renderer进程，还是独立webview/window？
2. 预览器是否集成进现有的React路由体系，还是独立的Electron窗口？
3. 骨骼动画文件（.skel + .atlas + .png）的加载路径：放在renderer静态资源，还是通过主进程读取本地文件？
4. 占位/测试用骨骼资产从哪里获取？自绘还是使用Spine官方免费示例（spineboy等）？
5. 预览器需要支持哪些功能？（动作切换、速度调节、换肤测试等）最小可行集合是什么？

**老尺关注点：**

- 预览器是第二步的可交付验收物，范围必须明确，不能无限扩张
- 如果用Spine官方spineboy占位，需要确认其许可证是否允许开发用途
