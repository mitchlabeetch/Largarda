## 一、产品定位（一句话版）

**给AI Agent一个有灵魂的家，用户以养人的心态经营自己的AI团队。**

不是工具管理台，不是Agent调度器，是一个让用户"每天想打开看看"的养成游戏。

---

## 二、核心用户问题

产品要解决三个问题，按重要性排序：

1. **"我有一个AI助手，但感觉在用完即弃的工具，没有积累感"**
   → 解法：工作空间积累、记忆成长、让用户感受到"这个员工越来越懂我"

2. **"我想同时让AI帮我干活、又想随时问它问题，但两件事会互相打断"**
   → 解法：三实例机制，工作和聊天物理隔离

3. **"我不知道AI在做什么，不知道有没有卡住，不知道完成了什么"**
   → 解法：主场景上帝视角，HEALTH/STRESS/PRODUCTIVITY实时可见，异步通知

---

## 三、产品形态定义

### 3.1 整体结构

两个主要场景，对应两种使用模式：

```
虚拟办公室（主场景）
├── 上帝视角：所有员工一屏展示，看状态、扫进度、快速发任务
└── 员工工作空间（子场景）
    ├── 像素风房间：员工在里面活动
    ├── 交互入口：点击家具触发功能（电脑=任务、书架=记忆、通讯设备=聊天）
    └── 底部HUD：始终可用的输入栏
```

### 3.2 视觉风格定稿

**核心方向：Lofi/Cozy 暖色像素风。**

这是确定的，不是建议。用户参考图明确指向这个方向，和原来赛博朋克冷色调的设定冲突，后者需要放弃。

| 要素 | 具体要求                           |
| ---- | ---------------------------------- |
| 色调 | 暖色系为主：暖黄、橙调、木色、米白 |
| 光源 | 台灯、窗光为主，有温度感           |
| 氛围 | 舒适、有家的感觉、cozy             |
| 细节 | 植物、书、台灯、窗帘、小物件       |
| 窗外 | 日夜循环（跟随系统真实时间）       |
| 对比 | 不是高科技感，是温馨感             |

**日夜循环（必须上MVP或P1）：**

- 读取本地系统时间
- 窗外光线随时间渐变（清晨橙粉 → 白天亮蓝 → 黄昏橙红 → 夜晚深蓝）
- 室内随时间切换（白天自然光 → 傍晚台灯亮 → 深夜仅屏幕光）
- 开发成本相对合理（PixiJS滤镜 + 时间驱动颜色渐变）

### 3.3 主场景员工卡片

每张卡片展示：

- 员工名字 + 职位
- 状态色（绿=工作中 / 灰=空闲 / 橙=高压 / 红=出错）
- HEALTH / STRESS / PRODUCTIVITY 三条进度条（实时映射，见第六章）
- 当前在做什么（一句话，实时刷新）

用户从快到慢的信息获取路径：

1. 看状态色 → 0.5秒判断谁在干活、谁出问题
2. 看进度条 → 2秒判断员工状态健康度
3. 看状态文字 → 5秒了解具体内容

## 四、员工体系定稿

### 4.1 员工基础属性

| 属性       | 说明                       | 是否必填                   |
| ---------- | -------------------------- | -------------------------- |
| 名字       | 用户自定义                 | 是（起名是情感投入的起点） |
| 职位       | 从模板选择                 | 是                         |
| 头像       | 像素风预设角色             | 是                         |
| 个人简介   | 背景故事（影响提示词风格） | 否                         |
| 提示词基底 | 自动生成，高级用户可编辑   | 自动                       |

### 4.2 三实例机制（核心架构，不变）

| 实例          | 职责                     | 运行时机            |
| ------------- | ------------------------ | ------------------- |
| 实例A（工作） | 执行任务，调用工具       | 有任务时活跃        |
| 实例B（记忆） | 整理、压缩、归纳工作记忆 | 工作结束后/定时触发 |
| 实例C（社交） | 处理用户即时对话         | 用户发消息时        |

**MVP阶段：** 实例A + 实例B + 实例C 全部实现。员工记忆是用户核心价值第2项，不能延后。实例B的MVP最小实现：工作完成后触发一次压缩，摘要持久化到工作空间文件。不需要复杂的向量数据库，参考Claude的autoCompact机制（对话历史压缩成摘要写入文件）即可。

三实例通过工作空间文件系统协调，不做内存共享。

### 4.3 员工生命周期

```
招募 → 入职磨合 → 成长积累 → 协作配合 → 可能的调整 → 退休
```

每个阶段有情感信号，不是账号的增删改查。

**退休 ≠ 删除：**

- 退休后员工变灰，记录保留
- 提示"XX天内可以恢复"
- 真正物理删除有二次确认，说明"工作空间数据将一并清除"

### 4.4 模板设计（MVP三套）

MVP阶段内置三套模板，隐藏模块概念：

| 模板       | 能力倾向             | 预装工具（用户不感知） |
| ---------- | -------------------- | ---------------------- |
| 全栈工程师 | 代码、调试、技术分析 | 文件读写、代码工具     |
| 产品经理   | 需求分析、文档整理   | 文件读写、笔记工具     |
| 通用助手   | 万用型               | 基础工具集             |

**模块对用户隐藏（MVP原则）：**
用户通过模板创建员工，模块自动预装，用户不需要知道"模块"是什么概念。模块安装界面收进高级设置，只有主动探索的用户才会找到。

## 五、员工工作空间定稿

### 5.1 工作空间 = 像素风房间，不是面板

**这是最重要的设计决策之一。**

用户点击员工卡片，期待的是"推开房门，看到员工在做什么"，不是"打开个人档案页"。

工作空间必须是一个可进入的像素风房间，不是标签页+文件树的信息面板。

### 5.2 房间基础元素（MVP）

| 家具/物品    | 对应功能                    | 交互方式        |
| ------------ | --------------------------- | --------------- |
| 电脑         | 当前任务 + 工作记录         | 点击 → 抽屉     |
| 书架         | 记忆库 + 工作笔记           | 点击 → 抽屉     |
| 通讯设备     | 聊天记录                    | 点击 → 抽屉     |
| 员工角色本身 | 快捷菜单（发任务/查看状态） | 点击 → 浮层菜单 |

顶部固定状态栏：HEALTH / STRESS / PRODUCTIVITY / TIME / STATUS（始终可见）

底部固定HUD输入栏：发任务/发消息，始终可用，不依赖抽屉是否打开。

### 5.3 进入/离开工作空间的动画

- **进入（主场景 → 工作空间）：** 卡片展开成房间，400ms转场
- **离开（工作空间 → 主场景）：** 左上角"← 返回办公室"始终可见，点击返回

转场期间底部HUD已可用，不等动画结束。

### 5.4 工作空间内容权限

**记忆文件 MVP 就允许用户编辑（不能只读）。**

用户看到员工记忆里有错误信息，必须能手动修正。"只读"不是保护用户，是限制用户。加警告提示即可："直接修改记忆文件可能影响员工行为，确认修改？"

## 六、状态栏定稿（HEALTH / STRESS / PRODUCTIVITY）

**这三个数值不是装饰，是Agent实际运行状态的实时映射。**

| 数值         | 映射的Agent真实指标 | 如何计算                                         |
| ------------ | ------------------- | ------------------------------------------------ |
| HEALTH       | 实例稳定性          | 响应率、错误率、最近N次调用失败比例              |
| STRESS       | 任务负载            | 上下文使用率（占context window比例）+ 队列任务数 |
| PRODUCTIVITY | 产出效率            | 近期任务完成率 + 用户评分（如有）                |

**为什么比等级系统更值得做：**

- 等级/经验：时间积累的游戏化包装，和Agent实际能力不直接挂钩
- HEALTH/STRESS/PRODUCTIVITY：实时运行状态，数字有真实技术依据，用户可以验证

STRESS高 → 员工在超负荷，用户该暂停或清理队列  
HEALTH低 → 实例不稳定，需要干预  
PRODUCTIVITY低 → 任务完成质量有问题，可能需要调整任务描述

## 七、状态切换动画定稿（A/B/C分级方案）

### 7.1 用户给出的动画底线（2026-04-01确认）

**用户底线：状态枚举 + 退场/入场过渡，不需要寻路。**

这把技术复杂度大幅降低：

- 不需要A\*寻路算法
- 不需要实时坐标位置管理
- 不需要碰撞检测
- PixiJS原生能力足够，不强依赖Phaser

具体模型：每个状态对应一组固定动画帧（spritesheet），状态切换时播放退场动画→切换到新状态帧→播放入场动画。员工在房间内的位置是固定的（工作=在电脑前，休息=在床边，聊天=在通讯区），不需要实时移动计算。

---

### 7.2 A/B/C 三档方案

**用户已确认：骨骼动画是P0，MVP必须有。** A/B/C分级调整如下：

**A档（MVP底线）：骨骼动画 + 状态枚举 + 淡入淡出过渡**

- 骨骼动画（DragonBones/Spine），每个状态有对应的循环动作
- 状态切换：当前状态淡出（0.3s）→ 新状态淡入（0.3s）
- 角色在房间内位置固定，不移动
- 开发量：需要骨骼动画资产+PixiJS集成，但无寻路逻辑
- 体验效果：动作自然流畅，有状态感，无移动感

```
状态枚举（骨骼动画，每个状态生动循环）：

- WORKING  ：坐在电脑前，手指敲键盘（自然节奏循环）
- READING  ：坐在书架旁，手翻书页（翻页动作循环）
- SLEEPING ：躺在床上，胸部起伏呼吸（柔和舒缓循环）
- IDLE     ：随机小动作轮换——看手机滑屏 / 伸懒腰 / 发呆（多组随机切换）
- CHATTING ：面向用户，嘴部动作+手势（自然对话循环）
- THINKING ：坐在电脑前，托腮/视线游移（慢节奏循环）
- ERROR    ：停在当前位置，叹气+摊手（偶发触发，不持续循环）
```

**循环动画关键要求（三档通用，不可降级）：**

- 骨骼动画保证节奏自然，无卡顿
- IDLE至少3组不同小动作随机轮换，是"生命感"主要来源
- 每个循环动画末帧能自然接回首帧（无跳帧感）

**B档（P1）：骨骼动画 + 退场/入场过渡（无寻路）**

- 在A档基础上，状态切换时补充退场动作和入场动作
- 退场：播放起身/停止动作（0.3-0.5s）→ 淡出
- 入场：淡入 → 播放坐下/站定动作（0.3-0.5s）→ 进入循环
- 角色位置仍固定，不移动，但切换有明显的"结束-开始"仪式感
- 体验效果：状态切换更有层次感，比A档丝滑度明显提升

```
切换示例（IDLE→WORKING，B档）：
1. IDLE循环自然结束（或打断）
2. 播放起身骨骼动作（0.4s）
3. 淡出+淡入（位置切换到电脑桌前，0.3s）
4. 播放坐下骨骼动作（0.4s）
5. 进入WORKING循环
总时长：~1.4s
```

**C档（P2）：骨骼动画 + 真实位移行走**

- 在B档基础上，退场后角色实际走到目标位置（线性平移或预设路径点）
- 多方向行走动作（骨骼动画支持方向切换）
- 不需要A\*寻路，目标位置预设固定坐标
- 体验效果：员工有真实空间感，"走向"目的地

---

### 7.3 优先级与技术选型对应（已更新）

| 档位 | 优先级     | 技术栈              | 核心特征                        |
| ---- | ---------- | ------------------- | ------------------------------- |
| A档  | **P0/MVP** | PixiJS + 骨骼动画库 | 骨骼循环动画+淡入淡出，位置固定 |
| B档  | P1         | PixiJS + 骨骼动画库 | 退场/入场骨骼动作，位置固定     |
| C档  | P2         | PixiJS + 骨骼动画库 | 真实位移行走，无寻路            |

**产品立场（更新）：骨骼动画是MVP门槛，不可降级。** A档是MVP交付标准，B档是P1体验提升，C档是天花板。帧动画spritesheet不再作为备选——骨骼动画已由用户确认为P0必须。

---

### 7.4 产品对动画的核心要求（三档通用）

无论选哪档，以下是产品层面的硬性要求：

1. **状态切换期间交互不锁定**：动画播放中用户仍可点击发任务/聊天
2. **动画不阻塞功能**：后台实例状态变化不等待动画结束
3. **第一次进入房间**：员工直接在当前状态位置，不播放完整入场序列
4. **动画可跳过/加速**：频繁状态切换时压缩动画时长，不积压动画队列

## 八、成长体系裁决

### 8.1 保留什么

**保留 HEALTH / STRESS / PRODUCTIVITY（第六章），这是真实映射，必须做。**

**保留成长感的产品呈现：**

- 工作空间内容随时间积累（笔记增多、记忆文件增厚）
- 员工引用过去经验（"上次你让我做XX时，我发现..."）
- 任务完成数量、工作天数简单统计（展示在卡片上）
- 定期"工作报告"：员工总结自己近期做了什么（P1，但留位置）

### 8.2 砍掉什么

| 功能                     | 决策   | 原因                                       |
| ------------------------ | ------ | ------------------------------------------ |
| 等级数字（Lv.X）         | 砍掉   | 底层LLM能力不随等级变，用户测试会戳穿      |
| 经验值系统               | 砍掉   | 和任务完成数量统计重复，且有"虚假积累"风险 |
| 能力树（角色专属分支）   | 降到P2 | 维护成本高，通用节点验证后再扩展           |
| 技能系统（/技能名 触发） | 降到P2 | 本质是预设提示词，意义有限                 |
| 成就系统                 | 不做   | 装饰性功能，没有解决用户真实问题           |

**核心判断：** 养成要么做真实的（可验证地影响输出质量），要么不做数值化包装。同一道题问Lv.1和Lv.20的员工，如果没有差异，用户会直接在社交媒体说"这是骗局"。

## 九、MVP边界确认

### 9.1 必须做（Hard Must Have）

| 功能                                 | 为什么                                             |
| ------------------------------------ | -------------------------------------------------- |
| 像素风主场景（lofi/cozy暖色调）      | 门槛，不是加分项。打开和普通SaaS一样，产品定位失败 |
| 员工有物理位移的状态切换             | 核心情感差异化，没有这个员工是"动态壁纸"           |
| HEALTH/STRESS/PRODUCTIVITY实时状态栏 | 用户上帝视角的核心信息源                           |
| 至少3个内置模板（模块对用户隐藏）    | 没有创建就没有体验                                 |
| 工作实例+社交实例并发（实例A+C）     | 聊天不打断任务，是差异化技术基础                   |
| 员工工作空间 = 像素风房间            | 进入员工空间是"推门进房间"，不是打开档案页         |
| 任务失败通知 + 异步通知机制          | 用户不守着等，必须有通知才能离开                   |
| 员工记忆文件可查看+可编辑            | 可查=工作空间有内容；可编辑=用户有控制权           |

### 9.2 不做（Hard No）

| 功能                             | 理由                                         |
| -------------------------------- | -------------------------------------------- |
| 等级/经验值/成就系统             | 假的成长感比没有更伤用户信任                 |
| 程序员模块（P0降到P1）           | 需要沙箱安全机制，技术门槛比其他P0高一个量级 |
| 多员工协作任务                   | 单员工体验还没打磨好，多员工协作是P1         |
| 工作空间全文搜索                 | 内容积累之前搜索没意义，P1                   |
| 模板导出分享                     | 社交功能是扩张用的，留存先做好               |
| 向量索引/语义搜索                | 额外基础设施，关键词搜索P1够用               |
| 虚拟桌面UI（办公软件模块可视化） | 工具能力P1再上，视觉包装是P2                 |
| 自定义提示词编辑器（主流程暴露） | 高级用户功能，隐藏在高级设置                 |

### 9.3 MVP通关标准

**定性（感受，但是硬标准）：**

- 5个不熟悉AI工具的用户测试，至少3个说"这也太酷了"
- 第一眼，测试用户会停顿至少3秒（在认真看）
- 完成第一次任务分配后，至少1个用户主动问"还能做什么？"

**定量：**

- 员工创建流程 ≤ 3步
- 员工状态更新延迟 < 2秒（实际状态变化到UI更新）
- 第一次打开到开始第一次交互 < 5分钟

## 十、第一次打开（空状态）的用户引导

**郭聪明5份PRD都没有覆盖这个场景，这是转化率最关键的漏斗节点。**

### 10.1 空状态设计原则

空不等于无聊。空在暗示："**你来填满它**"。

第一次打开时：

1. **进入动画：** 像素风空办公室场景，灯光逐渐亮起，有一张空桌子在等待
2. **引导词：** 不是"点击创建员工"，而是"你的团队还没人——要招募第一个成员吗？"
3. **模板展示：** 不是表格，是"求职卡"——每个模板角色有一句话自我介绍 + 小动画
4. **默认只展示3个模板**，其他折叠在"更多"里（超过5个用户会选择困难）
5. **一键创建：** 员工"走进"办公室，有入职动画

### 10.2 新员工入职磨合（前7天/前10次交互）

- 新员工卡片有"首次任务建议"（根据角色类型预设，选容易成功的，不是最复杂的）
- 完成第一个任务后，有明显的成就感反馈："[员工名] 完成了第一次任务"
- 工作空间开始积累内容，用户能看到员工"在成长"

用户容易在入职阶段流失。如果前几次交互没有成果感，用户会认为这个产品没用。

## 十一、遗漏场景（必须在PRD中覆盖）

以下场景郭聪明的草案没有覆盖，开发前必须有答案：

### 11.1 员工空转处理

用户给了任务但描述不清，员工开始工作但方向错了，一直在消耗token，用户完全不知道。

**需要决策：**

- 员工任务中途需要澄清时，是主动发消息问用户，还是自己决策继续？
- 长时间运行的任务（>X分钟）有没有成本警告？
- token消耗达到阈值有没有自动暂停机制？

### 11.2 任务中途发现能力缺口

员工在执行任务过程中发现某个工具没有安装（比如需要git但程序员模块没装）。

**需要决策：**

- 员工主动告知用户"我没有这个能力，需要安装XX功能"？
- 还是任务失败后用户才知道？
- 前者更好，后者让用户感觉被骗

### 11.3 关机后的任务状态

桌面应用意味着员工只在用户开机时工作。用户关电脑后，正在进行的任务怎么办？

**需要决策：**

- 任务被挂起，下次开机继续（需要任务状态持久化）？
- 还是超时失败，明确提示？
- 这个约束对"员工真的在工作"的感知影响很大

### 11.4 多员工写同一文件（P1需要解决）

多员工协作时，可能多个工作实例同时写工作空间里的同一个文件。

**MVP简化方案（可参考）：** 实例A写操作记录，实例B读+归纳，时序隔离，不真正并发写同一文件。多员工协作是P1，届时需要文件锁或版本机制。

## 十四、老乔判断总结

**产品方向是对的。** 养成游戏感 + AI Agent实际工作能力的结合，是有差异化的。

**最大的风险是：做了很多功能，但没有一件事做到位。**

MVP的关键不是功能数量，是两件事：

1. 第一眼就"哇"（视觉+动画，门槛不是加分项）
2. 员工真的完成了一件事（功能闭环，不能只是好看）

这两件事都做到了，用户明天会回来。有一件事没做好，用户就不会回来，加再多功能也没用。

---

_老乔最终判断：宁可砍掉一半功能，也要把剩下的做到真实惊艳。_
_等级系统不做，成长感还在。做了假等级系统，成长感反而归零。_

### 裁决2：记忆系统 — 核心中的核心，必须持久化，P0 MVP 命脉

**用户原话：** 核心中的核心，必须持久化，参考 Claude 记忆系统，MVP 命脉。

**落地说明：**

实例 B（记忆实例）是 P0，不是 P1。老乔已研究 Claude 记忆系统源码（`~/.claude/projects/`），提炼如下关键机制，**直接映射到员工记忆的技术实现**：

**Claude 记忆系统核心机制（源码级）**

1. **文件结构**：`MEMORY.md`（索引）+ 多个主题记忆文件（如 `user_role.md`、`feedback_testing.md`）。索引是轻量目录，不存实际内容；内容在独立文件里，通过 frontmatter（`name`/`description`/`type`）标注。

2. **四种记忆类型**：`user`（用户画像）、`feedback`（协作偏好）、`project`（项目上下文）、`reference`（外部资源指针）。类型决定读取策略和使用场景，不是装饰。

3. **按需召回**：不是把所有记忆一次性塞进上下文。`findRelevantMemories` 函数在每次对话开始时，用一个轻量的 Sonnet 侧查询扫描记忆文件的 frontmatter（description 字段），根据当前 query 选出最多 5 条相关记忆，再加载到上下文。这解决了"记忆多了上下文爆炸"的问题。

4. **鲜度感知**：每条记忆有 `mtimeMs`，超过1天的记忆会附带 staleness caveat（"这条记忆是 X 天前的，code 引用可能过时"）。避免陈旧记忆被当作事实断言。

5. **200 条上限**：`MAX_MEMORY_FILES = 200`，超出后按 newest-first 排序截断。

6. **日志模式（KAIROS）**：长期 session 下，不维护 MEMORY.md，而是按日期 append-only 写日志（`logs/YYYY/MM/YYYY-MM-DD.md`）。夜间有独立进程把日志蒸馏成 MEMORY.md + 主题文件。这个模式对员工长期记忆积累有参考价值。

**员工记忆系统映射（产品层设计）**

| Claude 机制       | 员工记忆等价物                                      | 产品呈现                                       |
| ----------------- | --------------------------------------------------- | ---------------------------------------------- |
| `MEMORY.md` 索引  | `核心记忆.md`（员工工作空间根文件）                 | 书架点击后首页展示的记忆索引                   |
| 主题记忆文件      | `任务记录_20260401.md`、`用户偏好.md` 等            | 书架子文件，可点击查看+编辑                    |
| `type: feedback`  | 用户对员工的偏好记录                                | "你上次说我的代码风格太啰嗦"                   |
| `type: project`   | 正在进行的项目上下文                                | 员工主动引用："上次那个 PR 我注意到..."        |
| 按需召回（top 5） | 每次对话前选出最相关的 3-5 条记忆注入上下文         | 用户不感知，员工表现为"记得事情"               |
| staleness caveat  | 记忆文件超过 N 天未更新时，员工在引用时自动标注     | "我记得之前...不过那是3周前的事了，你确认一下" |
| KAIROS 日志模式   | 每次任务结束写一条日志，定期（每日/每周）压缩成摘要 | 书架积累感的技术基础                           |

**MVP 最小实现（开发组可按此交付）：**

```
实例 B 触发时机：
  - 任务完成后自动触发（P0）
  - 对话结束时自动触发（P0）
  - 用户手动点"整理记忆"按钮（兜底）

实例 B 执行逻辑：
  1. 读取本次工作实例(A)的对话历史
  2. 调用 LLM 压缩成结构化摘要
     - 用了什么工具
     - 完成/失败了什么
     - 发现了什么用户偏好或项目信息
  3. 分类写入对应记忆文件
     - 新信息 → 追加或新建主题文件
     - 与现有记忆矛盾 → 更新旧文件，标注"已更新"
  4. 更新 核心记忆.md 索引

不需要：向量数据库、语义搜索、嵌入模型
需要的：文件读写 + 一次 LLM 调用 + 简单 frontmatter 解析
```

### 裁决3：员工数量 — 不限

**用户裁决：** 员工数量不限。

**产品侧影响分析：**

这个裁决对主场景展示方式有重大影响，需要明确设计策略。

**场景 A：员工少（1-5人）**
主场景可以一屏展示所有员工卡片，和当前设计一致。无需特殊处理。

**场景 B：员工多（6-20人）**
主场景必须支持横向/纵向滚动，或者分组/分页。老乔建议：**网格布局+滚动，不分页**。分页会打断上帝视角的"一眼扫全局"体验。

**场景 C：员工超多（20+人）**
主场景面临性能和信息密度双重问题。

**老乔建议（供架构组参考）：**

| 员工数 | 主场景策略                                               | 技术要求                       |
| ------ | -------------------------------------------------------- | ------------------------------ |
| 1-5    | 固定网格，全部展示                                       | 无特殊要求                     |
| 6-15   | 网格+滚动，状态色优先                                    | 虚拟滚动或直接渲染（数量有限） |
| 16-30  | 网格+滚动+搜索/过滤                                      | 按状态/职位过滤                |
| 30+    | 需要"归档"机制：不活跃员工自动归档，主场景只显示活跃员工 | 归档不是删除，是"下线"         |

**归档机制（建议 P1 但需要 P0 预留入口）：**

- 超过 N 天无交互的员工自动提示"是否归档？"

- 这解决了"员工不限"带来的主场景拥挤问题，同时不强迫用户删除员工

**性能预判：**
每个员工卡片有骨骼动画（P0），30+ 员工同时渲染动画会有性能问题。架构组需要考虑：

- 只有在视口内的员工才播放完整动画
- 滚动到视口外的员工暂停动画/降级为静态
- 这是标准的可见性优化，PixiJS 支持

## 十六、Claude 记忆系统研究总结 — 为架构组准备的产品输入

> 本章专门为第二步（架构设计）输出产品侧需求，让架构组有清晰的技术选型依据。

### 16.1 员工记忆的核心产品需求

**用户要的不是"记住了什么"，而是"感觉这个人越来越懂我"。**

这个感觉来自三个可观测时刻：

1. 员工在对话中主动引用过去的事（"上次你说..."）
2. 用户在书架里看到记忆文件在增厚
3. 员工处理类似任务时，明显比第一次做得更好

技术上需要支撑的：

```
① 记忆写入：任务完成后 → 结构化压缩 → 写入工作空间文件
② 记忆读取：每次对话前 → 按相关性召回 top-N 条记忆 → 注入上下文
③ 记忆呈现：书架抽屉 → 文件列表 + 内容预览 + 可编辑
```

### 16.2 记忆文件结构建议（产品规格）

```
员工工作空间/
├── 核心记忆.md              ← 索引，类比 MEMORY.md
├── 用户偏好.md              ← type: feedback（用户对该员工的偏好）
├── 项目上下文.md            ← type: project（正在进行的项目信息）
├── 工具使用记录.md          ← type: reference（员工已熟悉的工具/API）
└── 日志/
    ├── 2026-04/
    │   ├── 2026-04-01.md   ← 每日日志（append-only）
    │   └── ...
    └── ...
```

**核心记忆.md 格式（索引）：**

```markdown
# [员工名] 核心记忆

- [用户偏好](用户偏好.md) — 代码风格偏好、沟通方式偏好
- [项目上下文](项目上下文.md) — 当前进行中的项目和目标
- [工具记录](工具使用记录.md) — 已掌握的工具和注意事项
```

**每条记忆文件格式（frontmatter）：**

```markdown
---
name: 用户偏好
description: 用户对该员工的工作风格偏好和反馈
type: feedback
updated: 2026-04-01
---

## 代码风格

- 偏好简洁，反对冗余注释
- 函数命名用中文拼音不接受

## 沟通方式

- 不喜欢被打断问"请问您是否需要..."
- 任务完成直接说结果，不总结过程
```

### 16.3 关键技术决策（架构组必须明确）

**问题 A：记忆召回用什么策略？**

Claude 的做法是用一个轻量 LLM 侧查询（Sonnet，max_tokens=256）做相关性匹配，读 description 字段，选 top-5。

员工记忆召回可以有三种策略：

1. **全量注入**（简单但粗暴）：每次把所有记忆文件塞进上下文。员工少/记忆少时可行，规模大了上下文爆炸。
2. **关键词匹配**（不够准）：当前任务描述和记忆 description 做关键词重叠匹配。简单，但会漏掉语义相关的记忆。
3. **LLM 侧查询**（Claude 的做法）：用一个小模型做语义相关性判断，选 top-N。准确，但每次对话多一次 API 调用。

**老乔建议：MVP 用全量注入（简单）+ 文件大小上限（安全阀）。** 记忆文件总大小超过阈值时触发压缩（类似 Claude 的 autoCompact）。规模大了再迭代成 LLM 侧查询。

**问题 B：记忆写入频率？**

Claude 是"用户明确要求时写"。员工记忆需要自动写，触发时机需要明确：

- 任务完成时（必须）
- 对话超过 N 轮时（可选）
- 用户明确说"帮我记住这个"时（必须）
- 定时（每天收工时压缩一次日志）（P1）

**问题 C：记忆文件对用户可编辑，错误修改怎么处理？**

Claude 的设计没有回滚机制——它是纯文件写入，靠文件系统版本管理（git）做回滚。

员工记忆建议同样不做复杂回滚，但要：

- 编辑时有警告提示："直接修改记忆可能影响员工行为，确认修改？"
- 保留最近一次备份（`.bak` 文件，下次写入时覆盖旧备份）

### 16.4 "员工不限数量"对记忆系统的影响

每个员工有独立的工作空间目录，记忆文件互不干扰。技术上无问题。

主要影响是磁盘占用和性能：

- 员工多，记忆文件多，启动时扫描所有员工记忆需要时间
- 建议：**懒加载**。只有打开某个员工的工作空间时，才加载该员工的记忆文件
