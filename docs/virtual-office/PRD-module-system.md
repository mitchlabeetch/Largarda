# PRD-module-system — 功能模块系统

## 产品定位提醒

模块是员工的"装备"，不是后台配置项。用户给员工安装程序员模块，就像给游戏角色装备武器——要有仪式感，要能感受到"这个员工变强了"。

---

## 1. 功能描述

### 1.1 模块的本质

模块 = 一组 Claude Tools 的封装 + 对应的提示词补丁 + 可选的虚拟UI界面。

当员工安装某个模块后：

1. 对应的工具集注入到工作实例的系统提示词
2. 员工的能力边界扩展（例如：可以操作 git、可以管理员工）
3. 工作空间可能出现新的虚拟界面（例如：程序员模块的代码编辑器视图）

**重要：面向用户的两条路径（响应老乔质疑3）**

| 路径                         | 适用用户             | 体验                                                                          |
| ---------------------------- | -------------------- | ----------------------------------------------------------------------------- |
| **模板路径（隐式安装）**     | 99% 用户             | 选择"全栈开发工程师"模板 → 员工创建时程序员模块自动预装，用户不感知"模块"概念 |
| **高级设置路径（显式安装）** | 有特定需求的高级用户 | 员工页面→高级设置→模块管理→手动安装/卸载                                      |

大多数用户不需要知道"模块"是什么。模板路径是主干路径，模块安装界面是隐藏的高级功能。US-101（"我想给开发员工安装程序员模块"）本身是错误的用户故事——用户想要的是"一个能写代码的员工"，而不是"执行模块安装操作"。

### 1.1.1 工具注入策略（源码洞察）

**来源：Claude 源码 Tool.ts + ToolSearchTool**

Claude 本身的 Tool 系统提供了一个关键机制解决提示词爆炸：**延迟加载（Deferred Tools）**。

Claude 的做法：

- 工具分为两类：`alwaysLoad=true`（始终注入，每次调用都看得见）和 `shouldDefer=true`（延迟加载，默认不注入）
- 延迟工具只在系统提示词中出现**名称**，不注入完整 JSON Schema
- LLM 需要某个工具时，先调用 `ToolSearch` 获取完整 schema，才能调用该工具
- 这样大量工具的 schema 不占用主上下文窗口

**虚拟办公室的应用：**
员工的工具注入应参照同样的分层策略：

| 工具类型             | 注入方式                               | 示例                         |
| -------------------- | -------------------------------------- | ---------------------------- |
| 核心工具（高频必用） | 始终注入完整 Schema                    | file_read、doc_create        |
| 模块工具（按需使用） | 延迟注入：只注入名称，按需加载完整定义 | git_commit、employee_dismiss |
| 危险工具             | 延迟注入 + 需要用户确认                | file_write、terminal_run     |

这样即使员工安装了5个模块（50+个工具），系统提示词中只有常用工具的完整 schema，其余只有名称列表，显著降低 token 消耗。

另一个关键设计：Claude 的 `ToolUseContext` 中有 `refreshTools?: () => Tools`，支持工具集在对话中途动态刷新（例如 MCP server 中途连接）。这意味着模块的安装/卸载可以在不重启员工实例的情况下生效，是重要的工程边界确认。

### 1.2 程序员模块

**定位：** 让员工具备完整的代码开发能力，是最重要的功能模块。

**包含工具：**

| 工具                | 功能              | 说明                  |
| ------------------- | ----------------- | --------------------- |
| git_status          | 查看仓库状态      |                       |
| git_diff            | 查看代码变更      |                       |
| git_commit          | 提交代码          | 需要用户配置授权      |
| git_push / git_pull | 同步远程仓库      |                       |
| git_branch          | 分支管理          |                       |
| file_read           | 读取代码文件      |                       |
| file_write          | 写入/修改代码文件 | 危险操作，需确认机制  |
| file_search         | 在仓库内搜索      |                       |
| terminal_run        | 执行终端命令      | 沙箱限制，白名单命令  |
| code_review         | 代码审查分析      | 纯 LLM 能力，无需工具 |

**房间视觉变化（模块安装仪式感）：**
安装程序员模块后，员工房间里新增：

- 第二块显示器（多屏，桌上）
- 服务器机箱（角落装饰）

两件道具有入场动效（从场景外滑入，400ms ease-out）。卸载模块时出场消失（300ms ease-in）。不允许静默刷新，必须有动效。

**第二块显示器 hitbox 行为（已裁决：合并到主显示器抽屉）：**

- 点主显示器 → 打开电脑抽屉，停留在上次打开的 Tab
- 点第二显示器 → 打开电脑抽屉，**直接跳到 [代码日志] Tab**（快捷入口）
- 两个 hitbox 触发同一个抽屉，不存在两个抽屉同时打开
- 未安装程序员模块时 [代码日志] Tab 不出现，第二显示器也不存在

**虚拟界面（代码终端日志 Tab）：**
装了程序员模块后，电脑抽屉从 2 个 Tab 变 3 个 Tab：**[任务] [沟通] [代码日志]**。
[代码日志] Tab 内容：员工正在操作哪些文件、执行什么命令（只读监控，不是真实终端）。

**渲染方式：DOM 层右侧抽屉**，不在 Canvas 内绘制像素风终端。终端日志是文字内容，用户需要阅读代码，DOM 渲染清晰度远优于 Canvas 绘制。

**安全边界：**

- `file_write` 和 `terminal_run` 默认需要用户确认，除非用户配置了"信任该员工"模式
- 禁止跨仓库访问（员工只能访问授权的仓库路径）

### 1.3 管理模块

**定位：** 让员工（通常是领导角色）具备管理其他员工的能力。安装此模块的员工可以代替用户管理团队。

**工具分两层（P0 基础协作 + P1 完整管理）：**

#### P0 — 基础协作工具（用户核心价值第3项：员工协作）

用户明确：员工协作是P0核心功能，不是加分项。即使没有完整管理模块界面，以下两个工具必须在MVP可用：

| 工具                 | 功能               | 说明                     |
| -------------------- | ------------------ | ------------------------ |
| employee_assign_task | 给指定员工分配任务 | 跨员工协作的基础，P0必须 |
| message_send         | 向其他员工发消息   | 员工间异步通信，P0必须   |

**P0 最小实现路径：** 领导员工（或任意员工）的系统提示词中注入这两个工具，不需要完整的管理模块安装界面。用户直接在HUD输入框发指令，领导员工调用这两个工具分发任务给其他员工。

#### P1 — 完整管理工具集

| 工具                   | 功能                           | 说明                     |
| ---------------------- | ------------------------------ | ------------------------ |
| employee_list          | 列出所有员工及状态             |                          |
| employee_create        | 创建新员工                     | 需要指定模板             |
| employee_dismiss       | 解雇员工（停止实例，归档数据） | 危险操作，需用户二次确认 |
| employee_status_update | 修改员工状态（休眠/激活）      |                          |
| team_summary           | 生成团队工作总结报告           |                          |

**约束：**

- 安装管理模块的员工不能解雇自己
- 解雇操作必须触发用户确认弹窗，AI 无法绕过
- 管理模块通常只分配给"领导角色"模板

### 1.4 办公软件模块

**定位：** AI 眼中的虚拟办公工具集，覆盖日常协作场景。通过动态注入实现虚拟界面。

**包含工具：**

| 工具               | 功能               | 说明             |
| ------------------ | ------------------ | ---------------- |
| doc_create         | 创建文档/笔记      | 写入员工工作空间 |
| doc_read           | 读取文档           |                  |
| doc_search         | 全文搜索文档       |                  |
| calendar_view      | 查看任务日历       |                  |
| calendar_add       | 新增日程/提醒      |                  |
| message_send       | 发送消息给其他员工 | 员工间异步消息   |
| message_read       | 读取消息           |                  |
| web_search         | 网络搜索           | 接入搜索引擎 API |
| screenshot_capture | 截图分析           | 多模态能力       |

**虚拟界面：**
员工工作空间内出现"办公桌面"视图，模拟一个简化的桌面环境，展示当前打开的文档、日历等（动态注入的假UI，实际是 LLM 工具调用的可视化）。

### 1.5 其他模块（Claude Tools 集成）

**定位：** 将 Claude 源码中的其他 tools 封装为可安装模块，扩展员工能力上限。

**实现思路：**

- 解析 Claude 工具定义格式（JSON Schema）
- 封装为标准模块接口
- 提供"模块市场"入口（P2）

**候选工具集：**

- 浏览器操作工具（Puppeteer/Playwright）
- 数据库查询工具
- API 调用工具
- 文件系统工具（比程序员模块更通用）

## 2. 用户故事

**US-101（P0）** 作为用户，我想给开发员工安装程序员模块，这样他可以真正帮我写代码、提交PR，而不只是给建议。

**US-102（P1）** 作为用户，我想看到员工正在用哪个工具做什么（例如：正在执行 git diff），这样我有掌控感，不是黑箱。

**US-103（P1）** 作为用户，当员工要写文件或执行终端命令时，我想收到确认提示，这样我不担心 AI 做破坏性操作。

**US-104（P1）** 作为领导员工的用户，我想让领导员工自动分配任务给其他员工，这样我可以只和领导沟通，不用一一对接。

**US-105（P2）** 作为用户，我想在模块市场找到更多工具模块安装给员工，这样可以随意扩展员工能力。

## 3. 验收标准

### P0

- [ ] 程序员模块可正确注入 git 工具到员工系统提示词
- [ ] 员工安装/卸载模块后，其工具能力立即生效/失效
- [ ] 每个模块有清晰的能力描述，用户安装前知道"装了之后员工能干什么"
- [ ] 基础协作：具备领导角色的员工可调用 `employee_assign_task` 给其他员工分配任务
- [ ] 基础协作：员工可调用 `message_send` 向其他员工发送消息（异步写入目标员工工作空间）
- [ ] 用户在HUD输入框给领导员工发指令后，领导员工能自动拆分并通过工具分发给下属员工

### P1

- [ ] 员工执行工具调用时，工作空间界面有实时操作日志展示
- [ ] `file_write` 和 `terminal_run` 执行前弹出用户确认弹窗
- [ ] 管理模块的"解雇员工"必须二次确认，且操作日志保留
- [ ] 办公软件模块的虚拟桌面界面至少支持文档创建和查看

### P2

- [ ] 模块市场展示所有可用模块，支持按类型过滤
- [ ] 支持自定义工具（用户提供 JSON Schema 定义）
- [ ] 员工间消息传递可在工作空间聊天记录中可视化查看

## 4. 优先级划分

| 功能                                              | 优先级             | 原因                                                                                      |
| ------------------------------------------------- | ------------------ | ----------------------------------------------------------------------------------------- |
| 程序员模块（基础git工具）                         | **P1**（从P0降级） | 需要沙箱安全机制，技术门槛比其他P0高一个量级。MVP阶段先验证员工交互闭环，P1再上程序员模块 |
| 模块安装/卸载机制                                 | P0                 | 架构地基，后续模块依赖此机制（但面向用户隐藏，只走模板路径）                              |
| 工具调用可视化日志                                | P1                 | 信任感来源——用户需要知道员工在干什么                                                      |
| 危险操作确认机制                                  | P1                 | 安全红线，不能省                                                                          |
| 管理模块 — 基础协作（assign_task + message_send） | **P0**             | 用户确认：员工协作是P0核心，最小实现不需要完整管理模块界面                                |
| 管理模块 — 完整管理工具集                         | P1                 | 完整团队管理体验（创建/解雇/状态管理）                                                    |
| 办公软件模块                                      | P1                 | 通用场景覆盖                                                                              |
| 模块市场                                          | P2                 | 扩展性，但MVP不需要                                                                       |

## 5. 风险与歧义

**风险1：提示词爆炸（已有解法，但需要正确执行）**
Claude 源码（Tool.ts + ToolSearchTool）已经给出了明确解法：**Deferred Tools 延迟加载机制**。工具按 `alwaysLoad` / `shouldDefer` 分层，只有高频核心工具全量注入，其余工具只注入名称，LLM 需要时再 `ToolSearch` 拉取完整 Schema。

对虚拟办公室而言，产品层面的要求是：架构组在设计工具注入时，必须强制要求每个模块的工具声明其加载优先级（核心 vs 延迟），不能所有工具默认全量注入。这是功能模块数量能无限扩展的前提。

**风险2：安全边界**
`terminal_run` 工具有极高风险，沙箱机制是必须。需要架构组明确：是 Docker 沙箱、进程隔离还是命令白名单？

**风险3：办公软件模块的"虚拟界面"实现复杂度**
原始需求提到"通过动态注入技术实现虚拟界面"，这在技术上需要大量前端工作。建议 P1 阶段先实现工具能力，虚拟界面作为 P2 增强。

**歧义1：员工间消息传递是实时的还是异步的？**
如果是实时，需要员工实例间的消息队列；如果是异步，复杂度低很多。建议先做异步（写入共享工作空间），P2 再升级实时。

**歧义2：管理模块安装在哪个员工身上？**
原始需求没说清楚管理模块是内置的（每个员工都有）还是只有特定角色才有。建议：管理模块只对"领导角色"模板开放，普通员工不能安装，避免混乱。

**歧义3（新增）：任务中途发现能力缺口——员工该怎么做？（响应老乔追问2）**

场景：员工接到任务"帮我提交代码"，但没有安装程序员模块，执行到一半发现用不了 git 工具。

产品定义：

- 员工应该**主动停下并告知用户**："我发现这个任务需要代码提交能力，但我目前没有程序员模块。你可以在高级设置里给我安装，或者告诉我换一种方式完成。"
- 不允许静默失败（任务失败但用户不知道原因）
- 员工不能自行安装模块（安全边界）
- 这个场景通过模板路径可以大幅减少：模板已经预装了对应模块，用户不太可能走到"已分配任务但缺少工具"的状态

验收标准（P1）：员工在发现工具缺口时，主动暂停任务并发消息给用户，说明缺少哪个能力以及解决方式。
