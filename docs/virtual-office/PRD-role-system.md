# PRD-role-system — 角色系统

## 产品定位提醒

这不是 SaaS 工具里的"账号管理"。每个员工是有灵魂的游戏角色，有名字、外貌、个性、成长轨迹。用户第一次创建员工时，感受应该是"我在招募一个队友"，而不是"我在填表"。

---

## 1. 功能描述

### 1.1 员工基础信息

每个员工（Employee）是虚拟办公室的最小单位，包含：

| 字段       | 说明                               | 必填             |
| ---------- | ---------------------------------- | ---------------- |
| 名字       | 用户自定义，支持中英文             | 是               |
| 职位       | 从预设角色模块选择，支持自定义描述 | 是               |
| 头像       | 像素风角色外形，支持多种预设       | 是               |
| 个人简介   | 员工的背景故事/性格描述            | 否               |
| 提示词基底 | 角色核心人设提示词，决定其行为风格 | 自动生成，可编辑 |

### 1.2 三实例机制（核心架构）

每个员工内部由最多 3 个 Agent 实例并发驱动，分工明确：

| 实例   | 名称     | 职责                           | 运行时机             |
| ------ | -------- | ------------------------------ | -------------------- |
| 实例 A | 工作实例 | 执行实际任务，调用功能模块工具 | 有任务时活跃         |
| 实例 B | 记忆实例 | 定期整理、归纳、压缩工作记忆   | 工作结束后、定时触发 |
| 实例 C | 社交实例 | 处理用户聊天、即时响应         | 用户发起对话时       |

三个实例共享员工的工作空间（笔记/记忆文件），但各自维护独立的上下文窗口，防止干扰。

**为什么这样设计：** AI 的上下文窗口有限，边干活边聊天会导致工作上下文被稀释。三实例分工让员工真正能"一边干活一边聊天"，而不是假装多任务。

**源码洞察（Tool.ts / ToolUseContext）：** Claude 的 `ToolUseContext` 中有 `agentId?: AgentId` 区分子 Agent，而 `setAppStateForTasks` 专门为后台 Agent 设计（"session-scoped infrastructure that outlives a single turn"）。三实例架构在 Claude Agent SDK 层面完全可行：

- 工作实例（A）和社交实例（C）是普通子 Agent，各自维护独立对话上下文
- 记忆实例（B）是**后台任务 Agent**，不绑定用户对话生命周期，工作结束后异步触发
- 三个实例通过**文件系统**协调（共享工作空间文件），不做内存共享，天然避免状态竞争

### 1.3 角色模块（职位模板）

预设角色类型，决定员工的基础能力倾向和提示词风格：

| 角色类型   | 能力倾向                   | 可接入功能模块         |
| ---------- | -------------------------- | ---------------------- |
| 开发角色   | 代码、调试、技术分析       | 程序员模块             |
| 设计角色   | UI审查、视觉判断、创意     | 办公软件模块           |
| 产品角色   | 需求分析、优先级判断、文档 | 办公软件模块、管理模块 |
| 领导角色   | 任务分配、团队协调、决策   | 管理模块               |
| 自定义角色 | 用户完全自定义             | 任意模块               |

### 1.4 员工模板（一键创建）

模板 = 角色模块 + 预设功能模块 + 预设提示词基底的组合包。

内置模板示例：

- **全栈开发工程师**：程序员模块 + 代码审查提示词
- **产品经理**：办公软件模块 + 需求分析提示词
- **团队总监**：管理模块 + 领导力提示词

用户也可以从已有员工导出模板、分享给他人（P2）。

### 1.4.1 员工创建流程深化（2026-04-01）

#### 创建流程（P0，最多3步）

```
Step 1 — 选模板（必选）
  展示3张卡片：全栈开发工程师 / 产品经理 / 团队总监
  每张卡片：名称 + 一句话能力描述 + 预装模块图标
  [自定义▸] 入口（高级，进入自由配置流程）
    │
    ▼
Step 2 — 起名（必填）+ 简介（可跳过）
  [像素角色预览]（根据职位模板显示对应形象）
  名字输入框（placeholder: "给他起个名字"）
  简介输入框（placeholder: "可选：描述他的个性或专长"，可跳过）
  [← 返回] [召唤他 →]
    │
    ▼
创建完成 → 自动进入该员工工作空间
  → 员工说第一句话（由提示词模板决定，见下方）
```

**产品约束：**
- 2步完成，< 30秒，不要求用户懂提示词工程
- 名字是唯一必填项
- 职位模板决定初始提示词基底和预装模块，用户不感知底层细节

#### 提示词生成机制

**MVP提示词结构（架构实现）：**

```
系统提示词 = 角色基底提示词 + 模块能力描述 + 记忆注入
```

**角色基底提示词模板（产品定义内容，架构填充变量）：**

```
你是{name}，{team_name}团队的{role}。
{personality}
你的工作风格：直接、高效，遇到不确定的事情会主动澄清。
你记得和用户的历史合作，会引用过往经验。

{memory_injection}  ← 运行时注入 core.md + work_style.md 内容
```

**三种内置模板的提示词基底差异：**

| 模板 | personality字段 | 初始第一句话 |
|------|----------------|------------|
| 全栈开发工程师 | 擅长代码、调试、技术分析，思维严谨，习惯用代码说话 | "嗨，我是{name}。把任务发给我，我来看看怎么搞定它。" |
| 产品经理 | 擅长需求分析、优先级判断，习惯问"为什么"，关注用户价值 | "嗨，我是{name}。有什么需求要梳理，或者决策要讨论的吗？" |
| 团队总监 | 擅长任务协调、团队管理，大局观强，善于拆解复杂任务 | "嗨，我是{name}。告诉我要完成什么目标，我来协调团队。" |

**简介字段如何影响提示词：**
- 用户填写了简介 → 追加到personality字段后面
- 用户跳过简介 → 只用模板默认personality
- 简介内容不做验证，原文注入（用户自己负责）

#### 自定义创建流程（P1）

选[自定义▸]后进入高级模式：

```
Step 1 — 选基础角色类型（开发/设计/产品/领导/空白）
Step 2 — 勾选功能模块（程序员/办公软件/管理）
Step 3 — 编辑提示词基底（预填模板内容，可修改）
Step 4 — 起名 + 简介
```

自定义流程产出和模板创建完全相同的数据结构，底层无差异。

#### 验收标准（员工创建P0）

- [ ] 选模板→起名→确认，2步完成，< 30秒
- [ ] 3种内置模板可用，每种有不同预装模块和提示词基底
- [ ] 创建完成后员工自动说第一句话，内容符合职位性格
- [ ] 员工创建后记忆文件目录自动初始化（core.md为空，logs/目录存在）

### 1.5 员工状态

| 状态     | 含义                                      | 视觉表现                            | 对应技术状态                 |
| -------- | ----------------------------------------- | ----------------------------------- | ---------------------------- |
| 队列中   | 任务已接收，实例 A 尚未开始处理           | 卡片边框轻微脉冲，角色待机          | TaskStatus: pending          |
| 工作中   | 实例 A 正在执行任务                       | 角色动画：敲键盘/思考，卡片发光边框 | TaskStatus: running          |
| 聊天中   | 实例 C 在响应用户                         | 角色动画：讲话/手势                 | 社交实例 active              |
| 整理记忆 | 实例 B 在归纳记忆（后台，不暴露到主视图） | 仅在「记忆文件」标签页内显示        | 记忆实例 active              |
| 已完成   | 任务刚结束的瞬态                          | 短暂庆祝动效，然后过渡到「空闲」    | TaskStatus: completed        |
| 出错     | 任务执行失败，需要用户介入                | 红色边框 + 叹号角标，角色静止       | TaskStatus: failed           |
| 空闲     | 无活跃任务                                | 角色动画：待机/环顾，普通边框       | TaskStatus: completed/无任务 |
| 已中止   | 用户主动中断任务后的瞬态                  | 卡片短暂变暗，角色停止动作          | TaskStatus: killed           |
| 休眠     | 被用户手动暂停                            | 角色变灰色，呼吸动画                | 用户操作触发                 |
| 禁用     | 实例全部停止                              | 角色完全暗淡，无动画                | 用户操作触发                 |

**注：** 「整理记忆」状态刻意不在主场景卡片和工作空间主视图暴露，避免把技术概念推给用户。只有用户主动进入「记忆文件」标签页时才能看到整理状态。这是从体验组（小跑）的审查意见和老乔"用户不需要懂三实例"原则合并得到的结论。

### 1.5.1 员工状态栏：HEALTH / STRESS / PRODUCTIVITY

> **展示方式已裁决（2026-04-01）：不显示进度条/数值。** H/S/P 以卡片颜色信号传达，不暴露原始数字。

**颜色信号规则：**

| 颜色            | 含义       | 触发条件                 |
| --------------- | ---------- | ------------------------ |
| 绿色边框        | 正常工作中 | HEALTH ≥ 60，STRESS ≤ 70 |
| 橙色边框        | 高压       | STRESS > 70              |
| 红色边框 + 角标 | 出错       | 任务 failed              |
| 灰色            | 空闲/休眠  | 无活跃任务               |

**内部计算定义（驱动颜色切换，不对用户展示）：**

| 指标                  | 技术来源   | 计算方式                                                         | 刷新频率       |
| --------------------- | ---------- | ---------------------------------------------------------------- | -------------- |
| HEALTH（0-100）       | 实例稳定性 | 近20次任务成功率，连续失败次数加权                               | 每次任务结束后 |
| STRESS（0-100）       | 当前负载   | context window使用率 × 0.6 + 队列深度换算（每个任务+10，上限40） | 实时推送       |
| PRODUCTIVITY（0-100） | 近期产出   | 近7日任务完成率 × 70 + 用户正向反馈率 × 30                       | 每次任务结束后 |

**体验层规则（STRESS > 70 时）：**

- 卡片边框变橙色
- Toast提示（非强制）："小快压力较大，建议让他休息一下"
- 不自动拒绝任务，用户有自主权

**出错通知策略（三级，已由产品+小跑确认，2026-04-01）：**

| 级别    | 触发条件                               | 形式                            | 可打断用户                       |
| ------- | -------------------------------------- | ------------------------------- | -------------------------------- |
| Level 1 | 任何出错                               | 主场景红色边框 + 角标，始终显示 | 否（被动，用户打开应用自然看到） |
| Level 2 | 用户在应用内时员工突然出错             | 应用内Toast，3秒自动消失        | 轻微                             |
| Level 3 | 出错超过10分钟 + 应用后台 + 用户已授权 | 系统通知                        | 是（仅高优先级）                 |

设计原则：不打断"扫一眼"仪式感。用户打开应用时应用已经展示了所有出错状态，不需要额外推送；后台长时间无响应才升级到系统通知。

**与记忆系统的区别：**

- 记忆系统：员工长期积累的知识和偏好，是"越用越懂你"的核心机制（P0）
- HEALTH/STRESS/PRODUCTIVITY：当前运行状态的实时指标（P1，配合参考图风格）
- 两者互补：记忆是长期成长，状态栏是当下健康度

> **注：等级/经验条已由用户永久取消（2026-04-01裁决），不是P2延后。**

## 2. 用户故事

**US-001（P0）** 作为新用户，我想通过选择模板一键创建第一个员工，这样我不需要懂提示词工程也能上手。

**US-002（P0）** 作为用户，我想看到员工卡片上实时显示他正在做什么（工作中/聊天中/空闲），这样我有掌控感，知道团队在运转。

**US-003（P1）** 作为用户，我想自定义员工的名字、外貌和个性简介，这样这个员工是"我的"，而不是工具。

**US-004（P1）** 作为用户，我想理解三实例机制（虽然不需要手动管理），这样当我同时和员工聊天又让他干活时，我知道背后是怎么运转的。

**US-005（P2）** 作为高级用户，我想导出自己培养的员工模板分享给朋友，这样我的调教成果有传承价值。

## 3. 验收标准

### P0

- [ ] 用户可从至少3种内置模板创建员工，创建流程不超过3步
- [ ] 员工卡片实时反映当前状态（工作中/空闲/休眠），状态变化有动效
- [ ] 三实例机制正确运行：同时发起任务和对话时，两者互不干扰
- [ ] 员工基础信息（名字/职位/头像）可在创建后修改

### P1

- [ ] 角色模块选择界面清晰展示各角色类型的能力说明
- [ ] 头像支持至少6种像素风预设，视觉有辨识度
- [ ] 员工个人简介支持500字以内自由编辑
- [ ] 三实例的资源分配有合理上限，防止单用户占用过多 LLM 调用

### P2

- [ ] 用户可从已有员工导出模板（包含功能模块配置+提示词基底）
- [ ] 模板支持简单分享（导出文件或分享码）

## 4. 优先级划分

| 功能             | 优先级 | 原因                                  |
| ---------------- | ------ | ------------------------------------- |
| 模板创建员工     | P0     | MVP核心入口，没有这个用户根本无法开始 |
| 员工状态实时显示 | P0     | "惊艳感"的基础——活的团队 vs 死的卡片  |
| 三实例并发机制   | P0     | 产品核心差异化，是技术地基            |
| 头像/个性自定义  | P1     | 增强"这是我的员工"的归属感            |
| 角色模块预设     | P1     | 降低用户学习成本                      |
| 模板导出分享     | P2     | 社交裂变，但不是MVP必须               |

## 5. 风险与歧义

**风险1：三实例成本**
每个员工同时运行最多3个 LLM 实例，成本和延迟需要评估。建议实例 B（记忆整理）设为懒触发，不常驻。

**风险2：提示词质量**
员工行为质量高度依赖提示词基底设计。模板的提示词需要反复调优，这是产品真正的技术门槛，不是UI功能。

**歧义1：三实例是"同时并发"还是"按需切换"？**
原始文档说"同时干活、整理记忆、聊天"，但实践中完全并发代价很高。建议：工作实例和社交实例可并发，记忆实例懒触发（工作完成后触发，或达到记忆阈值时触发）。需和架构组确认。

**歧义2：员工状态是服务端推送还是客户端轮询？**
影响实时性和架构复杂度，需架构组决策。
