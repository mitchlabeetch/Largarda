# PRD-workspace — 工作空间

## 产品定位提醒（已更新：用户明确要求）

工作空间是**一个真正的像素风房间，不是面板/Dashboard**。

员工的状态通过在房间里的位置和动作来表达，而不是文字标签或数据面板。用户进入工作空间，看到的是"这个角色在他自己的房间里做事"——他走到电脑前、在桌子旁记笔记、在沙发上休息——这些行为本身就是信息。

**底层数据（笔记/记忆/任务日志）仍然存在，但通过"点击房间里的家具/物品"来触达，而不是点击标签页。**

---

## 1. 功能描述

### 1.1 工作空间的本质：像素风房间

每个员工有一个专属的像素风房间。房间是可扩展的场景容器，随员工成长可以增加新家具、新物品、新交互点。

**房间的基础构成（参考用户提供的参考图更新）：**

**可交互家具（点击触达内容）：**

| 家具/区域       | 象征含义           | 员工动作                               | 点击触达的内容                                                          |
| --------------- | ------------------ | -------------------------------------- | ----------------------------------------------------------------------- |
| 电脑桌 + 显示器 | 工作/编程/通讯     | 员工坐在桌前敲键盘（戴耳机）           | 当前任务状态 + 工具调用日志；点击**电脑屏幕**打开聊天详情（含沟通记录） |
| 桌边笔记本      | 记忆整理/翻笔记    | 员工在桌前发呆或翻笔记本（整理记忆时） | 核心记忆文件 + 笔记列表（可编辑）                                       |
| 书架            | 记忆/知识积累      | 员工在翻书（记忆实例工作时）           | 核心记忆文件 + 经验方法论（书架和笔记本共享记忆数据源）                 |
| 床              | 休息/休眠          | 员工躺在床上睡觉（ZZZ气泡）           | **点击弹出唤醒确认框**（2026-04-01 用户最终更新，覆盖原裁决7"无交互"）  |

> **裁决更新（2026-04-01）：床从"无交互纯装饰"更新为"可点击唤醒"。** 原裁决7已被本次用户指令覆盖。
>
> **当前可交互家具：电脑 + 桌边笔记本/书架 + 床（唤醒）。** 沙发/聊天区等概念已取消。
>
> **角色本身**（头部/上身区域）可点击，触发快捷操作菜单。hitbox优先级高于家具hitbox，防止与电脑区域重叠时误触。

**纯视觉装饰元素（不可交互，构成"生活感"）：**

| 元素       | 说明                                                                               |
| ---------- | ---------------------------------------------------------------------------------- |
| 窗户       | 显示室外场景，随真实时间变化（白天=蓝天白云、傍晚=夕阳橙红、夜晚=城市天际线+星空） |
| 彩灯串     | 房间装饰，夜间提供温暖散光                                                         |
| 植物       | 书架旁/窗边，绿色点缀                                                              |
| 地毯       | 地板上，增加房间温度感                                                             |
| 咖啡杯     | 桌上，生活细节                                                                     |
| 游戏手柄   | 地上/床边，空闲时的个性元素                                                        |
| 猫（P2可选） | 参考图没有猫，不列入MVP交付范围；如用户要求再加                                  |
| 机器人手办 | 书架上，AI主题装饰                                                                 |

这些装饰元素是"房间有生命感"的主要来源，不需要点击交互，但会随员工等级/解锁模块增加新物件。

**员工在房间里的位置 = 当前状态的直接表达（RPG地图逻辑）：**

像红白机RPG游戏里角色在地图上站在不同位置就代表不同事件触发点——员工所在位置本身就是状态信息，不需要文字标签。

| 员工位置            | 对应状态         | 用户可触发的交互                           |
| ------------------- | ---------------- | ------------------------------------------ |
| 电脑前，坐着敲键盘  | 工作实例运行中   | 点击电脑屏幕 → 查看聊天/任务日志           |
| 桌前发呆或翻笔记本  | 记忆整理中       | 点击笔记本 → 查看记忆文件                  |
| 书架旁翻书          | 记忆实例活跃     | 点击书架 → 查看核心记忆/笔记               |
| 房间中央，闲站/发呆 | 完全空闲         | 点击角色 → 发起对话（底部输入框弹出）      |
| 床上躺着睡觉        | 休眠（用户触发） | 点击床 → 弹出确认框 → 确认后角色起身走到电脑前 |

- 整个房间灯光暗淡（sleeping状态）→ 所有家具交互仍可用，床的唤醒是最直观入口

### 1.1.2 状态切换动画（已更新：RPG移动模型，2026-04-01用户最终确认）

> **用户最终确认（2026-04-01）：角色在房间内像RPG角色一样移动，不是淡出瞬移。** 这覆盖了之前"淡出→瞬移→淡入"的旧方案。
>
> **原旧方案（淡出瞬移淡入）已废弃。** 原因：用户明确说"好玩有趣，像RPG角色在地图里走"，走路移动才能带来RPG的沉浸感，淡出瞬移是SaaS式的实现方式。

**切换流程（RPG移动版）：**

```
状态变化信号
    │
    ▼
角色当前动作自然收尾（不截断）
    │
    ▼
角色执行"起身/离开当前位置"骨骼动作
    │
    ▼
角色在房间内行走移动到目标位置（骨骼走路动画）
    │
    ▼
到达目标位置，执行"就位"骨骼动作（坐下/躺下/站定）
    │
    ▼
进入新状态循环骨骼动画
```

**各状态的固定位置（side view，横版侧视）：**

| 状态          | 角色位置                     | 到达后动作           |
| ------------- | ---------------------------- | -------------------- |
| 工作中        | 电脑桌前，坐姿               | 坐下→开始敲键盘      |
| 记忆整理      | 桌边或书架旁，坐/站姿        | 坐下→翻笔记本/翻书   |
| 空闲          | 房间中央偏右，站姿           | 站定→idle随机动作    |
| 休眠          | 床边，躺姿                   | 躺下→sleeping循环    |
| 出错          | 当前位置不移动               | 停下→error骨骼循环   |
| 唤醒（从休眠）| 床→走到电脑前                | 起身→走路→坐下→工作  |

**特别说明——唤醒流程（床的点击交互）：**

1. 用户点击床上睡觉的员工
2. DOM层弹出确认框："叫醒 [员工名] 吗？他正在休息。" → [取消] [叫醒]
3. 用户点"叫醒" → 员工执行"揉眼睛/起身"骨骼动作 → 站起来 → **走路移动到电脑前** → 坐下 → 进入idle状态（等待任务）
4. 休眠状态解除，员工可以接收新任务

**产品约束：**

1. 移动路径预设固定（状态A → 状态B 的路径由美术定义），不做运行时A\*寻路
2. 过渡移动期间后台状态已切换（工作实例已活跃），视觉滞后可接受
3. 动画期间用户可点击家具，抽屉正常展开，不等动画结束
4. 频繁切换（30秒内≥2次）：跳过移动动画，直接到位（瞬移降级）
5. 首次进入房间：员工直接在正确位置，无入场走路

**房间的扩展性原则（模块驱动，不是等级驱动）：**

模块安装 = 房间里出现对应道具。这是模块安装的仪式感来源，也是房间"活着"的信号。

| 模块         | 安装后房间新增道具                         | 道具是否可交互                         |
| ------------ | ------------------------------------------ | -------------------------------------- |
| 程序员模块   | 第二块显示器（多屏）、服务器机箱（角落）   | 第二块显示器：点击展开代码终端日志抽屉 |
| 管理模块     | 白板（墙上，写着任务分配）                 | P1：点击白板查看员工分配情况           |
| 办公软件模块 | 桌上出现不同软件图标贴纸（文件夹、便签堆） | 点击：展开文档/笔记列表                |

**规则：**

- 新家具/道具在模块安装后有入场动效（从场景外滑入或"出现"效果），不是静默刷新
- 卸载模块时对应道具消失（同样有出场动效）
- 所有新道具 = 潜在的新可交互 hitbox，保持可扩展
- **等级不触发房间变化**（等级/经验条已永久取消）；房间视觉变化的唯一触发是模块安装/卸载

### 1.1.1 工作空间入口

1. **点击主场景中的员工卡片** → 卡片扩展过渡，进入该员工的房间
2. **从任何地方的员工名字** → 点击跳转到该员工房间

房间之间相互隔离。**用户可以查看和编辑员工的记忆文件**（点击书架→选择文件→进入编辑模式，带警告提示），不能通过直接文件系统操作绕过。详见1.3记忆系统的编辑说明。

### 1.2 笔记系统

**定位：** 员工自己维护的工作文档，包括项目说明、任务记录、临时草稿等。

**结构：**

```
工作空间/
├── 笔记/
│   ├── 项目笔记/        # 按项目分类
│   ├── 任务日志/        # 每次任务自动生成的摘要日志
│   └── 草稿/           # 员工临时存储的中间结果
├── 记忆库/
│   ├── 核心记忆.md      # 持久化的经验和方法论
│   ├── 工作风格.md      # 员工自总结的偏好和习惯
│   └── 关系图谱.md     # 认识的员工、过往协作记录
└── 代码引用/            # 程序员模块：关联的仓库和文件引用
```

**笔记的生命周期：**

- **自动创建：** 每次任务结束后，工作实例自动生成任务摘要写入"任务日志"
- **员工主动维护：** 工作实例可以调用 `doc_create` 主动记录（通过办公软件模块）
- **记忆实例整理：** 定期压缩过期日志，将重要内容沉淀到"核心记忆"

### 1.3 记忆系统

**定位：** 员工的长期记忆，是他"越用越好"的核心机制。记忆质量直接决定员工的工作质量。

**源码洞察（context.ts）：** Claude 的 `context.ts` 揭示了 Agent 感知环境的实现方式：

1. **`getUserContext()`**：通过 `getClaudeMds()` 读取工作目录下的 `CLAUDE.md` 文件，作为持久化的"规则记忆"注入系统提示词。这就是虚拟办公室员工"核心记忆"的技术原型——**员工的记忆文件本质上就是他自己的 CLAUDE.md**。
2. **`getSystemContext()`**：注入 git 状态、当前日期等环境上下文，让 Agent 感知当前工作状态。员工需要类似机制感知：当前团队状态、自己的角色、已分配的任务。
3. **记忆缓存机制（memoize）**：Claude 的上下文读取有 `memoize` 缓存，对话开始后不重读。员工记忆注入也应该遵守同样规律：每个任务开始时注入一次，不在对话过程中反复读取文件（避免 I/O 和 token 浪费）。

**记忆层级：**

| 层级       | 内容                   | 更新频率         | 生命周期         |
| ---------- | ---------------------- | ---------------- | ---------------- |
| 工作上下文 | 当前任务的临时信息     | 实时             | 任务结束清空     |
| 任务摘要   | 已完成任务的摘要       | 每次任务后       | 保留，可压缩     |
| 核心记忆   | 通用经验、方法论、偏好 | 记忆实例定期整理 | 永久保留，可编辑 |
| 关系记忆   | 认识的人、协作模式     | 有新协作时更新   | 永久保留         |

**记忆整理机制（记忆实例 B 的工作）：**

1. 触发条件：任务完成后、记忆文件超过阈值大小时
2. 流程：读取近期任务日志 → 提炼关键经验 → 合并到核心记忆 → 归档或删除旧日志
3. 这个过程对用户可见（工作空间记忆面板显示"整理中"状态）

**为什么记忆是核心卖点：**
普通 AI 没有记忆，每次聊天从零开始。这个员工用得越久，记忆越丰富，工作越精准。这是游戏养成的核心循环——成长是真实发生的，不是数字游戏。

### 1.3.1 记忆系统深化方案（2026-04-01）

#### B实例触发时机

**MVP最小实现（P0）：无独立B实例，由A实例收尾写摘要**

A实例任务完成后，在关闭上下文之前执行记忆写入：

```
A实例任务执行中
    │
    ▼
任务完成（TaskStatus: completed）
    │
    ▼
A实例执行记忆写入（同步，任务关闭前）：
  1. 生成任务摘要（200字以内）→ 追加到 任务日志/YYYY-MM.md
  2. 检查核心记忆.md是否需要更新：
     - 发现新的用户偏好？→ 追加到核心记忆.md
     - 出现新的工作方法？→ 追加到工作风格.md
     - 涉及新的协作关系？→ 追加到关系图谱.md
  3. 检查任务日志大小：超过 50KB → 标记"待压缩"
    │
    ▼
A实例上下文关闭，下一个任务开始
```

**为什么不用独立B实例（MVP阶段）：**
- B实例需要独立进程管理，增加架构复杂度
- A实例收尾写摘要的质量在任务刚结束时最高（上下文完整）
- MVP阶段任务密度低，不需要异步压缩

**P1升级：独立B实例（后台压缩）**

触发条件：任务日志累计超过 50KB 或 30条记录，触发后台B实例：

```
检测到"待压缩"标记
    │
    ▼
启动B实例（独立上下文，不影响A实例运行）
    │
    ▼
读取近30条任务日志
    │
    ▼
提炼：哪些经验值得长期保留？
  - 出现≥3次的模式 → 写入核心记忆
  - 用户明确表达的偏好 → 写入核心记忆
  - 一次性任务细节 → 归档（保留索引，不注入上下文）
    │
    ▼
更新核心记忆.md + 归档旧日志
    │
    ▼
B实例关闭
```

#### 文件结构（完整规格）

```
~/.aion/employees/{employee_id}/
├── memory/
│   ├── core.md              # 核心记忆（注入系统提示词，≤4000 tokens）
│   ├── work_style.md        # 工作风格偏好（注入系统提示词）
│   ├── relationships.md     # 关系图谱（按需注入）
│   └── logs/
│       ├── 2026-04.md       # 当月任务日志（追加写入）
│       ├── 2026-03.md       # 上月（待压缩或已压缩）
│       └── archive/         # 已压缩归档（不注入上下文）
├── notes/
│   ├── projects/            # 项目笔记（员工主动创建）
│   └── drafts/              # 草稿（临时中间结果）
└── config.json              # 员工配置（模块列表、提示词基底ID）
```

**注入规则（产品定义，架构执行）：**

| 文件 | 注入时机 | 注入方式 | 大小限制 |
|------|---------|---------|---------|
| core.md | 每次任务开始 | System prompt 追加 | ≤4000 tokens，超出截断旧内容 |
| work_style.md | 每次任务开始 | System prompt 追加 | ≤1000 tokens |
| relationships.md | 任务涉及协作时 | 按需注入 | ≤500 tokens |
| logs/当月.md | 不注入 | 仅本地存储 | 无限制 |

**核心约束：**
- 注入总量不超过 5000 tokens（避免挤压任务上下文）
- 每次任务开始注入一次，任务过程中不重读（参考Claude context.ts的memoize机制）
- 用户可在书架界面直接编辑任意文件，有警告提示

#### 压缩策略（防止记忆文件失控）

**问题：** core.md如果无限追加，最终超过注入上限，早期记忆被截断，等于没有。

**策略：**
1. core.md 超过 3000 tokens 时，A实例收尾写摘要前先做一次自动压缩：
   - 保留：最近30天有引用过的条目（说明仍然相关）
   - 保留：用户手动编辑过的条目（有人工价值）
   - 归档：超过90天未被引用的条目（移到archive/）
2. 压缩不删除，只移动到archive/（用户仍可在书架里找到）
3. 压缩操作在任务日志里留记录：`[2026-04-01] 记忆压缩：移出12条旧条目`

#### 验收标准（产品层可验证）

- [ ] 第一次任务结束后，任务日志自动生成摘要条目
- [ ] 用户在任务里说"我们用驼峰命名"，第二次任务员工的代码输出遵守了这条规则（记忆注入生效）
- [ ] core.md在书架界面可查看、可编辑，编辑后下一次任务生效
- [ ] 任务日志在书架界面按月份分组显示，无上限

### 1.4 搜索功能

**定位：** 搜索是工作空间的"外脑检索器"，让用户和员工都能快速找到任何工作痕迹。

**搜索范围：**

| 范围             | 内容                   | 默认是否包含       |
| ---------------- | ---------------------- | ------------------ |
| 当前员工笔记     | 所有笔记文件           | 是                 |
| 当前员工记忆     | 核心记忆、任务摘要     | 是                 |
| 当前员工代码引用 | 关联仓库的文件名和内容 | 是（需程序员模块） |
| 跨员工搜索       | 所有员工的工作空间     | P2，需用户授权     |

**搜索类型：**

- **关键词搜索：** 全文匹配，高亮显示
- **语义搜索：** 用自然语言描述查找内容（P1，需向量索引）
- **时间筛选：** 按日期范围过滤

**搜索入口：**

1. 工作空间顶部搜索栏（员工维度）
2. 全局搜索快捷键（跨员工，P2）

### 1.5 信息触达方式：从"标签页"改为"点击家具"

原来设计的三个标签页（聊天视图/工作视图/记忆视图）现在通过房间里的家具点击来触达：

| 原标签页              | 家具 hitbox | 触发方式                                     | 抽屉内容                                                   |
| --------------------- | ----------- | -------------------------------------------- | ---------------------------------------------------------- |
| 工作视图（任务状态）  | 电脑屏幕    | Canvas hitbox 点击 → 通知 DOM → 右侧抽屉展开 | 当前任务状态 + 工具调用日志 + **沟通记录**（聊天入口在此） |
| 记忆视图（笔记/记忆） | 书架        | Canvas hitbox 点击 → 通知 DOM → 右侧抽屉展开 | 核心记忆文件 + 笔记列表（可编辑）                          |

> **聊天入口在电脑屏幕抽屉内，不是单独家具。**（2026-04-01 用户裁决）
>
> **关闭抽屉：点击房间空白区域（Canvas 无 hitbox 区域）关闭。**（2026-04-01 用户裁决）

### 1.6 渲染层架构（Canvas + DOM 分层）

产品层面明确分层边界，实现层遵守：

```
Canvas 层（PixiJS）
  ├── 房间场景渲染（地图、家具、装饰元素）
  ├── 角色骨骼动画
  ├── 家具 hitbox（点击热区定义）
  └── 环境动效（粒子、光效）

DOM 层（覆盖在 Canvas 上，Electron/Web）
  ├── 顶部状态栏（HEALTH / STRESS / PRODUCTIVITY / TIME / STATUS）
  ├── 右侧抽屉（点击家具 hitbox 触发，展示任务/记忆/聊天内容）
  ├── 底部 HUD 输入栏（发任务 / 发消息）
  └── 所有弹窗 / 确认框 / Toast 提示
```

**分层原则：Canvas 负责"看"，DOM 负责"操作"。**

- 所有需要用户输入、文字阅读、列表滚动的内容 → DOM 层
- 所有动画、场景渲染、角色表现 → Canvas 层
- 两层通过事件总线通信：Canvas hitbox 点击 → 发布事件 → DOM 层订阅并展开对应抽屉

**Canvas 背景点击行为（产品定义）：**
点击 Canvas 无 hitbox 区域（房间背景）= 关闭当前打开的 DOM 抽屉。无抽屉打开时，背景点击无响应。

**抽屉内容渲染：**
所有抽屉内容（代码终端日志、聊天记录、记忆文件）均在 DOM 层渲染，不在 Canvas 内绘制。理由：文字内容 DOM 渲染清晰度高，用户需要阅读代码/日志，不需要"像素风终端"视觉效果。

**悬浮面板的设计原则：**

- 面板（抽屉）从右侧或家具旁滑出，不全屏覆盖房间场景
- 面板是信息层，房间是感知层，两者共存不互斥
- 面板内容区域可滚动，宽度固定（不可拖拽调整，P2可考虑）

## 2. 用户故事

**US-201（P0）** 作为用户，我想进入员工的像素风房间，看到他正在电脑前工作/在书桌旁记笔记/在沙发上休息，这样我一眼就知道他在做什么，不需要看任何文字标签。

**US-202（P0）** 作为用户，我想点击员工房间里的电脑，看到他正在执行的任务详情，这样我有掌控感且交互有代入感。

**US-203（P0）** 作为用户，我想点击员工角色本身和他聊天，这样对话像是"走进他的房间和他说话"，而不是打开一个聊天软件。

**US-204（P1）** 作为用户，我想点击书架看到员工积累的记忆和笔记，这样书架里内容越来越多就是他"成长"的视觉证明。

**US-205（P1）** 作为用户，当员工解锁新模块后，我想看到他的房间里多出了新的家具/装备，这样能力扩展有仪式感。

**US-206（P1）** 作为用户，我想搜索员工工作空间中的内容，这样当我忘记某个结论时可以快速找到。

**US-207（P2）** 作为用户，我想看到员工升级后房间的装饰和家具发生变化（更豪华/更有个性），这样成长不只是数字，是看得见的生活质量提升。

**US-208（P2）** 作为用户，我想跨所有员工搜索内容，这样有全局视角。

## 3. 验收标准

- [ ] 点击房间里的电脑可以看到当前任务状态和工具调用日志

### P1

- [ ] 点击书架可浏览核心记忆文件和笔记
- [ ] 员工安装新模块后，房间里出现对应的新家具（程序员模块→第二块显示器）

## 5. 新增歧义（需总控裁决）

**歧义A：状态同步时序——已由用户需求确定**

用户要求完整过渡动画，这实际上已经给出了答案：**收到状态变化信号后，立即开始动画序列，动画过程中后台状态已切换**。动画本身就是延迟的视觉消化器。

产品定义（已确定）：

- 收到"工作实例活跃"信号 → 立刻触发"起身→行走→坐下"动画序列
- 动画过程中工作实例已在后台运行
- 用户看到角色走向电脑就理解"他要开始工作了"，不需要等坐下

**剩余技术歧义（交架构组）：** 如果状态推送本身有 3s 以上延迟，过渡动画开始时机要不要提前预判（看到任务队列有新任务就提前开始走）？这是技术实现细节，产品层面不做限定。

**歧义B：房间内的"聊天"是浮层还是全屏？**

点击员工角色聊天，输入框从哪里出来：

- 底部浮层（房间场景保持可见）：保持沉浸感，但输入区域小
- 全屏聊天视图（离开房间场景）：信息展示完整，但破坏沉浸感

**建议底部浮层**，和小跑的流程草案一致。全屏聊天视图作为"展开"选项。

### 7. 补充：消息队列 UI（对应 pendingUserMessages 机制）

员工工作中时（工作实例活跃），用户发送的消息不会打断工作实例，而是进入队列等待。

**视觉表达：**

- 房间顶部出现一个小气泡提示："消息已排队，小快忙完会看到"
- 员工角色不会立刻停下动作、转头——他依然在敲键盘
- 气泡在工作状态下持续显示，任务完成进入空闲状态后自动消失（触发员工"读到消息"动作：转头→走向聊天区→回复）

**例外：聊天实例（C实例）在线时：**

- 如果员工配置了独立聊天实例，用户消息不进队列，由C实例实时接收
- 视觉上：员工继续工作（A实例），但房间聊天区有光效或文字提示"他的助理在听"
- 这是高级功能（P1），P0不区分有无C实例，统一走消息队列

**用户主动中断（强制打断）：**

- 在聊天浮层有"立即打断"选项（⚠️ 会中止当前任务）
- 点击后弹出确认：当前任务将中止，未完成的工作可能丢失，确认打断？
- 确认后：工作实例执行安全停止，员工动画执行"停下→转向用户→走到聊天区"序列

### 8. 补充：出错状态的恢复流程

当工作实例进入"出错"状态（failed），用户在房间里看到：

- 员工角色停在某个中间位置，不再有工作动作
- 房间有轻微红色氛围光（不刺眼，只是提示）
- 员工上方有一个浮动图标（类似叹号或红色警告）

**用户点击出错的员工或警告图标时：**

1. 弹出悬浮面板（从员工位置弹出，不全屏）
2. 面板显示：任务名称 + 错误摘要（一句话）+ 完整错误日志（可折叠）
3. 操作选项：
   - **重试**：以相同参数重新执行任务
   - **查看日志**：展开完整工具调用日志
   - **手动修复后重试**：引导用户修正环境后再触发
   - **忽略/关闭**：关闭面板，员工回到空闲状态（错误记录保留在任务日志里）

**重试后的动画：**

- 关闭面板 → 员工执行"整理一下→走到电脑前→坐下→重新开始工作"的完整序列
- 不播放出错状态的特殊动画，直接进入正常工作序列

## 6. 风险与歧义（原有）

**风险1：记忆文件大小失控**
员工工作多了记忆文件会越来越大，影响上下文注入性能。记忆实例的整理压缩策略是关键，需要产品和架构联合设计，不能只靠 LLM 自由发挥。

**风险2：记忆质量无法保证**
LLM 整理记忆时可能"编造"或"遗漏"重要信息。记忆整理结果用用户可读格式存储（Markdown）。**MVP就允许用户手动编辑记忆文件**（响应老乔质疑2）——点击书架→选择文件→编辑，界面有警告："手动修改记忆文件会影响员工的工作方式，确定继续？"。用户需要控制感，特别是MVP阶段记忆整理质量不稳定时。

**歧义1：工作空间是否对所有用户可见？**
如果是多用户场景（团队共用），谁能看谁的工作空间？当前假设是单用户产品，所有员工的工作空间对该用户可见。如果未来支持多用户需要重新设计权限。

**歧义2：代码引用具体怎么存？**
员工"关联的仓库"是存路径引用还是把代码内容缓存进来？存路径的话需要实时读取，存内容的话存储爆炸。建议：存路径引用，搜索时实时读取，记忆层只存代码片段摘要。
