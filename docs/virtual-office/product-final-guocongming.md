## 二、功能边界定义（补充老乔定稿未覆盖的部分）

### 2.1 消息队列行为

员工工作中（实例A活跃）时，用户消息不中断工作实例，进入消息队列：

- 视觉：房间顶部出现气泡提示"消息已排队，小快忙完会看到"
- 员工角色不停手，继续当前动作
- 任务完成进入空闲后，触发"读消息"动作序列（转头→走向通讯设备→回复）
- 用户可以在聊天浮层选择"立即打断"（⚠️ 会中止当前任务），打断需要确认弹窗

**P0范围：** 统一走队列，不区分有无C实例。  
**P1范围：** C实例在线时，用户消息不进队列，由C实例实时接收，工作实例不受影响。

### 2.2 出错状态恢复流程

员工出错（实例A failed）时：

- 视觉：轻微红色氛围光 + 员工上方浮动警告图标
- 点击员工或警告图标 → 悬浮面板弹出（从员工位置弹出，不全屏）
- 面板内容：任务名称 + 错误摘要（一句话）+ 完整日志（可折叠）
- 操作选项：重试 / 查看日志 / 手动修复后重试 / 忽略关闭
- 重试后：员工执行正常工作动画序列，不单独设计"重试中"特殊动画

### 2.3 任务中途发现能力缺口

场景：员工在执行任务中发现所需工具未安装。

产品定义：

- 员工必须主动停下并告知用户："这个任务需要[能力X]，但我目前没有[对应模块]。"
- 不允许静默失败（任务失败但用户不知原因）
- 员工不能自行安装模块（安全边界）
- 通过模板路径可大幅减少这个场景出现概率

验收标准（P1）：发现工具缺口时，员工主动暂停并发消息说明缺少哪个能力。

### 2.4 首次进入体验（空状态）

第一次打开、0个员工时：

- 显示像素风房间场景，所有工位空置，有"虚影"暗示可以放员工
- 屏幕中央：招募入口按钮（发光边框，像素风格）
- 3-5秒叙事节拍：场景渐入 → 环境动效启动 → 提示文字出现 → 按钮召唤
- 这是转化率最关键的漏斗节点，**优先级P0**

员工创建完成后的回落动效：
新员工创建 → 对应工位亮起 → 角色从场景外走进来 → 走到工位坐下（3-4秒仪式感）

## 三、HEALTH / STRESS / PRODUCTIVITY 技术边界

> **展示方式已裁决（2026-04-01）：不显示进度条/数值。** 以下是内部计算定义，用于驱动卡片颜色切换，不对用户直接展示原始数字。

| 指标                  | 数值来源                                                                  | 刷新频率           | 颜色阈值                 |
| --------------------- | ------------------------------------------------------------------------- | ------------------ | ------------------------ |
| HEALTH（0-100）       | 近20次调用成功率，连续失败次数加权                                        | 每次任务结束后更新 | <60→橙，<30→红           |
| STRESS（0-100）       | context window使用率（0-100%）× 0.6 + 队列深度换算（每个任务+10，上限40） | 实时推送           | >70→橙色边框，Toast提示  |
| PRODUCTIVITY（0-100） | 近7天任务完成率（完成/总数）× 70 + 用户正向反馈率 × 30                    | 每次任务结束后更新 | 仅内部用，不触发颜色变化 |

**STRESS高时的产品行为（不强制，只提示）：**

- > 70：卡片边框变橙 + Toast提示"小快压力较大，建议让他稍作休息"
- 不自动拒绝任务，用户有自主权

## 四、养成体系定稿（已由用户裁决，2026-04-01）

> **用户裁决（终局，不可逆）：**
>
> - 等级/经验条：**永久取消**，不是P2延后，是永远不做
> - 能力树：**永久取消**，同上
> - 记忆持久化：**MVP命脉（P0）**，不是P1可选

### 4.1 养成感的唯一真实来源：记忆

用户要的"养"，本质是：**这个员工认识我，越用越懂我，换一个从零开始舍不得。**

这种感觉不需要等级条，需要的是**记忆积累的可见性**：

- 书架上记忆文件越来越多（有内容在那里）
- 员工做任务时引用了之前的偏好（行为上能感知到）
- 用户编辑记忆文件时感觉在"调教"员工（有控制感）

等级条解决的是"还差多少"的驱动感，但用户的核心诉求是"他认识我"——这个靠记忆系统直接传达，不需要数字代理。

### 4.2 记忆持久化的技术要求（P0）

参考 `~/.claude/memory/` 机制：

| 要求                | 说明                                           |
| ------------------- | ---------------------------------------------- |
| 任务结束自动写摘要  | 每次任务后，写入工作空间任务日志，不丢失       |
| 核心记忆注入提示词  | 每次任务开始，读取核心记忆文件，注入系统提示词 |
| 本地持久化          | 记忆文件存储在本地，App重启不丢失              |
| 用户可查看 + 可编辑 | MVP就要开放，用户有控制感，带风险提示          |

## 七、骨骼动画（已由用户裁决，P0）

> **用户裁决（2026-04-01）：骨骼动画是MVP必做（P0），不接受帧动画降级。PixiJS + Spine。**

### A档（P0，必须交付）

- 技术栈：PixiJS + pixi-spine（Spine格式）
- 一套骨架支持所有状态动作，换装区分职位
- 骨骼循环动画：working / reading / sleeping / idle / error
- idle ≥ 3组不同小动作随机轮换
- 状态切换：淡出200ms → 新位置淡入200ms
- 角色位置固定，不移动
- 序列帧仅作降级备选（骨骼动画资产加载失败时自动切换）

### B档（P1）

- A档基础上，切换补充专用退场/入场骨骼动作
- 退场：起身骨骼动作（0.3-0.5s）→ 淡出
- 入场：淡入 → 坐下骨骼动作（0.3-0.5s）→ 进入循环

### C档（P2）

- B档基础上，角色线性平移到目标位置，预设固定路径点，无需A\*寻路

**通用约束（三档都适用）：**

1. 过渡动画期间后台状态已切换（不等动画结束才切状态）
2. 动画期间用户仍可点击交互
3. 频繁切换（30秒内≥2次）：压缩动画时长，不积压动画队列
4. 首次进入房间：员工直接在正确位置，不播放入场动画

## 十、郭聪明对"用户到底要什么"的理解

用户的核心诉求只有一件事：**"养"。**

但"养"不是游戏里的等级升级。用户说的"养"更接近于**养一个真正认识你的人**。

这个产品最本质的价值主张是：

> **AI默认没有记忆，每次对话从零开始。这个产品打破了这一点——你的员工记得你，越用越懂你，这不是包装，这是真实发生的。**

**核心价值层次（按用户表达重新排序）：**

1. **上帝视角** — "我能看到我的团队在干什么"。掌控感。这是进入产品的第一秒体验。
2. **记忆** — "他认识我"。这是这个产品存在的理由。没有记忆，这只是一个更花哨的AI聊天界面。
3. **协作** — "他们互相配合工作"。这是和单个AI助手的差异化。
4. **任务进度** — "我知道他做到哪了"。透明度，给用户安全感。
5. **动画** — "他是活的"。情感连接的载体。没有动画，记忆系统也能用，但用户不会"舍不得"。

**为什么等级/经验条不做是对的：**

等级条传达的是"他在变强"，但用户真正的情感是"他认识我"。两件事不一样。"他认识我"靠记忆系统传达，靠书架内容变多传达，靠员工在任务里引用你之前说过的话传达——这些都是真实的、可感知的。等级条是对这种感觉的简化替代，但简化替代反而会弱化真实体验。

**养成的验证标准（产品层面）：**

一个新用户用了1个月后，应该能说出："这个员工知道我们的代码风格是用驼峰命名法，知道我不喜欢写注释，知道我们团队用的是bun不是npm。" 如果用户能说出这些，养成功了。等级条跟这件事完全没关系。

## 十一、给架构组的技术问题清单

以下是产品定稿后，架构组需要回答的核心技术问题。每个问题都直接影响MVP的实现方式，不能靠产品层面假设。

### Q1：记忆写入时机 — B实例还是A实例收尾？

**背景：** 记忆系统是P0命脉，但三实例架构中，B实例（记忆实例）是否需要MVP就实现？

**具体问题：**

- 任务结束后自动写任务摘要 — 这是A实例收尾动作，还是B实例的工作？
- 如果由A实例收尾写：A实例任务完成后上下文还在吗？写摘要需要多少额外token消耗？
- 如果由B实例写：B实例需要多快触发？能否异步不阻塞用户？
- MVP的最小可行方案是什么？能不能A实例写粗摘要，B实例定期整理精摘要？

**产品要求：** 记忆写入必须是P0，B实例是否P0可以商量，但写入机制不能P1。

### Q2：记忆注入机制 — 何时注入，注入多少？

**背景：** 核心记忆注入系统提示词是记忆生效的技术基础。

**具体问题：**

- 核心记忆文件多大时开始影响性能？建议的单文件上限是多少KB？
- 每次任务开始时全量注入核心记忆，还是按相关性筛选注入？
- 记忆文件注入是否有缓存机制（类似Claude的memoize）？

### Q3：骨骼动画资产 — Spine授权方案

**背景：** 用户裁决PixiJS + Spine是P0，Spine商用需付费。

**具体问题：**

- Spine的商用授权费用是多少？是否在预算范围内？
- DragonBones已停止维护，备选方案是什么？（Rive？自研骨骼？）
- 骨骼资产（角色骨架 + 动作）从哪里获取？自制还是购买？预估工期？
- 如果Spine授权搞不定，最接近的可行替代是什么？

### Q4：员工数量无上限 — 场景渲染性能边界

**背景：** 用户裁决员工无数量上限，但场景渲染有性能瓶颈。

**具体问题：**

- PixiJS在桌面端（Electron）渲染多少个骨骼动画角色时帧率开始下降？
- 性能边界建议（比如：<10个正常，10-30个建议降级，>30个强制降低动画质量）？
- 员工不在视口内时是否需要暂停动画（视口裁剪）？
- 场景滚动/缩放的技术实现方案？

### Q5：状态推送 — 服务端推送还是客户端轮询？

**背景：** 员工状态实时性影响"活的感觉"，但推送架构影响复杂度。

**具体问题：**

- P0阶段用轮询还是推送？轮询间隔建议多少？
- IPC通信在Electron中的状态推送延迟大概多少ms？
- 多个员工并发状态变化时，IPC消息是否会排队丢失？

### Q6：聊天入口在电脑屏幕上 — UI层实现方式

**背景：** 用户裁决聊天入口在电脑屏幕上（不是单独通讯家具），点电脑屏幕打开包含沟通记录的详情。

**具体问题：**

- 电脑屏幕点击后弹出的悬浮面板，同时承载任务状态 + 聊天记录两类内容，交互层级怎么设计？
- 悬浮面板是PixiJS层的UI还是DOM层的UI？两者混用的遮挡/事件冲突怎么处理？
- 聊天输入框是否也在悬浮面板内，还是单独从底部弹出？
