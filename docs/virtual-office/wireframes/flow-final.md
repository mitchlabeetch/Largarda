# 虚拟办公室 — 用户流程定稿

> 体验-小跑 出品 | 2026-04-01（用户裁决更新：2026-04-01）
> 核心优先级：上帝视角 > 员工记忆 > 协作沟通 > 任务进度 > 养成动画

---

## 用户裁决摘要（2026-04-01 生效）

| 裁决项        | 结果                                            |
| ------------- | ----------------------------------------------- |
| 等级和经验条  | 不要（已从所有流程删除）                        |
| 记忆系统      | 核心中的核心，持久化，参考 Claude 记忆系统      |
| 员工数量      | 不限，流程必须支持大量员工                      |
| 家具          | 固定（不可移动）                                |

| 点空白关抽屉  | 是                                              |

## 设计原则

**"养"是核心。** 员工有记忆、有积累、有状态变化。用户不是在管理工具，是在养一个会成长的团队。

**动画是增色，不是阻碍。** 所有核心管理操作（查看全局、追踪任务、查看记忆、发起协作）必须在动画之外也能立刻完成。动画可以跳过，功能不能等待。

**流程优先级排序：**

1. 上帝视角 — 用户一眼看清整个团队状态，是最高频操作
2. 员工记忆 — 进入员工工作空间查看积累，是核心差异化价值（参考 Claude 记忆系统）
3. 协作沟通 — 给员工发任务、查看沟通记录
4. 任务进度 — 追踪员工正在做什么
5. 养成动画 — 视觉上的惊喜感，锦上添花

## 一、上帝视角流程（最高优先级）

### 1.1 主场景：一眼看清整个团队

```
打开虚拟办公室
       |
       v
主场景：员工卡片展示（不限数量，见1.4节大量员工的展示策略）
       |
每张卡片显示：
  - 员工名字 + 职位
  - 状态色（绿=工作中 / 灰=空闲 / 橙=高压 / 红=出错）
  - 当前在做什么（一句话，实时刷新）
       |
用户获取信息路径（从快到慢）：
  1. 看状态色 → 0.5秒内判断谁在干活，谁出问题了
  2. 看状态文字 → 2秒内了解具体在做什么
```

**关键约束：**

- 主场景不需要进入任何子页面即可获取以上信息
- 状态实时推送更新，延迟 < 2秒
- 无员工数量上限，主场景必须支持大量员工的浏览和管理（见1.4节）

### 1.2 全局状态筛选

```
主场景顶部：[全部] [工作中] [空闲] [高压] [出错]
       |
点击筛选标签 → 卡片即时过滤
       |
场景：
  用户早上打开，想快速确认"有没有员工出错"
  → 点击[出错]标签 → 只看出错员工 → 1次点击完成
```

### 1.3 全局搜索

```
主场景顶部搜索框（始终可见）
输入关键词 → 实时出结果（不需要回车）
       |
结果按来源分组：
  [员工名] → 点击跳转到该员工工作空间
  [记忆片段 + 员工名] → 点击跳转到具体记忆内容
  [沟通记录片段 + 员工名] → 点击跳转到对话
       |
从搜索到找到内容：最多2次点击
```

**搜索框交互细节：**

```
初始状态：
  搜索框显示 placeholder："搜索员工、记忆、沟通记录..."
  宽度正常，不占用筛选标签空间

聚焦时（点击搜索框）：
  搜索框展开（宽度动态增加，200ms ease）
  筛选标签收起（淡出，不删除，失焦后恢复）
  背景轻微暗化（overlay rgba 0.2），引导视线到搜索结果

输入过程：
  实时搜索，不需要回车，输入停顿 150ms 后触发
  结果下拉层（DOM，position: absolute，z-index 高于卡片）
  按来源分组展示：
    ┌────────────────────────────────┐
    │  员工（2）                      │
    │  ● 小快  前端开发               │
    │  ● 阿构  后端开发               │
    │                                │
    │  记忆（3）                      │
    │  小快 › 代码风格偏好...          │
    │  小快 › 常用调试方法...          │
    │  阿构 › 数据库设计原则...        │
    │                                │
    │  沟通记录（1）                  │
    │  小快 › 修复登录bug...           │
    └────────────────────────────────┘

  关键词高亮显示（--vo-highlight 暖金色背景）
  结果最多显示每类3条，超出显示"查看全部 N 条 →"

空结果状态：
  "没有找到 [关键词] 相关内容"
  不显示空白，不卡住

关闭搜索：
  点击搜索框外任意区域 → 结果收起，搜索框恢复
  ESC 键 → 同上
  点击某条结果 → 跳转，搜索框恢复
```

**大量员工场景下，搜索是快速定位的主要入口之一。**

### 1.4 大量员工的主场景展示策略

> **MVP 范围说明：** 10人内平铺 + 虚拟滚动为 MVP 必须实现。"我的关注"列表（固定核心员工）为 **P2**，不进 MVP 实现范围，作为未来规划保留。

员工不限数量，主场景必须应对不同规模。三种规模的展示策略：

**10人以内：网格平铺**

```
所有卡片一屏或轻微滚动展示
无需分组，一眼扫完
```

**10-50人：分组 + 虚拟滚动**

```
卡片按状态分组展示：
  [出错 N] → 置顶，强优先级
  [工作中 N]
  [高压 N]
  [空闲 N]
  [休眠 N]
       |
虚拟滚动（只渲染可视区域内的卡片，避免性能崩溃）
顶部筛选栏始终固定可见
用户搜索框快速定位到某员工
```

**50人以上：搜索优先 + 分组折叠**

```
顶部搜索框成为第一入口（placeholder："找员工..."）
分组默认折叠，点击展开
出错员工组始终展开置顶（用户最需要看到的）
       |
"我的关注"列表（用户手动固定的员工，始终展开在最顶部）
  → 解决：用户有几个核心员工需要频繁查看，不能每次翻找
```

**跨规模通用原则：**

- 出错状态永远最醒目，不管多少员工都能0步发现
- 大量员工时，用户不需要也不应该一一查看所有人 — 状态色 + 搜索 + 关注列表足够
- 性能：50人以上必须用虚拟滚动，禁止一次性渲染所有卡片

### 1.5 卡片 Hover 菜单与快捷操作

卡片 hover 菜单是主场景最高频的交互入口，从主场景直接操作员工，不需要进入工作空间。

```
默认状态（无 hover）：
  卡片正常展示：姓名 + 职位 + 状态色 + 当前任务文字

Hover 状态（鼠标悬停在卡片上）：
  卡片 scale 1.0 → 1.02（200ms ease）
  底部出现快捷操作区域（+60px 高度，200ms 展开）：

  ┌─────────────────────────────────┐
  │ 任务完成: 47   工作天: 12        │  ← 简单统计（灰色小字）
  │─────────────────────────────────│
  │ [对话]  [查看工作空间]  [···]    │  ← 操作按钮
  └─────────────────────────────────┘

按钮行为：
  [对话] → 弹出快速发送浮层（不进入工作空间，直接发任务）
  [查看工作空间] → 进入该员工的像素风房间（400ms 转场）
  [···] → 展开更多操作菜单（见下）
```

**快速发送浮层（点击[对话]触发）：**

```
浮层从卡片底部向上弹出（DOM 层，200ms）：
  ┌─────────────────────────────────┐
  │ 给小快发消息                [×] │
  │─────────────────────────────────│
  │ [输入任务或消息...]              │
  │                        [发送]   │
  └─────────────────────────────────┘

  员工空闲时：发送后立即执行，Toast："已发送给小快"
  员工忙碌时：发送后排队，Toast："已排队（第2条），小快完成后执行"

关闭浮层：
  点击 [×] / 点击浮层外区域 / ESC → 浮层收起（150ms）
  不影响主场景卡片状态
```

**更多操作菜单（点击[···]触发）：**

```
下拉菜单从 [···] 按钮向下展开（DOM 层，150ms）：
  ┌──────────────┐
  │ 休眠          │
  │ 管理模块      │
  │ 编辑员工信息  │
  │──────────────│
  │ 删除员工      │  ← 红色，危险操作
  └──────────────┘

  休眠 → 弹出确认弹窗（同5.1节）
  删除员工 → 弹出二次确认："删除后该员工的所有记忆和任务历史将永久删除，无法恢复"
  点击菜单外区域 → 菜单收起
```

**出错状态的 hover 特殊处理：**

```
出错卡片 hover 时，快捷操作区替换为出错处理区：
  ┌─────────────────────────────────┐
  │ ❗ 执行 git push 时权限被拒绝    │
  │─────────────────────────────────│
  │ [查看日志]  [重试]  [放弃任务]  │
  └─────────────────────────────────┘
  不显示通用 [对话]/[查看工作空间] 按钮
  优先引导用户处理出错
```

## 二、员工工作空间流程（核心差异化）

### 2.1 进入工作空间

```
主场景点击员工卡片
       |
       v
转场（400ms，卡片展开成房间）
       |
       v
员工像素风房间（Canvas 渲染）
       |
房间内容：
  - 顶部固定：状态栏（TIME / STATUS / 当前任务一句话）
  - 房间场景：员工角色 + 家具（电脑/书架/床[暂不交互]）
  - 底部固定：HUD 输入栏（发任务/发消息，始终可用）
  - 左上角：[← 返回办公室] 始终可见
```

**重要：** 转场动画期间底部 HUD 输入栏已经可用，不等动画结束。

**家具说明：**

- 电脑 — 点击打开详情抽屉（含任务进度 + 沟通记录）
- 书架 — 点击打开记忆抽屉
- 床 — 暂时不交互（视觉装饰，后续迭代）

### 2.2 查看员工记忆（最核心功能）

记忆系统是整个产品的核心差异化。员工的记忆持久化，参考 Claude 记忆系统设计：分类存储、可搜索、随时间积累。

```
进入工作空间后
       |
点击房间内书架（Canvas hitbox）
       |
右侧抽屉滑入（DOM 层，300ms）
       |
抽屉内容（持久化，永久保留）：
  ┌─────────────────────────────────┐
  │ 小快的记忆                 [x]  │
  │                                 │
  │ [搜索记忆内容...]               │
  │                                 │
  │ 核心记忆                        │
  │   代码风格偏好...                │
  │   常用的调试方法...              │
  │   用户反馈的偏好...              │
  │                                 │
  │ 工作风格                        │
  │   偏好小步提交...                │
  │   遇到阻塞的处理方式...          │
  │                                 │
  │ 任务历史（按时间，无上限）        │
  │   [今天] 修复登录 bug            │
  │   [昨天] 重构权限模块            │
  │   [3天前] 添加代码审查流程       │
  │   [上周] ...                    │
  │                                 │
  │ 关于用户的记忆                   │
  │   用户喜欢的代码风格...          │
  │   用户不喜欢...                  │
  └─────────────────────────────────┘
       |
点击任意条目 → 展开全文（可编辑）
点击房间背景（空白区域） → 关闭抽屉
```

**记忆设计原则：**

- 持久化：不随会话结束而清空，永久积累
- 可搜索：抽屉内搜索框，跨所有记忆条目
- 分类：核心记忆 / 工作风格 / 任务历史 / 用户偏好四个维度
- 可编辑：用户可以手动修改或删除记忆条目
- 任务历史无上限 — 员工干过的每件事都有迹可循

**从进入工作空间到看到员工记忆：1次点击**

### 2.3 查看任务进度 + 沟通记录（电脑整合）

电脑是员工的工作中心，点击电脑打开详情抽屉，任务进度和沟通记录在同一个地方。

```
进入工作空间后
       |
点击房间内电脑（Canvas hitbox）
       |
右侧抽屉滑入
       |
抽屉内含两个 Tab：[任务] [沟通记录]

--- Tab: 任务 ---
  ┌─────────────────────────────────┐
  │ 小快的电脑            [任务][沟通] │
  │                                 │
  │ ▶ 正在执行：修复登录bug          │
  │   工具调用：git diff ...         │
  │   [展开详细日志 v]              │
  │                                 │
  │ 最近完成                        │
  │   [2h前] 重构权限模块 ✓          │
  │   [昨天] API超时修复 ✓          │
  │                                 │
  │ 队列中（2条）                   │
  │   · 添加单元测试                │
  │   · 代码审查PR #234             │
  └─────────────────────────────────┘

--- Tab: 沟通记录 ---
  ┌─────────────────────────────────┐
  │ 小快的电脑            [任务][沟通] │
  │                                 │
  │ 筛选：[全部] [任务记录] [聊天]   │
  │                                 │
  │ [今天 10:23] 用户                │
  │   帮我修复登录bug                │
  │                                 │
  │ [今天 10:24] 小快                │
  │   好的，我来看看...              │
  │                                 │
  │ [底部输入框] 发消息/发任务       │
  └─────────────────────────────────┘
       |
点击房间背景（空白区域） → 关闭抽屉
```

**失败状态的处理：**

```
任务 failed 时：
  电脑图标出现红色角标 ！
  抽屉[任务] Tab 顶部显示：
  ┌────────────────────────────────┐
  │ ❗ 任务执行出错                 │
  │    执行 git push 时权限被拒绝   │
  │    [查看完整日志] [重试] [放弃] │
  └────────────────────────────────┘
```

### 2.4 给员工发任务

```
两个入口（任意场景下均可用）：

入口A：主场景卡片悬浮菜单
  hover 员工卡片 → [对话] 按钮出现 → 点击 → 弹出快速发送框
  输入任务 → 发送 → 返回主场景继续看整体

入口B：工作空间底部 HUD 输入栏
  底部输入框始终可见
  输入 → 发送
       |
员工忙碌时的排队处理：
  发送后显示："已排队，小快完成当前任务后开始"
  卡片右上角出现数字角标（队列中的任务数）
```

## 三、协作沟通流程

### 3.1 用户手动跨员工协作（P0）

```
场景：用户想让小快（开发）和阿点（设计）协同改一个页面

步骤：
  1. 主场景 → hover 小快卡片 → [对话] → 发任务："参考阿点的设计稿修改登录页"
  2. 返回主场景（不需要等小快响应）
  3. hover 阿点卡片 → [对话] → 发任务："把登录页设计稿发给小快"
  4. 回到主场景看两人状态变化

总步骤：4步，全程不离开主场景视角
```

### 3.2 领导员工统一协调（P1）

```
进入总监员工的工作空间
       |
发综合任务："协调小快和阿点完成登录页改版"
       |
总监实例A调用管理模块
向小快和阿点下发子任务
       |
主场景可见：多员工同时状态变"工作中"
```

## 四、模块安装/卸载流程（P1）

### 4.1 给员工安装模块

```
入口：员工工作空间内
       |
点击顶部状态栏右侧 [模块管理] 按钮
  OR 点击员工角色 → 快捷菜单 → [管理模块]
       |
       v
右侧抽屉滑入：模块列表
  ┌─────────────────────────────────┐
  │ 小快的模块                 [x]  │
  │                                 │
  │ 已安装                          │
  │  [程序员模块] 已激活  [卸载]    │
  │  [办公软件模块] 已激活 [卸载]   │
  │                                 │
  │ 可安装                          │
  │  [管理模块]  任务协调能力 [安装] │
  │  [自定义模块...] [+]            │
  └─────────────────────────────────┘
       |
点击 [安装] → 确认弹窗（模块说明 + 确认按钮）
       |
安装完成 → Toast："程序员模块已安装，小快现在可以操作 git 了"
员工卡片模块标签更新
```

### 4.2 卸载模块

```
同上入口 → 模块列表抽屉
       |
点击已安装模块 [卸载]
       |
确认弹窗：
  "卸载后小快将无法使用 git 等代码操作，当前进行中的任务不受影响"
  [取消] [确认卸载]
       |
卸载完成 → Toast 提示
```

**操作步数：从工作空间到完成安装 = 3步（打开抽屉 → 点安装 → 确认）**

## 五、员工休眠/唤醒流程

### 5.1 休眠员工

```
入口A：主场景 hover 员工卡片 → [更多操作 ···] → [休眠]
入口B：员工工作空间顶部状态栏 → [休眠] 按钮
       |
确认弹窗：
  "让小快进入休眠？休眠期间不接受新任务，当前任务完成后停止"
  [立即休眠（中断当前任务）] [完成后休眠] [取消]
       |
休眠中：
  - 卡片变为灰色 + 降透明度
  - 角色播放缓慢呼吸循环动画（骨骼动画，不是静态图）
  - 卡片状态文字显示："休眠中"
  - HUD 输入栏变为禁用（不接受新任务）
```

### 5.2 唤醒员工

```
入口A：主场景 hover 休眠状态员工卡片 → [唤醒]
```

## 七、Canvas + DOM 分层交互规范

### 7.1 层级分工

```
Canvas 层（已裁决：PixiJS + pixi-spine，骨骼动画P0，2026-04-01用户裁决）：
  - 房间场景渲染（背景、家具、地板、窗户）
  - 角色状态动画（状态枚举切换：工作中/空闲/高压/出错）
  - 状态切换过渡（退场 + 入场，无走路寻路动画）
  - 家具交互热点（矩形 hitbox，hover 时绘制发光边框）
  - 日夜光线（窗外颜色 + 房间环境光随时间变化）

DOM 层（React，position: absolute，覆盖在 Canvas 上）：
  - 顶部状态栏（TIME / STATUS / 当前任务一句话）
  - 左上角返回按钮
  - 底部 HUD 输入栏（输入框 + 发送按钮）
  - 右侧信息抽屉（记忆/任务+沟通）
  - 所有弹窗、确认框、Toast 通知
```

### 7.2 家具点击热点规范

| 家具         | 触发区域                             | 打开内容                          |
| ------------ | ------------------------------------ | --------------------------------- |
| 电脑/显示器  | 矩形 hitbox，覆盖显示器像素区域 ×1.5 | 任务进度 + 沟通记录抽屉（双 Tab） |
| 书架         | 矩形 hitbox，覆盖书架区域            | 记忆抽屉                          |
| 床           | —                                    | 暂时不交互（视觉装饰）            |
| 员工角色本人 | 精确到角色像素范围                   | 快捷操作菜单                      |

**Hover 反馈：** Canvas 内在 hitbox 上绘制暖色发光边框 + DOM 层显示 tooltip（家具名称）

**点击背景空白区域：** 关闭当前打开的抽屉（类似点击遮罩）

**家具可发现性策略（两层结合）：**

Layer 1 — 持续策略（策略B，主要依赖）：

- hover 家具时 Canvas 绘制发光边框 + tooltip，视觉语言自解释
- 光效只在 hover 时出现，不持续闪烁（避免房间视觉噪音）

Layer 2 — 首次引导（策略A轻量版，一次性）：

- 用户首次进入任一员工工作空间时
- 电脑和书架各播放一次"轻微跳动"动画（1秒，无文字说明）
- 暗示可交互，之后永不再出现
- 不打断沉浸感，不弹全屏引导
- **首次跳动仅限可交互家具（当前：电脑/书架），不可交互家具（当前：床）不触发**
  （床无交互，跳动会误导用户去点击，形成错误预期）

### 7.3 动画分级规范（A/B/C）

**A 级 — 必须实现（MVP）**

> **骨骼动画是 P0，不是可选项。** 员工必须是"活的"，这是"养"的核心体现。序列帧只作为降级兜底，正常情况下不应出现。

```
每个状态必须有生动循环动画（不允许静态图，角色必须"活的"）：
  动画方案：骨骼动画（P0）；序列帧仅作降级兜底
  - 工作中：键盘打字骨骼循环（手部/身体节奏感，流畅）
  - 空闲：呼吸/眨眼骨骼循环（慢速，有生命感）
  - 高压：快速打字骨骼循环（节奏明显快于工作中）
  - 出错：停顿 + 望屏幕骨骼循环（困惑/皱眉，等待处理）
  - 休眠：缓慢呼吸骨骼循环（接近静止但不是静态图）

状态枚举切换（无走路寻路）：
  - 工作中 → 空闲：当前帧播完 → 切换到空闲循环
  - 空闲 → 工作中：直接切换到工作中循环
  - 任意状态 → 出错：立即切换到出错循环（红色角标同步出现）
  - 切换过渡：200ms 淡入淡出

退场 / 入场过渡（进入/离开工作空间）：
  - 进入：卡片展开 → 房间淡入（300ms）
  - 离开：房间淡出 → 返回主场景（200ms）

操作始终不等动画：
  - 退场/入场过渡期间 HUD 输入框可用
  - 退场/入场过渡期间家具可点击
  - 返回按钮始终可用
```

**B 级 — 体验加分（P1）**

```
Toast 出现时卡片轻微弹跳（scale 1.0 → 1.03 → 1.0，200ms）
高压状态角色轻微抖动叠加效果
```

**C 级 — 可选惊喜（P2）**

```
召唤新员工：像素角色从光点中出现（3秒召唤动画）
日夜循环：角色在深夜状态下打哈欠动画
成就解锁：短暂的粒子特效
```

**已明确取消：**

- ~~角色在房间内走路寻路~~（已降级，不实现）
- ~~连发指令时排队等走路动画结束~~（不再需要，状态切换即时）

## 七A、抽屉交互规范

### 7A.1 抽屉打开/关闭动画

```
打开抽屉（点击家具 hitbox 触发）：
  从右侧滑入，300ms ease-out
  同时：房间可视区域向左收缩（Canvas 不缩放，只是抽屉覆盖）
  抽屉宽度：320px，高度：100vh

关闭抽屉：
  方式1：点击房间背景空白区域（Canvas 无 hitbox 区域）→ 300ms ease-in 滑出
  方式2：点击抽屉顶部 [×] 按钮 → 同上
  方式3：按 ESC 键 → 同上
  不支持拖拽关闭（P2可考虑）
```

### 7A.2 多抽屉冲突处理

房间内只有两种抽屉（电脑 / 书架），同时只能打开一个。

```
场景：用户打开了书架抽屉（记忆），然后点击电脑：
  书架抽屉：300ms 滑出（关闭）
  电脑抽屉：等书架抽屉完全关闭后，300ms 滑入（打开）
  总时长：600ms，视觉上是"换一个"

  NOT：两个抽屉同时展开（空间不足，体验混乱）
  NOT：新抽屉直接替换旧抽屉内容（无动画的跳变感差）

场景：用户点击已打开的同一家具（如再次点击书架）：
  当前抽屉关闭（300ms 滑出）
  toggle 行为，不重新打开
```

### 7A.3 抽屉内滚动

```
抽屉内容超出高度时：
  内容区域可滚动（overflow-y: auto）
  滚动条：细线条暖色（--vo-border-dim），不占用宽度（overlay scrollbar）
  顶部 Tab 栏和底部输入框（如有）固定不滚动

  电脑抽屉：Tab 栏固定顶部，任务列表/聊天记录各自独立滚动
  书架抽屉：搜索框固定顶部，记忆列表滚动
```

### 7A.4 抽屉与 HUD 的关系

```
底部 HUD 输入栏始终在抽屉之上可见：
  HUD z-index > 抽屉 z-index
  抽屉打开时 HUD 不消失、不被覆盖
  HUD 发送的消息默认发给当前工作空间的员工

  视觉上：
  左侧3/4：Canvas房间（被抽屉部分遮挡）
  右侧1/4：抽屉（320px）
  底部：HUD（始终全宽可见）
```

### 7A.5 主场景浮层的冲突处理

主场景的浮层（快速发送浮层、更多操作菜单、搜索结果下拉）之间的冲突规则：

```
同时只能有一个浮层：
  打开快速发送浮层时，若更多操作菜单已展开 → 菜单立即关闭
  打开搜索结果时，若快速发送浮层已展开 → 浮层立即关闭
  点击任意浮层外区域 → 当前浮层关闭

  NOT：多个浮层叠加显示
```

## 八、状态体系定义

### 8.1 员工状态（主场景卡片可见）

| 状态         | 颜色信号           | 含义                                  | 触发条件                      |
| ------------ | ------------------ | ------------------------------------- | ----------------------------- |
| 工作中       | 绿色边框发光       | 工作实例活跃，在执行任务              | 有任务 running                |
| 空闲         | 正常边框           | 无活跃任务，等待指令                  | 无任务 / 任务 completed       |
| 高压         | 橙色边框           | STRESS > 70%，仍在工作                | STRESS 超阈值                 |
| 出错         | 红色边框 + 角标 ❗ | 任务执行失败，等待处理                | 任务 failed                   |
| 等待安全停止 | 黄色边框脉冲       | block类型任务中断中，正在完成当前步骤 | 用户点中断 + 任务类型为 block |
| 休眠         | 灰色，降透明度     | 用户手动暂停                          | 用户操作                      |
| 禁用         | 深灰，几乎不可见   | 实例全停止                            | 用户操作                      |

**等待安全停止状态说明：**

```
用户点中断时：
  - block 类型（不可立即停止的操作，如 git commit 进行中）：
    → 边框立刻变黄色脉冲
    → 角色播放"举手暂停"骨骼循环动画
    → 文字："正在完成当前步骤，稍后停止"
    → 完成后自动变空闲

  - cancel 类型（可立即停止的操作）：
    → 立刻停止，直接变空闲

用户可见信息：仅显示"已收到中断请求，等待安全停止"
不暴露 cancel/block 技术类型
```

### 8.2 STRESS 机制（体验约束）

注：等级/经验条不要，但 STRESS 作为员工的内在状态保留——它驱动动画变化和状态色，是"养"的感知来源。不在主场景显示进度条数字，改用卡片颜色信号传达。

**STRESS 计算规则：** 见 PRD 技术定义（context window 使用率 × 0.6 + 任务队列深度 × 0.4），流程文档不重复定义具体数值，以 PRD 为准。

```
STRESS > 70% 时 → 高压状态（体验层规则）：
  - 卡片边框变橙（颜色信号，不显示数字）
  - 角色切换为高压骨骼动画（更快的打字频率）
  - 不自动拒绝任务（用户自主决策）
  - Toast 提示："小快压力较大，建议让他休息一下"

不做强制限制的原因：用户有自主权，颜色信号透明后让用户决策
```

### 8.3 消息排队反馈

```
员工忙碌时（isIdle: false）用户发消息：
  发送后：
    ① HUD 输入框变为"已排队"状态（不是禁用，是确认反馈）
    ② Toast："已排队，小快完成当前任务后会看到"
    ③ 卡片右上角出现数字角标（显示队列中任务数）

队列中发了多条：
  角标数字更新（例：3）
  用户可点击角标查看队列列表，支持取消队列中的任务

队列上限提示（产品约束）：
  当队列中已有 4 条消息时，发第 5 条前弹出提示：
  "已有 4 条消息在等待，确定继续发吗？"
  [继续发送] [查看队列]
  （防止用户在不知情情况下堆积大量消息）
```

### 8.4 出错通知推送策略

```
分3级，不打断"扫一眼"仪式感：

Level 1 — 应用内被动展示（始终存在，无推送）
  主场景红色边框 + 角标 ❗
  用户打开应用即可0步发现
  适用：所有出错状态

Level 2 — 应用内 Toast（用户当前在应用内时触发）
  员工从工作中突然变出错时，弹一条 Toast
  文案："小快遇到了问题，点击查看"
  3秒自动消失，角标保留
  适用：用户在应用内但视线不在该卡片时

Level 3 — 系统通知（高优先级，需用户授权）
  触发条件：
    ① 员工出错超过 [待裁决：10分钟] 无人处理
    ② 应用在后台
    ③ 用户已授权系统通知
  文案："小快的任务执行失败，需要你处理"
  不用于普通出错，只用于长时间无人响应

设计原则：用户打开应用时不需要推送（应用已展示）；
          用户不在应用时，低优先级不打扰，高优先级推一次。
```

**待裁决：Level 3 触发时间阈值（建议10分钟，产品确认）**

## 十、降级策略

### 动画降级（不影响功能）

| 场景                 | 降级方案                                       |
| -------------------- | ---------------------------------------------- |
| 骨骼动画资源加载失败 | CSS sprite 序列帧代替（相同状态，流畅度降低）  |
| 骨骼动画库不可用     | 序列帧兜底，保持视觉"活的"要求                 |
| 低性能设备           | 检测 prefers-reduced-motion，序列帧 + 降低帧率 |
| 多员工同时动画卡顿   | IntersectionObserver：只播放可视区域内的动画   |

### Canvas 降级

| 场景            | 降级方案                            |
| --------------- | ----------------------------------- |
| Canvas 渲染失败 | 回退到静态像素图背景 + DOM 卡片布局 |
| 日夜光线不支持  | 固定白天状态，不影响功能            |
