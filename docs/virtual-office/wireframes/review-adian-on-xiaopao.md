## 一、6条用户流程 × 视觉规范匹配性审查

### A1 首次进入流程

**问题：空状态引导页的文案不符合像素游戏风格**

小跑写：

> 中央漂浮文字："你的团队还没有人"

这是标准 SaaS 空状态文案，和赛博朋克像素风严重割裂。

我的规范里（第九节）空状态：

> 网格显示虚线框 + 「招募第一位员工」引导文字（--vo-text-dim）

但这个文案也还不够。建议统一为：

```
NO AGENTS DETECTED
[ + RECRUIT FIRST AGENT ]
```

用 `Press Start 2P` 像素字体，`--vo-text-dim` 色，字母全大写，间距 0.15em。中文版保留为副标题，小号显示即可。

**结论：需修订文案和字体规范引用**

### A2 创建员工流程（B2 模版选择页）

**严重问题 1：emoji 和像素风格系统不兼容**

小跑原型里：

```
│  👨‍💻         │  │  🎨         │  │  📋         │
│ 全栈工程师  │  │  UI设计师   │  │   产品经理  │
```

以及 B1 主场景：

```
│[🖥️点击工作]│  │[💬点击聊天]│  │[💤已休眠]│
```

**这是最严重的视觉问题**。系统 emoji 是 Apple Color Emoji / Segoe UI Emoji 的矢量图标，渲染在 `--vo-bg-deep` 深色背景上时：

- 颜色不受 CSS 控制（无法应用 `--vo-state-*` 颜色系统）
- 缩放行为和像素图不一致（emoji 有自己的抗锯齿，和像素画的 `image-rendering: pixelated` 冲突）
- 不同系统上 emoji 外观差异极大（macOS 的 🖥️ 和 Windows 的 🖥️ 外观完全不同）

**全部替换方案**（对应我草案第四节的像素角色设计规范）：

| 小跑原文          | 替换为                                                    |
| ----------------- | --------------------------------------------------------- |
| `👨‍💻` (全栈工程师) | 24×24px CSS 像素画，青色系（同我草案第四节前端 Dev 规范） |
| `🎨` (设计师)     | 24×24px CSS 像素画，粉色系                                |
| `📋` (产品经理)   | 24×24px CSS 像素画，紫色系                                |
| `🤝` (团队总监)   | 24×24px CSS 像素画，金色系                                |
| `⚙️` (自定义)     | Unicode `⚙` 即可，配合 `--vo-text-secondary` 色           |
| `[🖥️点击工作]`    | `[▶ 工作区]`（Unicode，`--vo-primary` 色）                |
| `[💬点击聊天]`    | `[◎ 聊天]`（Unicode）                                     |
| `[💤已休眠]`      | `[○ 休眠]`（Unicode 空心圆，`--vo-state-idle` 色）        |

### A2 创建流程（B2 卡片交互歧义）

**问题：卡片点击行为定义矛盾**

小跑的 Step 1 写：

> 用户点击卡片 = 选中（高亮边框）
> 底部按钮：[下一步 →]

又在 B2 原型里，每个卡片底部有 `[选择]` 按钮。

**结果：用户不知道点卡片还是点按钮。** 这是经典的"双重入口"歧义。

我草案的卡片行为是 hover → scale 1.05 + glow，click → 进入工作空间。但在**模版选择这个特定场景**里，点击语义不同（是选中，不是进入）。

**建议统一方案**：

- 卡片点击 = 选中并直接进入下一步（移除 `[选择]` 按钮）
- 选中后卡片用 `--vo-border-active` 色高亮（`rgba(0,255,204,0.6)`）+ scale 1.03
- 底部只保留 `[← 返回]` 和 `[下一步 →]`，点击卡片自动激活 `[下一步 →]`

### A4 工作空间（B3 原型）

**问题：左侧面板尺寸严重不足**

小跑 B3 左侧面板内容清单：

1. 像素角色（站立动画）
2. 状态指示器（`● 工作中`）
3. 能力值（智慧 ████░ / 行动力 ███░░ / 社交 ██░░░）
4. EXP: 750/1000 + 进度条
5. 模块标签（[程序员模块] [已激活]）

按我草案的工作空间布局（第五节），左侧面板是 240px。用 11px JetBrains Mono：

```
计算：
- 像素角色 64×64px → OK
- 状态文字「● 工作中」10px → OK（30px行高）
- 能力值单条「智慧  ████░」→ 需要约 140px，240px 内 OK
- EXP 数字「750/1000」→ 9字符×7px=63px → OK
- 模块标签「[程序员模块]」→ 9字符×8px=72px → OK
```

**实际问题**：小跑原型里左侧列宽只有大约 14 个字符宽（约 98px），根本装不下能力值 bar。小跑在写原型时没有考虑实际字符宽度和内容量的比例。

**建议**：小跑草案的 B3 原型需要重绘，或者明确标注「左侧面板宽 240px，参考 adian-draft.md 第五节布局」，不要在 ASCII 原型里压缩左侧列宽。

## 二、动效规范一致性审查

### C1 卡片 → 工作空间转场

小跑方案：

```
1. 卡片轻微放大（scale 1.0 → 1.05，50ms）
2. 卡片中的像素角色"走出"卡片（向上漂浮）
3. 卡片背景扩展铺满屏幕（300ms expand动画）
4. 工作空间内容渐入（fade in，200ms）
```

总计：约 550ms（步骤 2 和 3 可能并行，但 2 的时长没写）

我的方案（adian-draft.md 第五节）：

```
1. 卡片 scale 放大到 1.2（100ms）
2. 背景虚化（blur:0 → blur:8px，100ms，与步骤1并行）
3. 卡片展开充满屏幕（200ms）
4. 工作空间内容淡入（100ms）
```

总计：400ms

**差异点：**

- 小跑：scale 1.05，我：scale 1.2（只差 0.15，可接受，统一用 1.1 作折中）
- 小跑加了「角色走出卡片」动画，这是额外的像素动画，我的规范里没有，但视觉效果好。**采纳小跑的这个细节，补充到我的规范里。**
- 总时长：小跑 ~550ms vs 我 400ms。超过 400ms 用户开始感觉慢。建议统一为 400ms，压缩步骤 2（角色走出 + 背景扩展并行，100ms 内完成走出动画，走出到一半时开始 expand）

**统一方案（需要小跑和我各自更新草案）：**

```
1. 卡片 scale 1.0 → 1.1（80ms，与步骤2并行）
2. 角色像素图向上浮出卡片 4px（80ms）
3. 背景虚化 blur 0 → 6px（80ms，与步骤1/2并行）
4. 卡片 expand 充满屏幕（220ms）
5. 内容 fade in（100ms）
总计：400ms
```

## 三、体验问题预判中缺少的视觉风险

小跑的 D4 是技术降级方案，但完全没有提视觉层面的风险。补充以下 5 条：

### 视觉风险 1：像素角色在高 DPI 屏幕上的模糊问题

像素艺术最大的敌人是浏览器的默认抗锯齿。16×16 的像素图在 2x Retina 屏幕上，如果没有正确设置 `image-rendering`，会自动双线性插值，变成模糊的一团。

**必须设置：**

```css
.vo-sprite {
  image-rendering: pixelated; /* Chrome/Safari */
  image-rendering: crisp-edges; /* Firefox */
  -ms-interpolation-mode: nearest-neighbor; /* IE（不重要，可忽略） */
}
```

这不是「降级」，而是**必须默认开启**的规范。小跑的草案完全没提，需要补充。

### 视觉风险 2：暗色背景下的色彩对比度问题

`--vo-bg-deep: #080B14` 背景上，以下颜色组合需要检查 WCAG 对比度：

| 文字色                         | 背景色              | 对比度 | 是否达标（WCAG AA = 4.5:1）     |
| ------------------------------ | ------------------- | ------ | ------------------------------- |
| `--vo-text-dim: #475569`       | `#080B14`           | ~5.1:1 | 勉强达标，边缘情况              |
| `--vo-state-idle: #4B5563`     | `#141E35`（卡片bg） | ~3.8:1 | **不达标** — 休眠状态文字看不清 |
| `--vo-text-secondary: #94A3B8` | `#080B14`           | ~9.2:1 | 达标                            |

**修复建议：** `--vo-state-idle` 色在卡片背景上要提亮到 `#6B7280`（对比度 ~5.2:1）。这个修改只影响卡片内的休眠文字，不影响其他场景。

### 视觉风险 3：主场景卡片网格在 13" 屏幕（1280×800px）下的压缩问题

我的草案设计基准是 1440×900px。在 1280×800px 屏幕上：

- 左侧面板（200px）+ 右侧面板（280px）+ 间距 = 520px
- 可用主画布宽度：1280 - 520 = 760px
- 标准卡片 160px + 20px 间距 = 180px
- 可放卡片数：760 / 180 = 4.2，即每行只能放 4 张卡片

**这意味着 9 个员工需要 3 行，主画布高度需要 3 × (220+20) = 720px，刚好 800px 内勉强放下，但几乎没有上下留白。**

小跑的「超过8个员工」问题预判是对的，但没给出视觉解决方案（见下一节）。

### 视觉风险 4：数字雨背景在多卡片时的视觉干扰

当员工卡片密集时（6+），数字雨背景即使只有 `opacity: 0.04`，也可能在卡片之间的间隙区域制造视觉噪音，让用户的眼睛无法聚焦在卡片上。

**解决方案：** 在 `z-index` 层级上，卡片要在数字雨之上（已在我的规范里），但需要额外给卡片背景加 `backdrop-filter: blur(0)` 以创建一个独立的渲染层，防止数字雨透过卡片的半透明区域「透出来」。

### 视觉风险 5：CRT 扫描线在低亮度显示器上的叠加效果

`--vo-scanlines` 的 `opacity: 0.03` 在标准亮度显示器上是 subtle 的，但在低亮度（<100 nit）的显示器或者暗室环境下，扫描线会变得很明显，甚至影响文字可读性。

**建议：** 将扫描线的 opacity 绑定 CSS 媒体查询：

```css
@media (prefers-color-scheme: dark) {
  .vo-scanlines {
    opacity: 0.02;
  }
}
```

并提供用户设置开关（「关闭扫描线效果」复选框，默认开启）。

## 四、6人以上员工的主场景可读性 — 视觉解决方案

小跑在 D1 识别了「员工超过 6 人后可读性」问题，但没有给出视觉解决方案。以下是我的方案：

### 方案一：状态分组显示（默认）

超过 6 个员工时，自动按状态分区渲染，而不是混排网格：

```
┌──────────────────────────────────────────────────────────┐
│  ◈ ACTIVE  [3]              ◈ IDLE  [4]  ◈ OFFLINE [2] │  ← 分组标签栏
├─────────────────────┬────────────────────────────────────┤
│  工作中 (3)          │  空闲/休眠 (6)                     │
│                     │                                    │
│  ┌────┐ ┌────┐ ┌────┐│  ┌──┐┌──┐┌──┐┌──┐┌──┐┌──┐       │
│  │卡片│ │卡片│ │卡片││  │小│ │小│ │小│ │小│ │小│ │小│    │
│  │(工)│ │(工)│ │(工)││  │卡│ │卡│ │卡│ │卡│ │卡│ │卡│    │
│  └────┘ └────┘ └────┘│  └──┘└──┘└──┘└──┘└──┘└──┘       │
│                     │  空闲卡片缩小到 80×120px            │
└─────────────────────┴────────────────────────────────────┘
```

**实现细节：**

- 工作中员工：保持标准尺寸（160×220px），左区域
- 空闲/休眠员工：缩小到 80×120px（精简版卡片，只显示头像+名字+状态点，无能力条）
- 两区域由竖分割线（`--vo-border-dim`）分隔
- 触发阈值：员工总数 ≥ 6

### 方案二：缩放模式（用户可切换）

在主场景顶部工具栏加入「视图切换」：

```
[◫ 标准视图]  [⊞ 紧凑视图]  [▤ 列表视图]
```

**紧凑视图（Compact）：** 所有卡片缩至 80×80px，只显示头像+状态颜色点
**列表视图（List）：** 员工变成行列表，每行包含头像（32px）+ 名字 + 状态 + 当前任务文本

### 方案三：虚拟滚动 + 视口外动画暂停

超过 9 个员工时，主画布变为可滚动区域：

- 视口内：正常渲染所有动效
- 视口外（IntersectionObserver 检测）：暂停像素动画（小跑 D4 已提，和此方案对齐）
- 滚动条：隐藏系统默认滚动条，用 `--vo-primary` 色的像素风自定义滚动条替代

### 推荐方案

**P0 阶段用方案一（状态分组），代码量最少，效果最好。** P1 阶段加入方案二的视图切换。方案三是性能优化，P1 后期加入。
