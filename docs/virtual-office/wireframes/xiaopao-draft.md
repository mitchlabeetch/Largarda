# 体验-小跑 用户流程草稿

> 视角：真实用户操作流程，关注"走不走得通"而不是"好不好看"

---

## A. 完整用户流程图

### A1. 首次进入流程

```
用户首次打开应用
       │
       ▼
  [主导航左侧栏]
  发现"虚拟办公室"入口
       │
       ▼
  点击进入虚拟办公室
       │
       ▼
  【首次进入判断】
  ┌─────────────────────────────────────┐
  │  检测：是否有任何员工存在？         │
  │  否 → 进入【空状态引导页】          │
  │  是 → 直接进入【办公室主场景】      │
  └─────────────────────────────────────┘
       │（无员工分支）
       ▼
  【空状态引导页】
  ┌──────────────────────────────────────┐
  │  全屏像素风场景：空旷的办公室        │
  │  中央漂浮文字："你的团队还没有人"   │
  │  发光按钮：[+ 招募第一个成员]        │
  │                                      │
  │  不强制教程，直接给入口              │
  │  （教程作为可选 "?" 角落按钮）       │
  └──────────────────────────────────────┘
       │
       ▼
  进入员工创建流程（见 A2）
```

**设计决策：不做强制引导教程**
原因：养成游戏的惊艳感来自"自己探索发现"。强制教程会打断沉浸感，且用户对游戏界面有直觉（能点就点，能拖就拖）。提供可选的 "?" 帮助入口即可。

### A2. 创建第一个员工的流程

**目标：3步内完成，不超过**

```
点击 [+ 招募成员] 或 [+ 新员工]
       │
       ▼
  【Step 1: 选择模版】（必选，降低门槛）
  ┌──────────────────────────────────────┐
  │  3~5个模版卡片展示：                 │
  │  · 全栈工程师                        │
  │  · 产品经理                          │
  │  · 团队总监                          │
  │  · [自定义（高级）]                  │
  │                                      │
  │  每个卡片有：角色头像 + 一句话描述   │
  │  + 能力标签（代码/设计/管理等）      │
  │                                      │
  │  用户点击卡片 = 选中（高亮边框）     │
  │  底部按钮：[下一步 →]               │
  └──────────────────────────────────────┘
       │
       ▼
  【Step 2: 起名字】（最快路径）
  ┌──────────────────────────────────────┐
  │  上方：选中的模版头像（像素风）      │
  │  中央：                              │
  │    给他起个名字：[___________]       │
  │    （可选）个性描述：[___________]   │
  │                                      │
  │  右侧：实时预览卡片（随输入更新）    │
  │                                      │
  │  [← 返回] [跳过描述，直接创建 →]    │
  └──────────────────────────────────────┘
       │
       ▼
  【Step 3: 完成动画】
  ┌──────────────────────────────────────┐
  │  像素风员工从卡片中"召唤出来"       │
  │  闪光 + 粒子效果（类似游戏招募）    │
  │  显示：员工名字 + "已加入你的团队!" │
  │  自动3秒后跳转到员工工作空间         │
  │  或 [立刻查看他的工作空间]          │
  └──────────────────────────────────────┘
       │
       ▼
  进入员工工作空间（见 A4）
```

**流程分析：**

- 最快路径：选模版 → 填名字 → 确认 = 3步，<30秒
- 最慢路径：选模版 → 填名字 + 描述 → 确认 = 仍是3步
- 不让用户在创建时配置提示词（太专业），工作空间里再调

### A3. 日常使用流程

```
打开 AionUi
       │
       ▼
  主导航 → 虚拟办公室
       │
       ▼
  【办公室主场景】（见 B1 原型）
  展示所有员工卡片/角色站位
  实时状态可见：发光=工作中，暗=空闲
       │
  ┌────┴─────────────────────────────┐
  │                                  │
  ▼                                  ▼
查看某个员工状态                  分配任务给员工
  │                                  │
  ▼                                  ▼
点击员工卡片                    点击员工 → 工作空间
转场动画进入工作空间              → 输入框发消息/任务
  │                                  │
  ▼                                  ▼
工作空间页面（见 B3）            员工开始执行，状态变"工作中"
  │
  ├─ 查看他在干什么（实例A状态）
  ├─ 和他聊天（实例C）
  ├─ 看他的笔记/记忆（工作空间）
  └─ 点击电脑图标→沟通记录（见 A5）
       │
       ▼
  看完，返回办公室主场景
  （面包屑/侧边栏返回入口）
```

**关键体验节点：**

- 状态实时可见（主场景就能感知团队在干啥）
- 从主场景到工作空间：1次点击
- 从工作空间回主场景：随时可见的返回入口

### A4. 员工工作空间使用流程

```
进入员工工作空间
       │
       ▼
  【工作空间主视图】
  左侧：员工角色（站立动画）
  右侧：分标签页
    [当前任务] [工作笔记] [记忆文件] [设置]
       │
  ┌────┴────────────────────────────────┐
  │                                     │
  ▼                                     ▼
发起新任务                          查看历史记忆
  │                                     │
  ▼                                     ▼
底部输入框 → 输入任务描述            点击 [记忆文件] 标签
→ 发送                               展示员工积累的经验
  │                                   知识条目列表
  ▼
员工角色动画切换 → "工作中"
实例A状态显示：
  [正在执行：写代码...]
  [工具调用：git commit...]
  │
  ▼
任务完成后
状态变回空闲
右侧自动刷新 [工作笔记] 中多了新条目
```

### A5. 搜索与查找流程

```
用户想找某个工作记录/笔记
       │
       ▼
  两个入口：
  A. 主场景顶部搜索栏（全局搜索）
  B. 员工工作空间内搜索（局部搜索）
       │
       ▼
  【A. 全局搜索】
  输入关键词 → 回车
       │
       ▼
  搜索结果页（见 B5 原型）
  按"员工名"分组显示
  每条结果：[员工头像] [内容片段] [来源类型：笔记/代码/聊天]
  点击结果 → 跳转到对应员工工作空间定位内容
       │
       ▼
  【B. 工作空间内搜索】
  右侧标签页内有搜索框
  实时过滤笔记/记忆列表
```

### A6. 员工协作流程（P1，流程设计先行）

```
用户需要多个员工协同完成任务
       │
       ▼
  方式一：用户手动跨员工分发
  → 进入员工A工作空间 → 发任务
  → 返回主场景 → 进入员工B → 发任务
  （当前P0阶段，流程虽笨但走得通）
       │
       ▼
  方式二（P1）：领导角色员工统一协调
  → 进入总监员工的工作空间
  → 发送综合任务
  → 总监实例A自动调用管理模块
  → 向其他员工下发子任务
  → 主场景可见多员工同时变"工作中"
       │
       ▼
  协作可视化（P1）：
  员工卡片间出现"连线"动效
  表示正在协作中
```

| 沟通记录和工作空间的区别不清楚       | 中                 | 用tab分开，工作空间是"他的地盘"，沟通记录是"聊天历史"        |

| 当前流程                                 | 问题        | 简化建议                                |
| ---------------------------------------- | ----------- | --------------------------------------- |
| 创建员工：选模版→起名→确认               | 已是3步，OK | 保持，不要再拆                          |
| 查看沟通记录：点卡片→进工作空间→切换标签 | 3次操作     | 主场景卡片的[🖥️]直接打开抽屉，减为1次   |

### D3. 首屏加载慢的过渡方案

```
加载顺序：
1. 立刻显示：深色背景 + 办公室轮廓线框（CSS，无依赖）
2. 100ms后：员工卡片骨架屏（灰色闪烁，占位）
3. 数据返回后：卡片内容渐入，角色动画开始
4. 动画资源加载中：先用CSS静态像素图代替，骨骼动画加载完再切换

关键原则：
- 绝不显示空白页+菊花转圈
- 先展示结构，数据后填
- 动画是增强，不是必要（降级到静态不影响核心功能）
```

### D4. 像素动画卡顿降级方案

| 场景                      | 降级策略                                             |
| ------------------------- | ---------------------------------------------------- |
| 骨骼动画库加载失败        | 回退到CSS sprite动画（预渲染帧序列）                 |
| CSS动画卡顿（低性能设备） | 检测 prefers-reduced-motion，关闭非核心动画          |
| 多员工同时动画（>6人）    | 只有可视区域内的员工播放动画（IntersectionObserver） |
| 粒子效果影响帧率          | 自适应粒子数量（基于requestAnimationFrame帧率检测）  |
| 转场动画跟不上用户操作    | 跳过中间帧，直接到结束状态                           |

### F2. 完整过渡动画对"任务开始"流程的影响

**用户需求：** 员工每个动作有完整的过渡动画，不是状态瞬间切换。
典型序列：下床 → 走到电脑前 → 坐下 → 开始打字

**这对用户流程的核心影响：**

"任务开始"的用户感知不再是状态标签变化（"工作中"文字出现），而是**观察到角色在房间里移动**。

```
用户发送任务（底部输入框）
       │
       ▼
系统接收任务，开始执行
       │
       ▼
【角色过渡动画序列】（用户看到的）
  若角色在睡觉：
    床上坐起（0.5s）→ 站起来（0.3s）→ 走向电脑（1-2s，取决于距离）
    → 坐下（0.3s）→ 开始打字动画
  若角色在看书：
    放下书（0.3s）→ 站起来（0.2s）→ 走向电脑（1s）→ 坐下 → 打字
  若角色已在电脑前（空闲状态）：

角色坐在电脑前开始打字 = 任务正在执行的视觉确认
```

**关键体验问题：过渡动画期间用户该看什么？**

过渡动画全程约 2-4 秒。在这段时间内：

- 底部HUD栏状态文字更新："收到任务，小快正在过来..."
- 不要在动画过程中弹出任何提示或loading，让用户安静看动画
- 过渡动画结束（角色坐下打字）后，HUD栏变为："正在执行：[任务名]..."

**降级处理：** 若动画资源加载失败，角色直接出现在电脑前，HUD栏状态立即更新。不能让用户看到角色"传送"到电脑前——降级时用淡入淡出代替移动动画。

### F3. 房间场景下的状态感知流程修订

**原流程（主场景）：** 员工卡片发光边框 + 状态文字 = 感知员工状态

**新流程（进入房间后）：** 看员工在做什么 = 感知员工状态

| 员工行为                       | 含义       | 对应实例     |
| ------------------------------ | ---------- | ------------ |
| 坐在电脑前打字                 | 执行任务中 | 工作实例活跃 |
| 站在书桌旁记笔记               | 整理记忆中 | 记忆实例活跃 |
| 坐在沙发和用户对话（气泡出现） | 聊天中     | 社交实例活跃 |
| 在房间里随机闲逛/看书/看电视   | 空闲等待   | 全部实例待机 |
| 躺在床上/坐着发呆              | 休眠       | 实例暂停     |

**主场景卡片仍需保留状态指示：** 用户在主场景不会进入每个员工的房间，还是需要卡片上的状态颜色（发光/暗/灰）快速判断。房间内的动画是"进入后的沉浸感知"，卡片是"场外快速扫描"，两套系统并行。

### F4. 体验风险：过渡动画时长 vs 用户耐心

**风险：** 用户连续发任务时，每次都要等 2-4 秒的"走到电脑前"动画，操作节奏会被打断。

**解决方案：**

```
场景1：角色已在电脑前（上一个任务刚完成）
  → 直接切入打字动画，无过渡，体验流畅

场景2：角色在休息区
  → 播放完整起身+走路动画（用户第一次看会觉得有趣）
  → 但如果用户连续3次以上看到这个动画，可以设置"快速模式"
     快速模式：走路动画加速到0.5倍时长

场景3：用户连续快速发多条指令（<3秒内发第二条）
  → 第二条任务排队，不打断正在进行的过渡动画
  → 过渡动画完成后，立刻开始第二条任务（无需再次走到电脑，已经在了）

场景4：用户明确不想看动画
  → 设置里可以关闭过渡动画，直接跳到目标状态
  → 对应 prefers-reduced-motion 系统设置
```
