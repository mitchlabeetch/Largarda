# 体验审查：小跑审阿点草案

> 审查角度：用户流程走得通吗？布局支撑用户操作吗？
> 参考文件：adian-draft.md v0.1 | 审查日期：2026-04-01

---

## 总体判断

阿点的视觉规范完整度很高，色彩体系、像素角色、动效细节都很扎实。
**但有5个流程级问题需要在实现前解决，否则会出现用户走不通的死角。**

严重程度说明：

- 🔴 阻断：流程走不通，用户会卡死
- 🟡 降级：流程能走但体验变差，用户会困惑
- 🟢 建议：可以更好，不做也不阻断

## 问题1：三栏布局中，"创建员工"流程无落脚点 🔴

**阿点的布局：** 左侧200px（导航/过滤）+ 主画布（卡片网格）+ 右侧280px（任务/资源）

**我的流程A2需要：** 3步创建流程，Step1选模版（卡片展示）→ Step2起名字 → Step3召唤动画

**问题：** 三栏布局里没有定义创建员工的流程如何承载。
主画布有 `[+招募]` 按钮，但点击后触发什么？

- 如果是弹窗（Modal）：主画布那么多动效背景，弹窗会显得很违和，且弹窗不适合"Step1选模版"这种需要横排展示多个卡片的布局
- 如果是页面跳转：三栏布局被完全替换，失去办公室场景带来的沉浸感

**建议：** 阿点需要明确"创建员工"这个全屏流程如何在三栏框架内呈现。我的建议是"全屏覆盖层"（overlay），保留背景虚化，内容居中展示选模版步骤，不是Modal也不是跳转。

## 问题2：160×220px 卡片在信息密度上存在优先级混乱 🟡

**阿点的卡片包含（从上到下）：**

1. 等级徽章 + 稀有度星级（LV.23 [★★★☆]）
2. 像素角色立绘（48×48px）
3. 姓名 + 职位
4. 状态文字（跑马灯）
5. HP bar + EXP bar
6. 能力值点阵（ATK/STL/MEM/SPD 共8个点阵）
7. 已装备模块标签

**按用户日常操作的信息需求排序（最重要→最不重要）：**

1. 员工名字（我在找谁）
2. 当前状态（他在干嘛）
3. 是否可以打扰（工作中/空闲）
4. 快速操作入口
5. 等级/成长信息（偶尔看）
6. 能力值点阵（几乎不需要实时看）

**问题：** 能力值点阵（ATK/STL/MEM/SPD）占用了卡片下方大约30px高度，但用户每次打开主场景只是想快速找到某个员工发任务，不会站在主场景看战斗数值。这8个点阵在160×220px的空间里字号只有10px以下，实际上扫一眼根本读不清楚。

数一下：160px宽里要塞进 `ATK ●●●●●○  STL ●●●○○○` 这26个字符+点阵，字号约9px，眼睛要凑近屏幕才能看清。

**建议：** 能力值点阵移到 hover 展开后才显示（200×280px 版本），或移到员工工作空间详情页。主卡片160×220px 只保留：状态指示、立绘、名字/职位、当前在做什么（一句话）、HP/EXP 条。

## 问题3：工作空间3实例面板：用户第一次看根本不知道那是什么 🔴

**阿点的3实例状态区域（右侧300px栏底部）：**

```
工作实例  ●  活跃     [任务名称] █████░ 70%
记忆实例  ◐  整理中   [最近记忆整理] ███░░ 40%
聊天实例  ○  待机     等待对话请求
```

**问题：** "工作实例/记忆实例/聊天实例"这三个词，用户第一次看到完全不知道是什么概念。PRD里说"三实例机制不需要让用户手动管理，但要理解"——但这个展示方式跟直接暴露实现细节一样，对用户没有任何翻译。

更严重的问题：这个3实例状态区域放在右侧300px的底部，是视觉权重最低的位置。但这其实是员工"正在干什么"的核心信息，应该是进入工作空间后第一眼看到的内容。

**实际用户心智模型对应：**

- 工作实例 → "他在完成我交给他的任务"
- 聊天实例 → "他在和我对话"
- 记忆实例 → "他在整理自己的笔记"（用户不关心这个）

**建议：** 把实例状态翻译成用户语言：

```
当前任务：修复登录bug ████░ 70%（工作实例的人话版）
对话：待机中（聊天实例，简化）
后台：整理记忆中（记忆实例，只在活跃时显示）
```

同时，这个信息应该放在中间主工作区最顶部，不是右侧栏底部。

## 问题4：状态动效规范漏了2个关键状态 🟡

**阿点定义的5种状态动效（严格说是6种）：**

1. 工作中（Working）
2. 休眠（Idle）
3. 忙碌（Busy）— 我的草案里叫"工作中"，这两个的区别没有明确定义
4. 禁用（Disabled）
5. 离线（Offline）
6. 角色动画表中还有：idle/working/thinking/busy/happy/offline

**我的流程里涉及但阿点漏掉的状态变化：**

**A. `failed`（任务执行出错）状态** — 这是我在源码分析中补充的，TaskStatus 有这个状态。
阿点定义了错误状态的颜色（`--vo-state-error: #EF4444`），但没有定义：

- 卡片出错时的具体动效（是边框变红？是闪烁？是叹号覆盖？）
- 工作空间里错误信息如何展示
- 错误后用户如何操作（重试/放弃/查看详情）的界面状态

**B. 消息排队等待状态** — 用户向忙碌员工发消息时，员工有`pendingUserMessages`队列。
卡片上没有定义"有待处理消息"的视觉提示。用户发了消息，卡片没任何变化，用户不知道消息是否被接收。

**建议：** 阿点需要补充：

- failed 状态的卡片视觉（建议：红色边框 + 叹号角标 + 角色静止）
- 待处理消息角标（卡片右上角小数字 badge，类似消息通知）

## 问题5：长时间使用的视觉疲劳风险 🟡

**问题分析：**

阿点的色彩方案使用了以下高亮度、高对比颜色作为主视觉：

- `--vo-primary: #00FFCC`（霓虹青，高亮度）
- `--vo-state-working: #00FF88`（纯绿色，高亮度）
- `--vo-state-error: #EF4444`（纯红色）
- 多处 glow 效果（`0 0 12px rgba(0,255,136,0.8), 0 0 24px rgba(0,255,136,0.4)`）

具体风险点：

**风险1：pulse 动画叠加**
当主场景有5个员工同时"工作中"，每个卡片都有 `vo-pulse-border 2s` 循环，5张卡片的光晕同时呼吸，视觉上会产生强烈的"闪烁感"。每个动画周期不同步（各自独立），整体画面会持续有不规则光影变化，20分钟以上使用容易视觉疲劳。

**风险2：`--vo-primary #00FFCC` 在深黑背景上的对比度**
WCAG 标准：正文需 4.5:1，大文字需 3:1。
`#00FFCC` 在 `#080B14` 上的对比度约为 9.8:1，本身符合标准，但饱和度极高的青色长时间注视会引起视觉残像（绿色残留在视网膜上）。

**风险3：扫描线 + 数字雨同时开启**
`opacity: 0.03~0.06` 的数字雨加上 `opacity: 0.02` 的扫描线，单独看确实 subtle，但两个叠加的低频闪烁在长时间使用时会引起轻度的眼部不适（类似看了太久电子墨水屏的感觉）。

**建议：**

1. pulse 动画改为"非同步但有相对平缓的相位"或加一个用户设置"减少动效"开关（对应系统的 `prefers-reduced-motion`，我在草案D4中已提到）
2. 为工作5个员工以上的场景提供"专注模式"，关闭卡片动效只保留状态色
3. 数字雨建议做成可关闭的设置项，而不是强制全局开启

## 问题6（额外发现）：搜索流程在三栏布局里没有容身之所 🟡

**阿点的顶部导航有搜索按钮**，但没有定义搜索结果如何展示：

- 是在主画布区域显示搜索结果（替换员工网格）？
- 是弹出一个新的覆盖层？
- 是在右侧面板里展示结果？

**我的流程A5**定义了搜索结果页的完整布局（按员工分组，支持类型筛选，可跳转到原文），但阿点的三栏框架里没有为这个页面预留位置。

这不算致命，但开发实现时会碰到——搜索结果需要独立的视图容器，三栏布局里谁来承载？

**建议：** 阿点定义一下搜索结果的展示层级（推荐：全屏覆盖层，保留背景，内容区域展示分组结果）。

## 互补确认（阿点做对的地方）

以下几点阿点的设计和我的流程完全对齐，不需要修改：

1. **状态色体系完整** — `--vo-state-*` 6种颜色覆盖了我定义的主要状态（除了漏掉的 failed 角标）
2. **hover 展开的快捷操作栏**（[对话][查看工作区][设置]）— 精准对应我流程里"从主场景快速触达"的需求
3. **新员工创建的卡片翻面动画** — 和我设计的"召唤感"一致，执行层面已经很具体
4. **骨架屏加载设计** — shimmer 动画方案和我流程D3的"先展示结构"策略完全一致
5. **错误状态的闪红处理** — `--vo-state-error 闪烁×3次`，符合我流程对错误提示的要求（虽然缺后续的错误恢复界面）

## 对阿点的回复

阿点提到"HP/EXP 数值来源需要小跑定义计算逻辑"，我的答复：

- **HP**：建议映射为"员工响应能力"，计算规则 = `100 - (任务失败次数 × 5)`，可以随时间缓慢恢复（类似游戏回血）
- **EXP**：任务完成数 → 经验值 → 等级换算，每完成一个任务 +EXP，任务复杂度影响加成
- **等级升级触发时机**：EXP满 → 升级动画 → 进入下一级（清零），这个时机告诉阿点以便她定义升级动效的触发点
