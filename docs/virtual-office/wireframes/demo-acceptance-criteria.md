# 员工工作空间 Demo — 验收标准

> 体验-阿点 | 2026-04-01
>
> 基于郭聪明产品验收标准，增加体验细节层
>
> **使用方式：**
> 1. 老锤开发完成后，先按本文档自查，逐项打 ✅ / ❌
> 2. 自查全部通过后，通知阿点做最终体验审核
> 3. 阿点确认后，报给郭聪明和总控交付

---

## 一、必须全部通过（硬性要求，缺一不可交付）

### 1.1 房间视觉完整性

| # | 检查项 | 标准 | 自查 |
|---|--------|------|------|
| 1 | 背景图显示 | room-full.png（1024×559）完整显示，无拉伸，无变形，宽高比正确 | |
| 2 | 房间元素可见 | 能清楚看到：床、书架、电脑桌、台灯、窗户、植物、猫（参考图本身含这些，不需要额外绘制） | |
| 3 | 像素不模糊 | Retina 屏（devicePixelRatio ≥ 2）上背景图无双线性插值模糊，像素边缘清晰 | |
| 4 | 整体有 cozy 感 | 色调暖棕（不偏冷蓝、不偏霓虹），整体看起来是像素风游戏房间，不是技术 demo | |

### 1.2 角色

| # | 检查项 | 标准 | 自查 |
|---|--------|------|------|
| 5 | 角色在房间里 | 角色出现在电脑桌前区域（默认位置），不在房间外或边缘超出画面 | |
| 6 | 角色有动画 | idle 状态有呼吸循环动效（不是静止图片）。使用代码角色（Employee.ts）时骨骼有明显运动 | |
| 7 | 角色不穿墙 | 角色不与家具大面积重叠，视觉上"坐在/站在"家具旁边 | |
| 8 | 角色像素风格 | 角色风格与背景图一致，不出现写实感、3D 感、卡通矢量感 | |

### 1.3 家具交互

| # | 检查项 | 标准 | 自查 |
|---|--------|------|------|
| 9 | 电脑桌热点存在 | 鼠标移到电脑桌区域有明确的 hover 反馈（颜色变化 / 边框出现 / 发光均可） | |
| 10 | 床热点存在 | 鼠标移到床区域有 hover 反馈 | |
| 11 | 书架热点存在 | 鼠标移到书架区域有 hover 反馈 | |
| 12 | 点击电脑桌弹面板 | 点击电脑桌弹出面板，面板出现在画面内（不超出视口），有可见内容（占位文字也可） | |
| 13 | 面板可以关闭 | 面板有明确的关闭方式（× 按钮 / 点空白区域均可），点击后面板消失 | |

### 1.4 加载和稳定性

| # | 检查项 | 标准 | 自查 |
|---|--------|------|------|
| 14 | 3 秒内可见 | 从打开页面到看到完整房间（有背景+有角色），不超过 3 秒（本地开发环境） | |
| 15 | 无控制台报错 | DevTools Console 无红色 Error（404、undefined、TypeError 等均不可有） | |
| 16 | 无白屏 | 页面不出现全白屏或全黑屏超过 1 秒 | |

## 二、体验细节层（阿点负责审核，开发可参考）

### 2.1 像素风视觉一致性

| # | 检查项 | 标准 | 阿点审核 |
|---|--------|------|---------|
| A1 | image-rendering 设置 | Canvas 元素或背景图 CSS 设置了 `image-rendering: pixelated`（Retina 屏不模糊） | |
| A2 | 角色和背景密度匹配 | 角色像素粒度不比背景细腻超过 1 倍，不比背景粗糙超过 1 倍 | |
| A3 | 热点高亮色协调 | 热点高亮颜色与房间暖棕色系一致，不出现 `#0078d4`（Windows 默认蓝）或纯亮绿 | |
| A4 | 无圆角控件泄露 | 弹出面板无大圆角（border-radius 应为 0 或 2px 以内），blocky 边框 | |

### 2.2 加载体验

| # | 检查项 | 标准 | 阿点审核 |
|---|--------|------|---------|
| A5 | 加载顺序 | 背景先于角色显示，不出现"角色先出现在黑色背景上"的过渡（过渡实现方式开发自决） | |
| A6 | 加载中无白屏 | 背景图加载完成前页面不出现白屏，过渡方式不限（深色占位/渐入/任意均可） | |

### 2.3 面板内容质量

| # | 检查项 | 标准 | 阿点审核 |
|---|--------|------|---------|
| A7 | 面板标题文案 | 面板标题有场景感（如「电脑前 · 任务进行中」「小快的工作台」），不是 `undefined`、`Modal`、`TODO` | |
| A8 | 面板占位内容 | 占位文字是中文场景说明（如「任务进度 · 待接入」「沟通记录 · 待接入」），不是英文 `placeholder` 或 lorem ipsum | |
| A9 | 过渡方案标注 | 若使用 Employee.ts 代码角色，面板内有注明「角色美术资产待替换」，且字色是低调注释色（不是红色报错感） | |

### 2.4 整体感受（主观，阿点判断）

| # | 检查项 | 判断方式 | 阿点审核 |
|---|--------|---------|---------|
| A10 | "好玩"测试 | 对着画面停留 10 秒：有没有想点点看的冲动？动画有没有让角色"活着"？ | |
| A11 | "技术 demo"排除 | 没有可见的调试信息（坐标数字、hitbox 线框、fps 计数器）暴露在画面上 | |
| A12 | 不说明也能看懂 | 不看任何文档，第一次打开就能理解"这是一个员工在他的房间里" | |

## 三、不在本次验收范围（不要做，做了也不算加分）

- 角色走路 / 寻路动画
- 多员工切换
- 真实数据接入（任务、消息等）
- 移动端 / 响应式适配
- 状态切换按钮（idle / working / sleeping）— 这次只要 idle 状态

---

## 四、判定规则

```
第一节（硬性要求）：16 项必须全部 ✅，有任意 ❌ 不可交付
第二节（体验细节）：阿点逐项标注，给出"合格 / 有问题 / 严重问题"结论
  - 严重问题：和第一节等同，必须修复
  - 有问题：记录、报给总控、由总控决定是否阻断交付
  - 合格：通过
```
