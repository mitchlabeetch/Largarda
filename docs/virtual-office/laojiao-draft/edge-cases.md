# 边界场景与异常处理

> 产品-老乔 独立产出
> 重点：用户遇到问题时，体验不能崩。

---

## 一、员工容量边界

### 场景：用户想创建第N+1个员工，但已达上限

**问题：** 上限应该是多少？怎么告知用户？

**用户心理：** 如果卡在创建流程里才发现上限，会非常沮丧。

**处理方案：**

- 在创建入口就显示"已使用 X/N 席位"，不要让用户走到一半才发现
- 达到上限时，创建按钮变灰，hover提示"已达上限，需要先退休一名员工"
- 不要用"请升级套餐"——至少在MVP阶段不做这种打断

**歧义：** MVP阶段上限是多少？需要产品决策。建议：3-5人（基于性能考虑，三实例×5人=15个并发LLM调用）

## 二、Agent实例异常

### 场景A：工作实例（实例A）挂了，但用户以为任务在进行中

**风险：** 静默失败是最糟糕的用户体验。用户等了半小时，发现什么都没发生。

**处理方案：**

- 实例状态监控，超过阈值无响应 → 自动标记为"疑似卡死"
- 员工卡片显示警告状态（不是直接变红，是一个"？"或感叹号）
- 推送通知："[员工名]的任务可能遇到了问题，点击查看"
- 用户可手动重启实例（而不是销毁员工重建）

### 场景B：社交实例（实例C）挂了，用户发消息没回应

**处理方案：**

- 消息发送后，超过X秒没有任何响应（哪怕"正在思考"的状态）→ 显示"[员工名]暂时无法回应，请稍后再试"
- 不要让对话框空转无响应

### 场景C：记忆实例（实例B）整理失败

**风险：** 这个最难排查，因为它是后台进程，用户感知不到失败。

**处理方案：**

- 记忆整理失败不影响其他实例正常运行（必须隔离）
- 失败有日志记录，但不打扰用户
- 如果连续多次失败，在工作空间显示轻量提示："记忆整理暂时中断，工作空间数据仍完整"

## 三、Token爆炸场景（基于Claude源码分析）

> 以下分析基于 Claude Code 的 `query.ts` + `compact.ts` 源码，是目前已验证可行的方案思路，不是假设。

### 场景：员工工作实例的上下文窗口接近满载

**这是最重要的边界场景之一。**

**用户感知：** 员工回答越来越慢、越来越奇怪、开始忘事

**Claude源码的处理方式（query.ts）：**
Claude Code 采用多层防御：

1. **自动压缩（autoCompact）**：监控token总量，达到阈值自动触发一次"压缩对话"——用fork出的单独agent把整个历史总结成摘要，替换原对话。
2. **微压缩（microCompact）**：每轮循环前检查是否需要轻量压缩工具调用结果，减少工具返回内容占用。
3. **工具结果预算（toolResultBudget）**：超过单条工具结果大小上限 → 保存到磁盘文件，对话里只保留路径引用。
4. **被动压缩（reactiveCompact）**：真正收到API的413 prompt-too-long才触发，属于兜底机制。
5. **max_output_tokens恢复**：输出截断时自动注入"继续，不要重复"的恢复消息，最多重试3次。

**对虚拟办公室的启示（产品层面）：**

- 压缩不是"失忆"，而是"摘要替换"——员工的核心能力和重要记忆会保留在摘要里
- 压缩对用户应该基本透明（Claude Code 做到了），员工工作不应该因此中断
- 工具模块（程序员模块等）的调用结果可能很大，需要有结果大小限制

**处理时机建议：**

1. 达到~70%上下文 → 触发记忆实例进行一次记忆整理（参考autoCompact触发时机）
2. 达到~90% → 显示轻量提示，但不中断工作
3. 真正超限 → 触发压缩，员工显示"正在整理记忆..."，压缩后继续

**不要做的：** 直接报错"context window exceeded" → 用户不知道这是什么意思

### 场景：工具模块调用结果体积巨大（如程序员模块读取大文件）

**Claude源码如何处理（Tool.ts中的maxResultSizeChars）：**
每个工具都有 `maxResultSizeChars` 字段限制结果大小。超出后结果保存到磁盘，只给Agent一个文件路径引用。这样对话token不会爆炸，但Agent仍可通过读文件获取完整内容。

**对虚拟办公室的启示：**

- 程序员模块读取代码文件时，必须有大小限制
- 超出限制 → 保存到员工工作空间，员工通过引用访问
- 用户视角：员工说"已把分析结果保存到工作空间"，而不是报错

### 场景：超长任务（用户要求分析整个代码库）

**处理方案：**

- 工作实例应该主动拆分任务，分批处理（Agent tool的subagent模式）
- 对用户的呈现："这是个大任务，我会分几步来做..."
- 不要接受然后崩溃

## 四、网络/API异常

### 场景A：网络断了，员工在工作中途断线

**处理方案：**

- 工作状态本地持久化（任务描述、已完成步骤）
- 网络恢复后自动续接，而不是从头开始
- 续接时告知用户："网络恢复，继续之前的任务..."

### 场景B：API限流（rate limit）

**处理方案：**

- 排队机制，不是直接报错
- 如果等待超过阈值，通知用户："当前请求较多，[员工名]需要等待一下"
- 排队中的员工状态显示为"等待中"，而不是"工作中"（避免用户误判）

### 场景C：API Key无效或余额不足

**这个必须明确处理，不能含糊。**

- 员工卡片整体变暗，悬浮提示"无法连接AI服务"
- 给出明确操作入口：设置 → API配置
- 不要让用户在员工这里绕圈子找不到解决入口

## 五、数据完整性边界

### 场景：用户手动删除了工作空间里的某个文件

**工作空间是员工记忆的载体，删掉可能导致员工"失忆"。**

**处理方案：**

- 工作空间文件操作有确认弹窗："删除后员工可能失去相关记忆，是否继续？"
- 重要文件（记忆核心文件）不能直接删，只能"归档"
- 归档文件保留30天，期间可恢复

### 场景：同一个员工被两个地方同时触发任务

**处理方案：**

- 工作实例同时只能跑一个主任务
- 第二个任务请求进入排队，对用户显示"[员工名]正在忙，你的任务已排队"
- 不要静默丢弃

## 六、用户误操作场景

### 场景：用户不小心点了"解雇"

**处理方案：**

- 二次确认，且说明后果
- 解雇后有"后悔期"（建议72小时），期间可以恢复
- 后悔期结束前，已退休员工在列表里有灰色展示，带"还有X天可恢复"

### 场景：用户编辑提示词基底后，员工行为变得很奇怪

**处理方案：**

- 提示词基底保留历史版本（至少保留最近3个版本）
- 操作路径：工作空间 → 提示词历史 → 回滚到某个版本
- 修改时有警告："修改提示词基底会影响员工后续行为，当前版本已自动保存备份"

---

## 七、多用户/多设备场景（边界情况）

**当前定位是桌面单机应用，但需要明确：**

- 员工数据是否跟账号走？（换电脑还在不在？）
- 同一账号多设备同时打开会怎样？

**建议MVP立场：** 明确声明"数据本地存储，不跨设备同步"，避免用户预期管理失败。

---

_老乔注：边界场景的处理原则——不能静默失败，不能用技术术语报错，每个异常都要给用户一个明确的下一步操作。_
