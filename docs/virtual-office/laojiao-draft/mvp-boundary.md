# MVP边界定义

> 产品-老乔 独立产出
> 核心原则：做少，但做到位。宁可少一个功能，不能有一个体验不完整的功能。

---

## 一、MVP的核心问题

**MVP要回答的问题只有一个：**

> 用户第一次用完之后，会不会明天再来？

如果会，MVP成功。如果不会，加再多功能也没用。

---

## 二、第一版必须有的（Hard Must Have）

### 2.1 视觉基础 — 打开就惊艳

- [ ] 虚拟办公室主场景（像素风，暗色赛博朋克）
- [ ] 至少1个有动画的员工角色（骨骼动画，有工作/空闲两种状态）
- [ ] 员工卡片展示（姓名 + 状态 + 当前在做什么）
- [ ] 整体视觉氛围到位 — **这是门槛，不是加分项**

**为什么是必须：** 如果打开和普通SaaS一样，产品定位就失败了，后面做再多也没用。

### 2.2 员工创建 — 用户的第一个行动

- [ ] 至少3个内置模板（程序员、产品经理、通用助手）
- [ ] 起名功能（必须，是情感投入的起点）
- [ ] 选头像（至少3-4种预设像素风角色）
- [ ] 创建流程不超过3步，创建后有入职动画

**为什么是必须：** 没有创建，就没有体验。

### 2.3 员工基础交互 — 真的能用

- [ ] 点击员工卡片 → 打开工作空间
- [ ] 给员工发消息（聊天/布置任务）
- [ ] 员工状态实时更新（工作中/空闲/聊天中）
- [ ] 员工完成任务后，工作空间有记录

**为什么是必须：** 这是"不只是好看"的证明。

### 2.4 三实例基础机制

- [ ] 工作实例和社交实例可以并发运行（发消息不打断正在进行的任务）
- [ ] 基础的实例状态监控（挂了能知道）

**为什么是必须：** 这是核心差异化，也是避免最糟糕的用户体验（聊天导致任务中断）。

### 2.5 最基础的异常处理

- [ ] 任务失败有通知，不静默失败
- [ ] API异常有明确提示 + 操作入口
- [ ] 解雇员工有二次确认

## 三、第一版不做的（Hard No）

### 3.1 功能模块（程序员/管理/办公软件模块）

**为什么不做：** 模块化工具是P1功能，MVP的目标是验证"养成游戏感"，不是验证工具能力。

不要为了看起来完整就硬塞。

### 3.2 员工成长数值体系（等级/经验/能力树）

**为什么不做：** 参见game-mechanics.md中的分析。数值系统不等于成长感，工作空间积累才是。

### 3.3 模板导出/分享

**为什么不做：** 社交功能是扩张用的，不是留存用的。MVP阶段先把留存做好。

### 3.4 多员工协作任务

**为什么不做：** 单员工的体验都没打磨好，多员工协作会引入大量异常场景，MVP驾驭不了。

### 3.5 搜索功能（工作空间全文搜索）

**为什么不做：** 用户工作空间还没积累内容，搜索没有意义。P1功能，等工作空间有内容了再上。

### 3.6 自定义提示词基底编辑器

**为什么不做：** 模板自带的提示词对新用户够用。提示词编辑器是高级用户功能，不是MVP。

### 3.7 离线/多设备同步

**为什么不做：** 技术复杂度高，MVP阶段明确声明"本地数据"即可。

### 3.8 骨骼动画库完整接入（多套角色）

**为什么不做：** MVP阶段1套动画能跑通即可，多样性是P1。先验证方向对不对。

## 四、边界模糊区（需要确认）

### 4.1 工作空间的展示

MVP阶段工作空间需要展示什么？

建议：只展示"最近任务日志"和"员工笔记"两个板块，其他内容P1上。
不要做成一个复杂的文件管理器。

### 4.2 员工数量上限

MVP阶段最多几个员工？

建议：最多3个（限制不是技术，是UX——太多员工在一个界面会乱，MVP先把3个做好看）

### 4.3 记忆实例是否上MVP

建议：MVP阶段记忆实例可以不做，工作实例+社交实例的并发体验是核心。
记忆整理是P1，等工作空间积累了内容再做自动整理。

## 五、MVP通关标准

**定性标准（主观感受，但必须达到）：**

- 5个不熟悉AI工具的用户测试，至少3个人说"这也太酷了"
- 打开应用的第一眼，测试用户会停顿至少3秒（意味着在认真看）
- 完成第一次任务分配后，至少1个测试用户主动问"还能做什么？"

**定量标准：**

- 员工创建流程：3步以内完成
- 员工状态更新延迟：<3秒（从实际状态变化到UI更新）
- 第一次打开到开始第一次交互：<5分钟（含创建员工时间）

## 五-B、技术约束对MVP决策的修正（Claude源码分析后补）

> 基于 Claude Code 源码（query.ts / Tool.ts / compact.ts），以下几个产品决策需要更新。

### 修正1：模块注入的token代价比预期高

`Tool.ts` 里每个工具的 `prompt()` 方法会把工具描述注入系统提示词。Claude Code 用 `shouldDefer` + `ToolSearch` 机制延迟加载不常用工具来控制token消耗。

**产品影响：**

- "功能模块 = 工具集动态注入"技术可行，但每激活一个模块确实增加token消耗
- MVP阶段模块数量要克制：只给当前任务用到的工具，不全量注入
- 参考 `shouldDefer` 思路：任务用不到的工具不进上下文，按需加载

### 修正2：三实例共享工作空间需要文件并发控制

源码中Agent之间通信依赖文件系统作为共享状态。工作实例写文件时，记忆实例可能同时读/写同一文件。

**产品影响：**

- 工作空间文件是三实例的通信媒介，架构可行
- 但需要简单的写锁或版本机制，防止并发写冲突
- MVP简化方案：实例A写操作记录，实例B读+归纳，时序隔离（不要真正并发写同一文件）

### 修正3：记忆实例的核心是"对话压缩+摘要持久化"

Claude Code 的 autoCompact 把对话历史压缩成摘要，核心信息保留。这意味着员工"记忆"不需要向量数据库，只需要：**定期压缩历史 + 把摘要写进工作空间文件**。

**产品影响：**

- 员工记忆 = 压缩摘要（自动）+ 工作空间笔记（员工主动）
- MVP阶段不需要独立的记忆数据库
- 记忆实例（实例B）的最简实现：工作完成后触发一次压缩，摘要持久化到工作空间

## 六、MVP之后的迭代节奏建议

| 阶段 | 核心目标          | 标志性功能                       |
| ---- | ----------------- | -------------------------------- |
| MVP  | 惊艳感 + 基础可用 | 视觉体验 + 单员工交互            |
| P1   | 真正有用          | 程序员模块 + 记忆实例 + 多员工   |
| P2   | 留存 + 传播       | 成长体系 + 模板分享 + 协作可视化 |

---

_老乔最终判断：_
_MVP如果把视觉做到位 + 员工真的能干活，就已经远超大多数AI工具。_
_不要因为功能多就觉得安心——用户要的是一次完整的惊艳体验，不是一堆半生不熟的功能。_
