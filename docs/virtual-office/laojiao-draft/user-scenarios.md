# 用户场景分析

> 产品-老乔 独立产出，基于用户真实行为路径推导

---

## 场景一：第一次打开虚拟办公室

### 1.1 用户心理状态

用户到达这里，之前看到过某个截图或介绍，心里的期待是："这好像挺酷的"。
但他同时也带着警惕：**又是一个花里胡哨没什么用的工具？**

这是最关键的30秒。产品要在这30秒内让他说"哇"，而不是"哦，我看懂了，很普通"。

### 1.2 第一次打开 — 理想路径

1. **进入动画**：不能只是一个空列表。应该有一个"空荡荡的办公室"场景，灯光逐渐亮起，有一台空桌子在等待。
   - 关键：**空不等于无聊**，空是在暗示"你来填满它"
2. **引导词**：不是"点击创建员工"这种SaaS话术，而是"你的团队还没人，要招募第一个成员吗？"
3. **模板展示**：展示时不能像表格，应该像"求职卡"——每个模板角色有一句话自我介绍、一个小动画
4. **一键创建**：点下去，员工"走进"办公室坐下，有入职动画

### 1.3 第一次打开 — 真实用户行为（需防范的反模式）

- 用户不会认真读说明文字，会直接乱点
- 模板太多会选择困难（超过5个就开始犹豫）
- 如果创建流程超过3步并且没有即时反馈，用户会以为卡了

**结论：默认只展示3个模板，其他折叠在"更多"里。**

### 1.4 "哇"效果的来源

不是功能多，而是：

- 员工**真的在动**（骨骼动画），不是静态图标
- 员工状态**实时变化**（工作中 vs 空闲），用户一眼能感受到"活着"
- 交互响应**有性格**：问他在干什么，他不是返回JSON，而是用符合人设的语气回答

## 场景二：日常使用（已有2-5个员工）

### 2.1 早上打开，扫一眼

用户期望：**不用操作，扫一眼就知道团队状态。**

这是最高频的使用行为，不是"管理员工"，而是"确认一切正常"。

需要在主视图给出：

- 谁在忙（工作中）
- 谁闲着（可以分配任务）
- 有没有卡住的（需要人工介入）
- 昨天完成了什么（简短的成就感反馈）

### 2.2 给员工布置任务

真实流程：

1. 用户脑子里有个想法（不是精确的任务描述）
2. 打开员工卡片，点"聊天"
3. 随便说一句"帮我整理一下今天的代码改动"
4. 等待/切换去做别的
5. 稍后回来看结果

**关键：用户不会守着等**。他要能离开，之后回来看结果。所以异步通知必须有。

### 2.3 查看员工工作日志

用户场景：员工说完成了，但用户想知道他到底做了什么。

- 点击员工卡片 → 工作空间 → 最近操作记录
- 这里不是debug日志，而是**用人类语言写的工作日志**：
  "今天分析了3个错误日志，发现2个是同类问题，已整理到笔记#47"

### 2.4 和员工即时聊天

用户场景：灵感闪现，想问员工一个问题，不想发任务。

- 点击员工 → 弹出聊天窗口（社交实例响应）
- 关键：**聊天和他正在做的任务互不干扰**（三实例机制的核心价值）
- 反模式：问他一句话导致他停下手头的工作——这会让用户失去信任

## 场景三：高级使用（重度用户）

### 3.1 多员工协作任务

用户场景：一个大任务需要多个员工配合。

典型流程：

1. 找领导角色员工，让他拆解任务
2. 领导员工自动分配给下属
3. 用户在主视图看到多个员工同时进入"工作中"状态
4. 任务完成后，领导员工汇报

**这是最有游戏感的场景。** 但也是最容易失控的：如果协作出错，用户很难排查。

### 3.2 调教员工人设

重度用户会希望员工有独特个性：

- 修改提示词基底
- 给员工"写故事"——背景经历影响行为风格
- 看员工的成长记录（做了哪些任务，有什么积累）

### 3.3 清理失控员工

用户场景：员工陷入循环，token一直在烧。

- 需要紧急停止机制（一键暂停所有实例）
- 操作后员工进入"休眠"，不是"死亡"——保留状态，可恢复
- 让用户能看到"这次任务烧了多少"

---

## 补充：用户不会做的事（不要为这些设计）

- 用户**不会**主动去查三实例的运行状态（这是技术细节，产品层面隐藏）
- 用户**不会**给员工精确的任务格式，他们说话是口语
- 用户**不会**在第一次使用时阅读"功能说明"

---

_老乔注：以上场景是基于真实用户行为推导，不是"假设用户会认真看说明书"。_
